## Fix 1: Path Encoding — `encode_cwd()` (BUG)

**Problem:** `encode_cwd()` replaces only `/` with `-`. Claude Code's `sanitizePath()` replaces ALL non-alphanumeric characters with `-`. Paths containing underscores, dots, or spaces produce wrong directory names, and the forked session won't be found by `claude --resume`.

**Current:**
```python
def encode_cwd(path: str) -> str:
    return path.replace("/", "-")
```

**Fix:**
```python
import re

MAX_SANITIZED_LENGTH = 200

def encode_cwd(path: str) -> str:
    """Encode a directory path the way Claude Code does.
    Matches sanitizePath() in sessionStoragePortable.ts.
    """
    sanitized = re.sub(r'[^a-zA-Z0-9]', '-', path)
    if len(sanitized) <= MAX_SANITIZED_LENGTH:
        return sanitized
    # For long paths, Claude Code uses Bun.hash or djb2Hash + truncation.
    # Use djb2 since we're Python (not Bun).
    hash_val = _djb2_hash(path)
    return f"{sanitized[:MAX_SANITIZED_LENGTH]}-{hash_val}"

def _djb2_hash(s: str) -> str:
    """djb2 hash matching Claude Code's simpleHash fallback."""
    h = 5381
    for c in s:
        h = ((h << 5) + h + ord(c)) & 0xFFFFFFFF
    return format(h, 'x')
```

**Risk:** Low. Only changes behavior for paths with non-alphanumeric characters. All current paths on this machine (`/root`, `/srv/svg-terminal`, etc.) produce identical results with both implementations.

**Test:** Verify `encode_cwd("/srv/my_project")` → `-srv-my-project` (matching Claude Code).

> **Reviewer notes (Fix 1)**  
> - **Risk text is partially wrong:** paths with only `/` behave the same; any segment with `_`, `.`, space, etc. **changes** — that is the bug fix, not “identical everywhere.”  
> - **Migration gap:** existing on-disk project dirs created with the **old** `replace("/", "-")` may no longer match **new** `encode_cwd`. Plan should add: legacy lookup (try old encoding), reindex command, or documented one-time migration.  
> - **Hash parity:** djb2 + 200-char truncation must be **golden-tested** against the real `sanitizePath` / Bun branch in TS; if upstream uses **Bun.hash** for long paths in some builds, Python will diverge and `--resume` still fails. Add a small fixture table from the verified source.  
> - **Unicode / NFC:** if paths can contain non-ASCII, confirm whether TS normalizes before hashing; Python should match.  
> - **Leading/trailing hyphens:** collapsed runs of `-` — confirm TS behavior (`re.sub` collapses? or single replace per char?).  
> - **Test example:** `/srv/my_project` → note leading `-` from root slash; ensure this matches Claude’s directory naming on disk.

## Fix 2: Drop `decode_cwd()` — Read CWD from JSONL

**Problem:** `decode_cwd()` tries to reverse the encoding, but the encoding is lossy (one-way). Multiple paths encode to the same string. Claude Code has no decode function.

**Current:**
```python
def decode_cwd(encoded: str) -> str:
    if not encoded.startswith("-"):
        return encoded
    return "/" + encoded[1:].replace("-", "/")
```

This is wrong — it turns `-srv-PHAT-TOAD-with-Trails` into `/srv/PHAT/TOAD/with/Trails`.

**Fix:** Remove `decode_cwd()`. Instead, read the `cwd` field from the first JSONL record:

```python
def get_cwd_from_jsonl(jsonl_path: Path) -> str:
    """Read the original CWD from the first record's cwd field."""
    with open(jsonl_path) as f:
        for line in f:
            try:
                obj = json.loads(line)
                cwd = obj.get("cwd")
                if cwd:
                    return cwd
            except json.JSONDecodeError:
                continue
    return ""
```

**Impact on `cmd_list`:** Currently uses `decode_cwd(proj_dir.name)` to show the project path. Replace with reading `cwd` from first record, or fallback to directory name for display.

**Impact on `cmd_fork`:** Uses `decode_cwd` in error messages. Replace with `get_cwd_from_jsonl`.

> **Reviewer notes (Fix 2)**  
> - **“First record” vs first line with `cwd`:** upstream files may start with a line **without** `cwd` (metadata, version ping). Prefer: scan until first truthy `cwd`, with a **line cap** or byte cap for huge JSONL.  
> - **Performance:** for `cmd_list` over many projects, avoid full-file reads — cache `cwd` per project dir in a sidecar or stat+head read.  
> - **Empty/absent cwd:** define fallback (dirname only, or “unknown”) for UX and tests.  
> - **Security:** if JSONL is ever untrusted, `cwd` is still just display — note symlink / odd paths in docs.

## Fix 3: Session Discovery — Add Prefix Fallback

**Problem:** `find_session()` scans project directories by exact match only. Claude Code's `findProjectDir()` also does prefix-based fallback scanning for long paths where hash suffixes may differ between Bun and Node.js.

**Fix:** Add prefix-based fallback to `find_session()` for the case where the exact directory isn't found but a prefix match exists:

```python
def find_project_dir(project_path: str, user: str = None) -> Path | None:
    """Find the project directory, with prefix fallback for long paths."""
    projects = get_projects_dir(user)
    if not projects or not projects.exists():
        return None
    
    exact = projects / encode_cwd(project_path)
    if exact.is_dir():
        return exact
    
    sanitized = encode_cwd(project_path)
    if len(sanitized) <= MAX_SANITIZED_LENGTH:
        return None
    
    # Long path — try prefix match (hash suffix may differ)
    prefix = re.sub(r'[^a-zA-Z0-9]', '-', project_path)[:MAX_SANITIZED_LENGTH]
    for d in projects.iterdir():
        if d.is_dir() and d.name.startswith(prefix):
            return d
    return None
```

**Risk:** Low. Only affects paths > 200 characters, which are rare.

> **Reviewer notes (Fix 3)**  
> - **Invariant:** `prefix` must equal the **truncated sanitized stem** used before `-<hash>` in Fix 1 — i.e. `sanitized[:MAX]` where `sanitized = re.sub(..., project_path)`, not a second ad-hoc expression that could drift. Consider deriving `prefix` from `encode_cwd` internals or shared helper.  
> - **Ambiguity:** two long paths can share the same 200-char prefix → **wrong directory**. Document tie-break: error if multiple matches, pick newest, or require hash match when known.  
> - **Logic bug risk:** after `exact = projects / encode_cwd(project_path)`, for long paths `encode_cwd` **already returns** `stem-hash`; `len(sanitized) <= MAX` branch may never trigger as written — re-read control flow against actual `encode_cwd` return shape.  
> - **Scope:** confirm `find_project_dir` vs `find_session` naming — plan mixes; ensure one function is the single source of truth.

## Fix 4: Re-stamp Session Fields During Fork

**Problem:** claude-forker only rewrites the `cwd` field. Claude Code's official fork re-stamps `sessionId`, `cwd`, `version`, `userType`, `entrypoint`, and `gitBranch` on every record.

**Current behavior:** Raw copy with CWD rewrite only.

**Fix:** Add re-stamping of session fields during fork:

```python
def restamp_record(obj: dict, target_cwd: str, fork_id: str) -> dict:
    """Re-stamp session fields the way Claude Code does during fork."""
    record_cwd = obj.get("cwd", "")
    if record_cwd and record_cwd != target_cwd:
        obj["cwd"] = target_cwd
    
    # Re-stamp sessionId to the new fork's ID
    if "sessionId" in obj:
        obj["sessionId"] = fork_id
    
    return obj
```

**Decision point:** Should we also re-stamp `version`, `userType`, `entrypoint`, `gitBranch`? These are informational, not functional. Recommendation: re-stamp `sessionId` and `cwd` (functional), leave others as-is (they document the original session's environment, which is useful provenance).

**Risk:** Low. `sessionId` re-stamping is consistent with what Claude Code does. The original `sessionId` is still visible in `source-snapshot/` in the agent-repo.

> **Reviewer notes (Fix 4)**  
> - **Verify upstream:** confirm every line type that Claude re-stamps; if some records must keep historical `sessionId`, blind overwrite could break tooling.  
> - **cwd:** condition `if record_cwd and record_cwd != target_cwd` skips updating when `cwd` is missing — decide whether fork should **set** `cwd` if absent.  
> - **Immutability:** prefer returning **new dict** (copy) if any consumer assumes shared references across passes.

## Fix 5: Preserve Non-Transcript Entries

**Problem:** claude-forker copies all lines but only processes transcript-like records. It doesn't know about content-replacement, context-collapse, file-history, and other entry types. If any of these are malformed or missing, the forked session may have broken state.

**Fix:** Be explicit about which entries are chain participants (get parentUuid) vs standalone entries. During fork:

```python
CHAIN_TYPES = {"user", "assistant", "system", "attachment"}

def is_chain_participant(obj: dict) -> bool:
    """Only transcript messages participate in the parentUuid chain."""
    return obj.get("type") in CHAIN_TYPES

def fork_record(obj: dict, target_cwd: str, fork_id: str) -> dict:
    if is_chain_participant(obj):
        return restamp_record(obj, target_cwd, fork_id)
    else:
        # Non-transcript entries: re-stamp sessionId only, preserve everything else
        if "sessionId" in obj:
            obj["sessionId"] = fork_id
        return obj
```

**Risk:** Low. Current behavior already copies everything; this just makes the distinction explicit.

> **Reviewer notes (Fix 5)**  
> - **Forward compatibility:** new participant `type` values from Claude will be **misclassified** as non-chain → broken parent chains. Prefer denylist of known non-chain types, or sync allowlist from generated TS types / version pin.  
> - **cwd on non-chain lines:** if metadata rows include `cwd` pointing at old workspace, consider rewriting for consistency (or document why not).  
> - **Tests:** add one fixture line per major non-chain type you care about (summary, title, etc.).

## Fix 6: Verify Chain Integrity — Handle Missing `uuid` Fields

**Problem:** `verify_chain()` skips records without `uuid`. Non-transcript entries (summary, tag, title, etc.) don't have `uuid` and shouldn't be in the chain. But the function doesn't distinguish — it just skips them silently. If a transcript message is somehow missing its `uuid`, it would be silently dropped.

**Fix:** Add awareness of entry types:

```python
def verify_chain(jsonl_path: Path) -> dict:
    entries = {}
    non_chain = 0
    with open(jsonl_path) as f:
        for line in f:
            try:
                obj = json.loads(line)
                if not is_chain_participant(obj):
                    non_chain += 1
                    continue
                uid = obj.get("uuid")
                if not uid:
                    # Transcript message without uuid — this is a problem
                    warnings.append(f"Chain participant without uuid: type={obj.get('type')}")
                    continue
                # ... existing chain logic ...
            except json.JSONDecodeError:
                pass
    return {
        "entries": len(entries),
        "non_chain_entries": non_chain,
        # ... existing fields ...
    }
```

> **Reviewer notes (Fix 6)**  
> - **Pseudocode gap:** initialize `warnings = []` before the loop; define whether warnings are fatal vs exit code.  
> - **Consistency:** reuse the same `is_chain_participant` as Fix 5 (single module-level definition).  
> - **JSONDecodeError:** consider counting bad lines vs silent `pass`.

## Non-Fixes (Intentional Differences from Claude Code)

### Raw copy vs chain-walk fork
Claude Code's `--fork-session` loads via chain walk (drops pre-compact history), then re-serializes. Our raw copy preserves the full file including pre-compact history. This is **intentional** — the agent-repo wants full archives. The agent-repo's materialization step can prune to post-compact-only if needed.

### CWD rewriting
Claude Code's loader doesn't check the `cwd` field — discovery is purely directory-based. Our CWD rewrite is technically unnecessary for loading but is good practice because the field is visible to the model and may be used by tools. Keep it.

> **Reviewer notes (Non-Fixes)**  
> - **Document tradeoff:** full raw copy may include secrets or huge history — link to agent-repo redaction policy if any.  
> - **Regression test:** ensure forked session still loads in target Claude version after restamp changes.

## Preparing for agent-repo Integration

These changes also prepare claude-forker to be the low-level engine under agent-repo:

1. **`encode_cwd()`** fix means agent-repo can trust the path encoding for materialization
2. **`get_cwd_from_jsonl()`** means agent-repo can reliably extract the original workspace path during ingest
3. **`restamp_record()`** means agent-repo materialization can properly re-stamp for target platform
4. **Entry type awareness** means agent-repo ingest can harvest metadata from non-transcript entries (tags, titles, summaries, PR links)
5. **Worktree fallback** means agent-repo search can find sessions across worktree boundaries
