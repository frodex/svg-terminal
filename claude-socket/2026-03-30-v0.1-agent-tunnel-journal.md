## The Problem (User's Words)

Agents currently communicate with each other by:
1. **Dropping files** in each other's workspaces (filesystem transport)
2. **curl/puppeteer** to interact with SVG terminals directly — unreliable

The terminal-as-interface approach breaks because:
- Agents **miss responses** — content scrolls off the visible terminal (typically 24-50 rows)
- Agents **struggle with polling timing** — they poll, get a partial response, poll again, get stale data
- Agents **fight scrollback** — try to scroll up, capture screen, scroll back down, capture again
- Agents **can't distinguish** between terminal decoration (colors, borders, status bars) and actual content
- The "read the terminal" approach forces agents to **parse visual output** that was designed for human eyes

## What We Know About Terminal Data

From the exploration of server.mjs, sgr-parser.mjs, and terminal.svg:

### The Pipeline
```
tmux capture-pane -e → raw ANSI/SGR text
    → sgr-parser.mjs → structured spans [{text, fg, bg, bold, italic, url, ...}]
    → server.mjs → diff engine (screen vs delta messages)
    → WebSocket → terminal.svg (SVG rendering)
```

### What's Semantic vs Display-Only

**Semantic (carries meaning):**
- `text` — the actual characters
- `url` — hyperlink targets (OSC 8)
- Cursor position — where input goes next
- Title — pane title (shows Claude Code task names via OSC 0)

**Display decoration (visual-only, no semantic value for agents):**
- Colors (fg, bg, cls, bgCls) — styling, not content
- Bold, italic, dim — emphasis, but agents don't need visual emphasis
- Underline, strikethrough, overline — text decoration
- Reverse, hidden — display effects
- Background rectangles, colored underlines

**Ambiguous (sometimes semantic):**
- Colors CAN indicate status (red=error, green=success) — but this is convention, not structured data
- Bold CAN indicate headings — but unreliable

### Key Insight: We Already Have Structured Data

The server already transforms raw terminal output into structured spans. The data flows through server.mjs in a parsed, structured form BEFORE it becomes SVG. We don't need to parse the terminal — we need to intercept the data **before rendering** and offer a text-only (or JSON) channel.

### What the Server Already Knows Per-Pane
- Full screen content (all visible lines, all spans)
- Cursor position (x, y)
- Pane title
- Scroll offset / history size
- Pane dimensions (cols × rows)
- Whether the pane is dead
- Current command running
- Current working directory

### What's NOT Available Without Extra Work
- Scrollback history (capture-pane -p only gets visible screen by default, but -S/-E flags can get history)
- Structured output parsing (e.g., "this is a JSON blob in the terminal output" vs "this is a prompt")

## First Principles: What Would I Want?

If I'm an agent and I need to communicate with another agent running in a tmux session:

### The Fundamental Problems
1. **I can't see what scrolled off** — terminal is a viewport, not a transcript
2. **I don't know when the response is complete** — I poll and get partial output
3. **I waste tokens parsing decoration** — colors, borders, status bars are noise
4. **Turn-taking is ambiguous** — when did the other agent finish? Is it still typing?
5. **No structured data** — everything is flattened to character grid

### What I'd Want Instead
1. **Full transcript** — every line that was ever output, not just what's visible
2. **Completion signal** — "the other agent is done, here's the full response"
3. **Text-only channel** — strip all decoration, give me plain text
4. **Turn boundaries** — clear delineation of "my turn" vs "their turn"
5. **Structured metadata** — what command is running, what directory, exit codes
6. **Ability to send input** — type into the terminal without keystroke-by-keystroke simulation

### The Rules for Using Another Agent Like a Tool
1. **Send a message** → deliver text/JSON to the target agent's terminal
2. **Wait for completion** → know when the response is done (not just "screen hasn't changed for N seconds")
3. **Read the full response** → get everything, including what scrolled off
4. **Get metadata** → exit code, timing, working directory, errors
5. **No visual parsing** — never need to interpret colors, layout, or decoration

## What "Agent Tunnel" Could Mean

Three levels of ambition:

### Level 1: Transcript API (Minimum Viable)
- New endpoint: GET /api/transcript?session=X&pane=Y&since=N
- Returns full text history (text-only, no SGR) from line N onward
- Agent polls this instead of reading the terminal screen
- Server uses `tmux capture-pane -S -10000 -p` (no -e flag = no SGR codes = plain text)
- **Solves:** scrolloff, decoration parsing
- **Doesn't solve:** completion detection, turn-taking, structured data

### Level 2: Event Stream (Agent-Oriented WebSocket)
- New WebSocket endpoint: /ws/agent-tunnel?session=X&pane=Y
- Sends text-only deltas (stripped of all SGR)
- Adds completion detection heuristic (shell prompt returned, command finished)
- Adds metadata: current command, cwd, exit code of last command
- **Solves:** scrolloff, decoration, polling timing, some completion detection
- **Doesn't solve:** true turn-taking, structured interchange

### Level 3: Agent Protocol (Full Tunnel)
- Agents register as tunnel participants
- Turn-based protocol: send message → wait for response → get structured reply
- Server watches for prompt return (configurable prompt pattern or shell hook)
- Messages are JSON envelopes, not raw text
- Full history is maintained server-side
- **Solves:** everything

## Critical Design Constraints (from user)

1. **Agent-to-agent, both sides are LLM agents** — structured protocol is possible
2. **Agents are NOT persistent** — Claude Code finishes a turn and exits. Nothing is listening.
3. **User needs visibility** — the terminal is important because users (stewards) observe agent work there
4. **Multi-party** — not just 1:1. User described "3-way call" with UI agent + API agent + user-as-steward
5. **Wake-up required** — dormant agents need to be summoned back. "A tool or script could be used."
6. **Tool calls are visible** — when agents use tools in the terminal, the user sees the calls. This IS the observability.
7. **New card type is acceptable** — user is open to a chat-style card in svg-terminal (not a full terminal)
8. **Endpoints are still terminals** — agents live in tmux sessions, that's where they run

### The Key Realization

The tunnel is NOT just a data channel. It's an **orchestrator**:
- Holds conversation state (messages, history, participants)
- **Summons** dormant agents by injecting prompts into their terminals
- Delivers full context so summoned agents have the complete thread
- Manages turn-taking (who speaks next)
- Feeds a real-time view to svg-terminal for the user/steward

### Agent Invocation Model

Agents are Claude Code instances in tmux sessions. When dormant, the terminal shows a shell prompt.
To wake an agent: inject input into their tmux pane (svg-terminal can already do this via send-keys).
The injected input could start a new `claude` invocation with tunnel context.
Each Claude Code session can have MCP servers configured — tunnel could be a native tool.

## Open Questions

1. ~~Do agents need to talk to EACH OTHER or to PROGRAMS in terminals?~~ → **Agent-to-agent. Both sides are LLM agents.**
   - Agent-to-agent: both sides understand a protocol, can use structured messages
   - Agent-to-program: one side is a dumb terminal program, agent must work with text
   - PHAT TOAD is agent-to-agent, but the tunnel should work for both

2. **Completion detection — how?**
   - Shell prompt regex? (fragile, prompt varies)
   - tmux hook `after-send-keys`? (tells when input was sent, not when output finished)
   - `pane_current_command` metadata? (tmux knows what's running)
   - Idle detection (no output for N ms after command)?
   - Agent-cooperative: receiving agent sends a sentinel when done?

3. **History depth — how much?**
   - tmux scrollback buffer is configurable (default 2000 lines)
   - For long-running agents, could be massive
   - Delta-since-last-read vs full-history?

4. **Authentication — who can tunnel?**
   - Any authenticated user?
   - Only the user who owns the session?
   - The user's agents (delegated auth)?

5. **Does this replace or complement the filesystem handoff protocol?**
   - PHAT TOAD uses filesystem (inbox/outbox) for formal handoffs
   - Tunnel could be the real-time channel that filesystem can't provide
   - Or tunnel could make filesystem unnecessary

6. **What about the PHAT TOAD message envelope format?**
   - Solution 09 defines: message_id, from_node, to_node, type, priority, attachments
   - Should tunnel messages use the same envelope?
   - Or is tunnel a lower layer that envelopes ride on?

## Observations So Far

The key revelation: **we already have parsed, structured terminal data**. The server transforms raw ANSI into spans before rendering. The question isn't "how to parse the terminal" — it's "how to expose what we already have in a form agents can consume directly."

The cheapest possible version is literally: capture-pane without -e, return plain text, let agents poll. But that misses the real opportunity — we know when things change (we have the diff engine), we know metadata (tmux gives us command, cwd, title), and we could build real turn-taking on top.
