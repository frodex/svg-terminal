## New Information Since v0.1

### User Clarifications (Q&A)

**Q: Agent-to-agent or agent-to-program?**
A: Agent-to-agent. Both sides are LLM agents. But the terminal remains important because:
- Users (stewards) observe agent work through svg-terminal
- Tool calls are visible in the terminal — this IS the observability layer
- The terminal is the monitoring interface, not the data transport

**Q: How are agents invoked?**
A: Claude Code instances in tmux sessions, managed by claude-proxy. claude-proxy was designed for persistent, long-lived sessions that survive SSH drops and reboots. It has an API for session management. Both codebases are owned — all possibilities are open.

### The Scenario (User's Words, Paraphrased)

> User is working with 2 expert agents on different parts of an application — UI and API. UI agent hits a block, needs to debate technical details with API agent who recently finished the API but may have missed documentation or features. User (or an agent) initiates a "3-way call" — UI agent + API agent + user-as-steward (or a steward agent moderating). The user needs visibility via svg-terminal. Agents are not persistent — when one ends their turn, there needs to be a way to "wake them up."

### Key Constraint: Agents Are Ephemeral

Claude Code finishes a turn and exits. The process is gone. Nothing is listening. This means:
- The tunnel must be a **persistent service** independent of any agent
- Wake-up requires **injecting a new invocation** into the agent's terminal
- The agent needs **full conversation context** on every wake-up (no session memory between invocations, unless Claude Code's conversation history persists — which it does if the session is resumed via claude-proxy)

### Key Enabler: claude-proxy Session Lifecycle

claude-proxy already handles:
- Creating persistent sessions
- Reconnecting after SSH drops
- Session lobby and discovery
- API for session management

The tunnel doesn't need to reinvent session lifecycle — it can delegate to claude-proxy.

## Three Approaches Evaluated

### Approach 1: MCP-Native Tunnel (RECOMMENDED)

**Core idea:** The tunnel is a persistent service. Agents interact with it through MCP tools — their native interface. Tool calls are visible in the terminal, preserving user observability.

```
┌──────────────────────────────────────────────────────┐
│              Tunnel Service (persistent)              │
│  - conversation rooms with full message history      │
│  - participant state (active / dormant / invited)    │
│  - turn management (who speaks next)                 │
├─────────┬──────────┬──────────┬──────────────────────┤
│   WS    │    WS    │   WS     │  REST/WS             │
│         │          │          │                       │
▼         ▼          ▼          ▼                       │
MCP       MCP        Chat      claude-proxy API        │
Server    Server     Card      (wake-up / spawn)       │
│         │          │                                  │
▼         ▼          ▼                                  │
Agent A   Agent B    User/Steward                      │
(UI)      (API)      (svg-terminal dashboard)          │
└──────────────────────────────────────────────────────┘
```

**How agents interact:**
- MCP server configured globally in `~/.claude/mcp_servers.json`
- Every Claude Code session gets tunnel tools automatically
- Tools: `tunnel_join`, `tunnel_send`, `tunnel_read`, `tunnel_yield`, `tunnel_list`
- Tool calls appear in the terminal — user sees them in real-time via svg-terminal
- Structured JSON in and out — zero terminal parsing

**How wake-up works:**
1. Message arrives for dormant Agent B
2. Tunnel service calls claude-proxy API: `POST /api/session/{id}/resume` with tunnel context
3. claude-proxy starts (or resumes) the Claude Code session
4. Agent B's system prompt / initial message includes: "You've been summoned to tunnel 'auth-review'. Your MCP tunnel tools are available. Read history and respond."
5. Agent B calls `tunnel_read` → gets full conversation history as structured JSON
6. Agent B calls `tunnel_send` → posts response
7. Agent B calls `tunnel_yield` → signals turn complete
8. Tunnel service notifies next participant

**How the user observes:**
- New card type in svg-terminal: **Chat Card** — renders the tunnel conversation
- Shows messages with attribution (Agent A, Agent B, User)
- Shows agent status indicators (active/thinking, dormant, invited)
- User types into the chat card — their messages enter the tunnel
- User can ALSO watch each agent's actual terminal card to see tool calls and reasoning
- Two views of the same conversation: the structured chat card + the raw terminal cards

**MCP Tools Spec (Draft):**

```
tunnel_list_rooms()
  → [{ id, name, participants: [{id, name, status}], messageCount, created }]

tunnel_join(room_id)
  → { joined: true, history: [...messages], participants: [...] }

tunnel_read(room_id, since?: message_id)
  → { messages: [{ id, from, content, timestamp, type }], participants: [...] }

tunnel_send(room_id, content: string | object, to?: participant_id)
  → { message_id, delivered_to: [...] }

tunnel_yield(room_id, summary?: string)
  → { yielded: true, next_participant?: string }

tunnel_invite(room_id, agent_id, context?: string)
  → { invited: true, wake_up_status: "sent" | "already_active" }
```

**Trade-offs:**
- (+) Native tool interface — zero friction for agents, structured data
- (+) Tool calls visible in terminal — observability is inherent, not bolted on
- (+) Full history managed server-side — nothing scrolls off
- (+) Wake-up via claude-proxy API — clean session lifecycle
- (+) Chat card provides unified view for user/steward
- (+) Multi-party by design — rooms, not point-to-point
- (+) Complements PHAT TOAD filesystem protocol (real-time layer, formal handoffs stay in filesystem)
- (-) Requires MCP server deployed to all agent environments
- (-) MCP subprocess dies with Claude Code — must reconnect on wake-up (but this is fine — tunnel service holds state, MCP server is stateless client)
- (-) More moving parts than alternatives

### Approach 2: CLI Wrapper Tunnel

**Core idea:** A script wraps agent invocation. Instead of Claude Code running bare in tmux, it runs inside `tunnel-agent` which handles transport.

```bash
# In Agent B's tmux pane:
tunnel-agent --room auth-review --identity api-agent
```

The script:
1. Connects to tunnel service via WebSocket
2. Reads conversation history
3. Invokes `claude --prompt "..."` with history as context
4. Captures Claude's output, sends to tunnel
5. When Claude exits, sends yield message
6. Blocks waiting for next summon

**Trade-offs:**
- (+) Simpler — no MCP configuration needed
- (+) The wrapper IS the wake-up mechanism (always listening)
- (+) Works with any LLM CLI, not just Claude Code
- (-) Wrapping Claude Code loses tool-level visibility — user sees the wrapper, not Claude's native tool calls
- (-) Agent can't use tunnel mid-conversation — only sees context at invocation
- (-) Loses interactive feel — more like batch processing than conversation
- (-) Agent's response must be parsed from stdout — reintroduces the parsing problem

### Approach 3: Filesystem + Notification Hybrid (PHAT TOAD Extension)

**Core idea:** Extend PHAT TOAD's inbox/outbox filesystem protocol with a real-time notification and rendering layer. Messages remain files (immutable, auditable). A watcher service pushes WebSocket notifications.

**Trade-offs:**
- (+) Preserves PHAT TOAD's immutable audit trail natively
- (+) Every message is a file — easy to inspect, git-track, debug
- (+) No new transport protocol — extends what exists
- (-) Slow — filesystem polling adds latency even with fs.watch
- (-) Agents still read files, not tools — still parsing text
- (-) Wake-up still unsolved on top of this
- (-) Doesn't address core complaint (agents struggling to get data cleanly)
- (-) Chat card would need to render from filesystem, not a live stream

## Why MCP-Native Wins

The fundamental complaint: agents are forced to communicate through a visual medium (the terminal) when they need structured data. The MCP approach solves this at the right layer:

1. **MCP tools ARE structured data** — they're the native way Claude Code agents interact with external systems. Making the tunnel a tool makes it invisible infrastructure.

2. **Tool calls are already visible in the terminal** — when Agent A calls `tunnel_send({ to: "api-agent", message: "What's the auth token format?" })`, the user sees that tool call in Agent A's terminal card. Observability isn't designed in — it's inherent in how Claude Code works.

3. **The chat card gives the user a unified view** — instead of watching multiple terminals and mentally reconstructing the conversation, the user sees a clean multi-party thread. They can still drill into any agent's terminal for the raw reasoning.

4. **Wake-up is clean** — claude-proxy already manages session lifecycle. The tunnel tells claude-proxy "wake this agent up," claude-proxy handles the mechanics.

5. **Multi-party is natural** — rooms, participants, turn management. Not hacked onto a point-to-point protocol.

6. **Complements rather than replaces PHAT TOAD** — the filesystem protocol handles formal handoffs (PRDs, constraint declarations, break tests). The tunnel handles real-time conversation. Different tools for different interaction types.

### Component Breakdown

**1. Tunnel Service** (lives in server.mjs or as separate module)
- REST API for room management, history retrieval
- WebSocket for real-time message streaming
- Room state: participants, messages, turn order
- Persistence: SQLite (already in the project for user-store)
- Wake-up: calls claude-proxy API when a dormant agent needs to respond

**2. Tunnel MCP Server** (standalone Node.js process, configured globally)
- Stateless client — connects to tunnel service via WebSocket
- Exposes tools to Claude Code: join, send, read, yield, invite, list
- Dies and restarts with each Claude Code invocation — that's fine
- On startup, reconnects to any active rooms the agent was in

**3. Chat Card** (new svg-terminal card type)
- Rendered by dashboard.mjs alongside terminal cards
- Shows tunnel conversation as a threaded message list
- User can type messages (participates as steward)
- Shows participant status indicators
- Can be focused/unfocused like terminal cards
- Created via `createChatCardDOM(tunnelId)` — uses the card factory

**4. Wake-Up Flow** (tunnel service → claude-proxy → agent terminal)
1. Tunnel service detects message for dormant participant
2. Tunnel service calls claude-proxy: `POST /api/session/{id}/send-input`
3. claude-proxy injects into the tmux pane: `claude --resume --prompt "You've been summoned to tunnel 'auth-review'. Use tunnel_read to get context."`
4. Claude Code starts, MCP server subprocess starts, connects to tunnel
5. Agent reads history, responds, yields
6. Claude Code exits, MCP server dies — tunnel service marks agent as dormant

## The Tunnel Message Format

Designed to complement (not replace) PHAT TOAD's filesystem message envelope.

```json
{
  "id": "msg-a1b2c3",
  "room_id": "auth-review",
  "from": {
    "id": "agent-ui",
    "name": "UI Agent",
    "type": "agent"
  },
  "to": null,
  "type": "message",
  "content": "What format should auth tokens use? I'm implementing the login flow and need to know whether to expect JWT or opaque tokens.",
  "timestamp": "2026-03-30T14:23:01Z",
  "turn_id": "turn-003",
  "metadata": {
    "phat_toad_node": "/root/ui-agent",
    "session_id": "cp-ui-session"
  }
}
```

**Message types:**
- `message` — regular conversation message
- `join` — participant joined the room
- `yield` — participant finished their turn
- `leave` — participant left
- `invite` — someone invited a new participant
- `system` — tunnel service announcements (agent woken, agent dormant, etc.)
- `steward` — message from user/steward (distinct from agent messages for clarity)

**Room state:**
```json
{
  "id": "auth-review",
  "name": "Auth Token Format Discussion",
  "created": "2026-03-30T14:20:00Z",
  "participants": [
    { "id": "agent-ui", "name": "UI Agent", "status": "active", "session_id": "cp-ui-001" },
    { "id": "agent-api", "name": "API Agent", "status": "dormant", "session_id": "cp-api-002" },
    { "id": "user-root", "name": "root", "status": "active", "type": "steward" }
  ],
  "turn": {
    "current": "agent-api",
    "mode": "round-robin" | "free" | "steward-directed"
  },
  "messages": [ ... ],
  "phat_toad": {
    "constraint_docs": ["auth-CONSTRAINT-ui.md"],
    "related_trails": ["/root/ui-agent/trails/turn-047"]
  }
}
```

## Relationship to PHAT TOAD Protocols

The tunnel is a **real-time conversation layer**. PHAT TOAD's filesystem protocol is a **formal artifact exchange layer**. They serve different purposes:

| Aspect | Tunnel (Real-time) | Filesystem (Formal) |
|--------|-------------------|-------------------|
| Speed | Sub-second | Seconds to minutes |
| Format | Chat messages (JSON) | Documents (Markdown) |
| Persistence | SQLite + server memory | Files on disk |
| Audit trail | Message log | Immutable numbered files |
| Use case | Discussion, Q&A, debate | PRDs, constraints, handoffs |
| Participants | Multi-party rooms | Usually bilateral |
| Formality | Conversational | Structured artifacts |

**They complement each other:**
- Tunnel for: "What format should auth tokens use?" (quick question, needs fast answer)
- Filesystem for: "Here is my constraint declaration for the auth system" (formal artifact)
- A tunnel conversation might PRODUCE a filesystem artifact: "OK, we agreed on JWT. Let me write the constraint doc."

The tunnel could even have a tool for this: `tunnel_formalize(room_id, message_range)` → generates a PHAT TOAD-formatted document from a conversation range.
