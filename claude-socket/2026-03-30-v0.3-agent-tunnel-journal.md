**Changes from v0.2:** Major UX shift — from abstract rooms to physical drag-to-connect cables

## The Shift: Connections, Not Rooms

User reviewed the screenshot of two side-by-side Claude Code sessions in svg-terminal and proposed:

> "I'm thinking like a little extension cord object you can just drag off the title bar of one to the card of the other to make the connector."

This replaces the "create room → invite agents" abstraction with a **physical gesture** — you literally wire agents together in 3D space. The cable is visible. The connection is tangible.

### Why This Is Better

The room-based model required:
1. Create a room (abstract concept, needs a name)
2. Invite Agent A
3. Invite Agent B
4. Both agents join

The cable model:
1. Drag from Card A to Card B (or agent calls `tunnel_connect`)
2. Done. They're connected.

The room still exists as an implementation detail (something needs to hold the message history), but the user never thinks about rooms. They think about **wires between things**.

### Direction Matters

The user specified that drag direction has meaning:
- Drag FROM Card A TO Card B → A is the requester, B is the responder
- B gets woken up with context about what A needs
- The cable visual could show direction (arrow, glow, color)

### Three Connection Modes

The user specified three ways to connect:

```
AGENT1->AGENT2                    1:1 direct wire
AGENT1->(AGENT2,AGENT3,separated) Separate wires to each (they don't see each other)
AGENT1->(AGENT2,AGENT3,combined)  Shared channel (everyone sees everything)
```

Implementation mapping:
- `A->B` = one private channel (room with 2 participants)
- `A->(B,C,separated)` = two private channels (two rooms, A is in both, B and C are each in one)
- `A->(B,C,combined)` = one shared channel (room with 3+ participants)

### Agent-Initiated Connections

User confirmed agents can also create connections. When the user says "ask the API agent directly," the agent calls a tool and the cable appears visually in the dashboard — same as if the user dragged it.

This means the MCP tools shift from room-oriented to connection-oriented:
- `tunnel_connect(target_agent, context?)` instead of `tunnel_create(room_name)`
- `tunnel_connect([agent1, agent2], {mode: "combined"})` for multi-party

### Visual Design Implications

- Each card needs a "socket" affordance on the title bar (small dot, plug icon, or similar)
- Dragging from the socket creates a visible cable that follows the cursor
- Dropping on another card's socket completes the connection
- Cables render in the 3D scene — lines/curves between connected cards
- Cables should show state: active (glowing), idle, data flowing (pulse animation?)
- Direction shown via arrow or asymmetric styling
- Multiple cables can come from one card (separated mode)
- Combined mode might show a hub/junction where cables merge

### Schema Impact

The "rooms" table becomes "connections" or "channels" — but the structure is the same. A connection is just a room created implicitly by the wiring gesture. The key difference is how they're created and presented, not how they're stored.

## Updated Mental Model

```
Before:  Room (abstract container) ← agents invited into it
After:   Cable (physical connection) → creates channel implicitly

The cable IS the connection.
The channel is where messages live.
The cable is what the user sees and manipulates.
```

This maps beautifully to svg-terminal's spatial 3D metaphor — everything is a physical object in space. Terminals are cards. Connections are cables. The dashboard is a workbench.
