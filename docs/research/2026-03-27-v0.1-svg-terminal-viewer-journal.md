## Starting Question

Can an SVG contain a live terminal? Short answer: no — SVG is a vector format, not a runtime. But the real question is: can we render terminal output as SVG and update it in real-time? Yes.

## Key Insight

All existing SVG terminal tools (termtosvg, svg-term-cli, ansi-to-svg) are **recorders** — they capture a session and produce a static/animated SVG for playback. Nobody has built a **live** SVG terminal viewer. But the rendering problem is already solved by these tools. Our novel contribution is the live update loop.

## Why SVG Instead of xterm.js/Canvas?

- Vector rendering = infinite zoom, crisp at any scale
- Embeddable as a self-contained file
- Works in `<object>`, `<iframe>`, or standalone
- PHAT TOAD dashboard wants SVG tree visualization — SVG terminals fit naturally
- Read-only viewer avoids all the hard problems xterm.js solves (input, cursor handling, clipboard, selection)

## Why Read-Only Changes Everything

The "SVG is too slow for terminals" argument assumes a full interactive emulator:
- Hundreds of redraws per second
- Input handling
- Full VT100 state machine

A read-only viewer polling `tmux capture-pane` at 150ms:
- Updates ~1-3 lines per frame
- 24-50 `<text>` DOM elements total
- SGR-only output (no cursor movement codes to parse)
- Trivial DOM mutation cost

## Existing Tool Analysis

### What We're Stealing

From **termtosvg** (Python, best SVG patterns):
- `dominant-baseline: text-before-edge` — y=0 means top of text, simple row math
- `textLength` attribute — forces monospace spacing regardless of font
- CSS class colors `.c0`-`.c15` — themeable without touching SVG structure
- Cell dims: `CELL_WIDTH=8`, `CELL_HEIGHT=17` at `font-size: 14px`

From **ansi-to-svg** (Node.js, simplest approach):
- `<rect>` before `<text>` for colored backgrounds
- Skip unstyled whitespace
- `<path>` for underline (more portable than `text-decoration`)

From **svg-term** (TypeScript/React):
- `<defs>` + `<symbol>` + `<use>` deduplication pattern
- Not adopted: React rendering pipeline, horizontal frame stacking

### What We Don't Need
- CSS keyframe animation (we do live updates, not playback)
- Frame stacking (vertical or horizontal)
- Full VT100 emulation (pyte) — `capture-pane -e` already gives us rendered state
- asciinema recording format

## Architecture Decision: Approach 3

Three approaches considered:
1. Express/Fastify + Node.js — npm dependency overhead for 2 endpoints
2. Python + pyte/ansi-to-svg — different runtime from target projects
3. **Bare `http` module, zero deps** — selected

Why #3: Both claude-proxy and PHAT TOAD are Node.js. Zero deps means trivial embedding. The JSON API contract is the integration surface — any server that speaks it can drive the SVG.

## tmux Integration

`tmux capture-pane -p -e` is the key command:
- `-p` prints to stdout
- `-e` includes SGR escape codes (colors, bold, etc.)
- Output is **re-serialized screen state** — only SGR codes, no cursor movement
- This means our parser only handles `ESC[...m` sequences, not the full VT100 spec

For metadata: `tmux display-message -p '#{pane_width} #{pane_height} #{cursor_x} #{cursor_y}'`

## Visibility-Aware Polling

Novel feature for multi-terminal dashboards:
- IntersectionObserver detects viewport entry/exit (zero cost)
- getBoundingClientRect measures rendered character cell size
- Three tiers: >=4x6px → 150ms, <4x6px → 2000ms, offscreen → stopped
- Handles zoom naturally — pinch-zoom a tiny terminal and it auto-upgrades

## Open Questions
