## Implementation Discoveries

### Default pane ID
tmux `capture-pane -t session:pane` expects the pane index (`0`, `1`, `2`), not the pane ID (`%0`, `%103`). The `%`-prefixed IDs are unique identifiers shown by `list-panes`, but the `:pane` syntax in target strings uses the index. Fixed default from `%0` to `0`.

### textLength is wrong for live rendering
The spec originally called for `textLength` attribute on `<text>` elements to enforce monospace spacing (following termtosvg's pattern). In practice, this causes characters to stretch/compress to fill the specified width, producing wildly uneven spacing when the browser's font metrics don't match the assumed 8px cell width.

**What termtosvg gets away with:** It generates static SVGs with known font metrics. The viewer is a fixed-size image, not a responsive element.

**What we need:** Dynamic rendering where the SVG scales to its container. The font metrics must be measured at runtime.

**Solution:** Remove `textLength`. Instead:
1. Add a hidden `<text>MMMMMMMMMM</text>` element
2. Measure its `getBBox().width / 10` to get actual character cell width
3. Give each `<tspan>` an explicit `x` attribute based on `charOffset * measuredWidth`
4. Set `viewBox` using measured width

This approach is robust — works regardless of what font the browser loads.

### Chrome blocks fonts in standalone SVGs
**This was the biggest surprise.** When an SVG is served as `image/svg+xml` and loaded directly in the browser or via `<object>`, Chrome's security model blocks ALL external resource loading — including `@font-face` with `data:` URIs.

**Investigation path:**
1. Embedded FiraCode Nerd Font as base64 woff2 data URI in SVG `<style>` — didn't work
2. Created diagnostic page (`font-test.html`) showing same font in HTML vs SVG context
3. Result: Font renders perfectly in HTML `<div>`, fails silently in inline `<svg>` within standalone SVG file
4. BUT: Font works in inline SVG within an HTML document

**Root cause:** Chrome treats standalone SVGs as "SVG as an image" context which has restricted capabilities. This is documented but not obvious.

**Solution:** Serve terminals via HTML wrapper (`terminal.html`) that:
- Declares `@font-face` in the HTML `<style>` (which Chrome allows)
- Contains the SVG inline (not as a separate resource)
- Uses `document.fonts.ready` to wait for font load before measuring

**Architectural impact:** Dashboard changed from `<object data="/terminal.svg">` to `<iframe src="/terminal?session=X">`. The standalone `.svg` file remains for contexts where fonts don't matter (or where the consuming page provides its own font). The `.html` wrapper is the primary rendering path.

### Font subsetting
FiraCode Nerd Font Mono Regular is 1.4MB. Subsetting to terminal-essential characters (ASCII + box-drawing + block elements + powerline + prompt symbols + arrows + Latin-1) produces a 31KB woff2. The subset is embedded as base64 in the HTML, adding ~43KB to the page. Acceptable for a terminal viewer.

Used `fonttools` with `pyftsubset` — the Python font toolchain. Characters included:
- ASCII printable (U+0020-U+007E)
- Latin-1 supplement (U+00A0-U+00FF)
- Box drawing (U+2500-U+257F)
- Block elements (U+2580-U+259F)
- Powerline (U+E0A0-U+E0B3)
- Prompt/UI symbols: ❯ ✔ ✘ ● ○ ■ ▶ ◀ … • arrows

Key discovery: Standard Fira Code from Google Fonts does NOT contain U+276F (❯). Only the Nerd Font patched version has it.
