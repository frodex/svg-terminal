## Actual Breakthroughs (Confirmed Working)

### 1. Locked-to-Ring Rotation — The Lazy Susan

**The problem:** Getting cards to rotate as if physically glued to a spinning disc. Spent many iterations getting this wrong — counter-rotating around wrong origins, using wrong reference frames, etc.

**What actually solved it:**

The math is simple once you get the reference frame right:
```js
// θ_i = starting angle for card i on the ring
var theta = -Math.PI / 2 + (2 * Math.PI * i) / N;

// Position: rotate θ by the ring's accumulated angle
var spunTheta = theta + ringAngle;
var px = cx + Math.cos(spunTheta) * radius;
var py = cy + Math.sin(spunTheta) * radius;

// Rotation: local orientation + ring spin, around card's OWN center (not ring center)
var localRot = theta - Math.PI / 2;  // top toward center, upright at bottom
var finalRotDeg = (localRot + ringAngle) * 180 / Math.PI;

el.style.transformOrigin = '50% 50%';
el.style.transform = 'rotate(' + finalRotDeg + 'deg)';
```

**Key insight:** `localRot = theta - PI/2` (not `+ PI/2`). The sign determines whether the upright card is at 6 o'clock (bottom, correct) or 12 o'clock (top, wrong). Getting this backwards wasted significant time.

**Key insight:** Use `transform-origin: 50% 50%` (card center) and compute the final position mathematically. Do NOT use `transform-origin` set to the ring center — it works for single cards but breaks when combined with other transforms.

**Key insight:** Each card starts at a different angle on the ring, so each has a different `localRot`. At any given moment, the card currently at the bottom of the ring is upright. As the ring spins, different cards take turns being the upright one. This is the lazy susan behavior.

**Validation test:** Set up a single-card test with 0°, 90°, 180°, 270° buttons. The card at the bottom must always be upright. Then expand to 5 cards and verify that at 72° rotation intervals, each card looks identical when it passes through the bottom position.

### 2. CSS 3D vs Faked 3D

**The problem:** Ring tilt (making the ring plane tilt toward/away from the viewer) was implemented as fake 2D math — squishing Y coordinates and scaling cards. The wireframe ring tilted via real CSS `rotateX` but the cards used different math. They never aligned.

**What actually solved it:** Put the cards INSIDE the same container as the wireframe ring. Apply `transform-style: preserve-3d` to the container. Apply `rotateX/Y/Z` to the container. The cards are positioned in 2D on the flat circle, and the container's 3D transform tilts everything together — wireframe and cards move as one.

```css
.ring-container {
  transform-style: preserve-3d;
}
```

```js
container.style.transform = 'rotateX(' + tiltX + 'deg) rotateY(' + tiltY + 'deg)';
```

Cards just need 2D positioning (`left`, `top`) inside the container. The browser handles all the 3D projection.

**Anti-pattern:** Don't fake 3D with scale and Y-offset math when CSS can do real 3D. The fake version will never match the wireframe's CSS 3D projection. I wasted significant time on this approach.

### 3. Card Tilt Must Be Uniform

**The problem:** Card tilt X was modulated by `sin(spunTheta)` to create "organic variation." In practice, every card had a different tilt, which looked broken — not organic.

**What actually solved it:** Apply the tilt value directly. `rotateX(30deg)` means 30° for every card, period. Organic variation should come from the ring's rotation and layout, not from per-card tilt randomization.

### 4. Font Rendering in SVG — 2x Scale Trick

**The problem:** CSS3DRenderer applies `matrix3d()` transforms that cause Chrome to rasterize text at the wrong resolution, making SVG terminal text blurry.

**What actually solved it:** Render the terminal DOM elements at 2x CSS size (640×496 instead of 320×248) and set `CSS3DObject.scale.setScalar(0.5)`. Chrome rasterizes at the larger size, then the 3D scale reduces it — resulting in crisp text.

### 5. CSS3DRenderer Billboarding Direction

**The problem:** CSS3DObjects face in the -Z direction. Using `Matrix4.lookAt(terminal, camera, up)` makes them face backwards (text mirrored).

**What actually solved it:** Swap the arguments: `Matrix4.lookAt(camera, terminal, up)`. This makes the matrix face FROM camera TO terminal, which when applied to the object makes it face the camera.

### 6. Click Detection Through CSS3DRenderer

**The problem:** CSS3DRenderer creates a wrapper div that intercepts all mouse events. Click events on terminal cards don't bubble through.

**What actually solved it:** Add a click handler on the renderer's DOM element. On click, iterate all terminal elements, check `getBoundingClientRect()` overlap with click coordinates, and pick the closest one (using projected Z for depth sorting).

## Why Things Ended Up This Way

### Why 2D positioning with 3D container, not full 3D positioning

We tried three approaches for ring tilt:
1. **Fake 2D** — squish Y axis + scale cards for depth. Failed because the wireframe ring used CSS `rotateX` (real 3D) and the cards used different math. They never aligned.
2. **Full CSS 3D per card** — `translate3d(x, y, z)` on each card. Works but makes rotation math (locked-to-ring, upright mode) extremely complex because you're composing 3D rotations.
3. **2D positioning inside a 3D container** — cards use simple `left`/`top` in 2D, the container's `rotateX/Y/Z` tilts everything. Won because it separates concerns: the positioning math stays simple (2D circle), the 3D effect is a single container transform.

Approach 3 won because it's composable — you can change the ring tilt without touching the card positioning code, and vice versa.

### Why transform-origin 50% 50% and not ring center

Early locked-to-ring attempts used `transform-origin` set to the ring center, so `rotate(ringAngle)` would orbit the card around the center. This worked in isolation (the single-card test was perfect) but broke when combined with card tilt, face-camera counter-rotation, or mode switching. The problem: CSS applies all transforms relative to the same origin, so adding `rotateX(30deg)` for card tilt would rotate around the ring center, not the card center.

The solution: compute the card's final screen position mathematically (`cos`/`sin` on the spun angle), place it there with `left`/`top`, and use `transform-origin: 50% 50%` for all rotations. This means rotations are always relative to the card itself, which is predictable and composable.

### Why `localRot = theta - PI/2` and not `+ PI/2`

In screen coordinates, Y increases downward. The bottom of the circle (6 o'clock) is at `theta = PI/2` (sin = +1 = positive Y = downward). We want the card at the bottom to be upright (rotation = 0). So: `localRot = theta - PI/2`. At `theta = PI/2`: `localRot = 0`. At `theta = -PI/2` (top): `localRot = -PI` (upside down). This matches the lazy susan: the card at the bottom is always right-side up.

The `+ PI/2` version puts the upright card at the top (12 o'clock), which is mathematically valid but doesn't match the user's mental model of "placed at 6 o'clock, disc spun."

### Why the visual companion is the right tool for this work

We tried implementing 3D layouts directly in the dashboard.mjs (Three.js CSS3DRenderer) and iterating there. This was slow — every change required restarting the server, hard refreshing, fighting with Three.js import maps, debugging `setViewOffset`, etc. The issues were obscured by the complexity of the full stack.

Moving to the brainstorm visual companion (single-file HTML served by the superpowers tool) was a turning point. Changes are instant (save → browser auto-reloads). No build step. No Three.js dependency. Pure CSS 3D transforms are simpler and more predictable than CSS3DRenderer. We could iterate on rotation math, layout, and controls 10x faster.

The design studio (ring-mega.html) became the definitive visual development tool — all the control sliders let the user explore the parameter space interactively. The plan is to take confirmed-good settings from the studio and implement them in the production dashboard.

### Why separate ring and card tilt

Ring tilt and card tilt do fundamentally different things:
- **Ring tilt** changes the spatial arrangement — the ring plane rotates in 3D space. All cards move together.
- **Card tilt** changes each card's individual orientation within the ring plane. Cards can lean forward/back independently of the ring.

These compose: you can tilt the ring 45° on X AND tilt each card 20° on Y. If they were a single control, you couldn't achieve this.

The "Face Camera" toggle adds a third layer — it counter-rotates cards against the ring tilt so they always face the viewer. This is another composition: ring tilt for spatial arrangement + face camera for readability.

### Why From/To oscillation instead of just a single value

A single tilt value gives you a static pose. The From/To + Speed pattern lets you define a range and the system oscillates between them. This is how you create the "attract mode" motion the user wanted — terminals slowly rocking, rings gently tilting back and forth. It's more expressive than a single value + amplitude because you can set asymmetric ranges (e.g., tilt from -10° to +40°).

## Non-Obvious Implementation Details — Don't Remove These

Things that look weird in the code and you might be tempted to "fix" but are there for a reason:

### Terminal panels are 640×496 CSS pixels, scaled to 0.5 in Three.js

**Why it's there:** CSS3DRenderer applies `matrix3d()` transforms to DOM elements. Chrome rasterizes the element at its CSS pixel size BEFORE applying the 3D transform. If the element is 320×248 and the 3D transform scales it up, the text is blurry — Chrome stretched a 320px raster to fill a larger space.

**The fix:** Make the DOM element 2x larger (640×496) with 2x font sizes, dot sizes, border-radius, etc. Then set `CSS3DObject.scale.setScalar(0.5)` so it appears at the correct 3D size. Chrome rasterizes at 640px (crisp), the 3D scale halves it, result is sharp.

**If you remove this:** Terminal text will be blurry in the 3D dashboard. The 2x sizing is in `dashboard.css` (`.terminal-3d { width: 640px; height: 496px; }`) and the 0.5 scale is in `dashboard.mjs` where CSS3DObjects are created.

### The embedded font is base64 in terminal.svg

**Why it's there:** Chrome blocks all external resource loading (including `@font-face` with data URIs) inside standalone SVGs served as `image/svg+xml`. The font works in the HTML wrapper (`terminal.html`) but NOT in the standalone `.svg`. We kept the font in the SVG anyway because it works when the SVG is loaded via `<object>` tags (which is how the dashboard loads them), and because the HTML wrapper exists as an alternative.

**The font is a subset:** FiraCode Nerd Font Mono, stripped to ~300 characters (ASCII + box-drawing + powerline + prompt symbols). 31KB woff2. The full font is 1.4MB. The subset was created with Python `fonttools`/`pyftsubset`.

**If you remove this:** The `❯` prompt character and some box-drawing characters won't render correctly. PuTTY users need FiraCode Nerd Font Mono installed locally.

### `CELL_W` and `CELL_H` are measured at runtime, not hardcoded

**Why it's there:** Originally `CELL_W = 8` and `CELL_H = 17` were hardcoded based on termtosvg's values for DejaVu Sans Mono at 14px. This caused massive character spacing issues because different fonts and browsers render at different widths. The `textLength` approach (from the spec) was even worse — it stretched characters to fit.

**The fix:** A hidden `<text>MMMMMMMMMM</text>` element is measured via `getBBox()` on page load. `CELL_W = bbox.width / 10`, `CELL_H = bbox.height`. The SVG viewer uses `document.fonts.ready` (in the HTML wrapper) to wait for the font to load before measuring.

**If you remove this:** Character spacing will break. Each `<tspan>` is positioned at `charOffset * CELL_W` — if CELL_W is wrong, characters overlap or spread apart.

### Each `<tspan>` has an explicit `x` attribute

**Why it's there:** Originally used `textLength` on the `<text>` element to enforce monospace spacing. This caused characters to stretch/compress unpredictably. Switching to explicit `x` per `<tspan>` gives precise control — each colored span starts at exactly the right pixel position.

**If you remove this:** Characters within a line will lose their alignment. The browser's default `<tspan>` positioning flows inline and won't maintain the monospace grid.

### The dashboard uses `<object>` tags, not `<iframe>` or `<img>`

**Why `<object>` won:** We tried three approaches:
1. `<img src="terminal.svg">` — no JavaScript execution, SVG is static
2. `<iframe src="/terminal">` — works but heavier, each iframe is a full document
3. `<object type="image/svg+xml" data="/terminal.svg">` — SVG JavaScript runs, lighter than iframe

We briefly switched to `<iframe>` for font loading (standalone SVG can't load fonts, HTML can) but reverted because the font issue is cosmetic and `<object>` is simpler.

### Click detection uses bounding rect hit testing, not event bubbling

**Why it's there:** CSS3DRenderer creates a container div that intercepts all pointer events. Click events on the terminal cards inside don't bubble up to where we can catch them. The solution: add a `click` handler on the renderer's DOM element, then for each click, iterate all terminal elements and check `getBoundingClientRect()` overlap. The closest terminal (by projected Z) wins.

**If you remove this:** Clicking terminals in the 3D dashboard won't work.

### Pane title comes from tmux `#{pane_title}`, not the session name

**Why it's there:** Claude-proxy sets the terminal title via OSC 0 escape sequences (`\x1b]0;...\x07`). tmux stores this as the pane title. The `/api/pane` response includes a `title` field from `tmux display-message -p '#{pane_title}'`. The dashboard shows this instead of the raw session name (e.g., "Debug persistent test failures" instead of "cp-greg_session_001").

**If you remove this:** The dashboard will show raw session names instead of descriptive task names.

### The design studio files are standalone HTML, not served by server.mjs

**Why:** The superpowers brainstorm companion serves single HTML files and auto-reloads on change. This is much faster for visual iteration than going through the Node server. The prototypes use pure CSS 3D (no Three.js), which is simpler and more predictable for exploring layouts. Once a layout is finalized, it gets reimplemented in dashboard.mjs with Three.js.

## Guidance for Next Session

### How to approach changes to the design studio

**Good patterns:**
1. **Test in isolation first.** Before modifying the mega control panel, create a minimal test page with just one card/one ring and the specific feature you're testing. Confirm it works, THEN integrate.
2. **Use Puppeteer to self-verify.** Don't ask the user to check things you can check yourself. Take screenshots at specific angles, read computed transforms, verify the math.
3. **Preserve what works.** Before modifying a working file, save a copy (`cp ring-mega.html ring-mega-saved.html`). The user explicitly asked for this pattern.
4. **Clean rewrites over patches.** When a function has been patched 3+ times, rewrite it from scratch with clear comments. The `positionCard` function should be the canonical reference.
5. **The lock/sync pattern for sliders.** From/To pairs with a lock button and speed for oscillation — this is the established control pattern. New parameters should follow it.
6. **Real CSS 3D for spatial effects.** Use `transform-style: preserve-3d` on containers, position children in 2D, let the container's 3D transform handle projection. Don't fake it with scale/offset math.

**Anti-patterns:**
1. **Don't modulate per-card values by ring position** unless the user specifically asks for organic variation. Uniform values are predictable and debuggable.
2. **Don't change the meaning of controls.** If "push" was scaling the scene wrapper and working, don't silently change it to per-card perspective projection. Add a new control instead.
3. **Don't accumulate mode transition/blending code** unless specifically asked. Instant mode switching is cleaner and easier to debug. The auto-cycle blending caused multiple bugs where modes were fighting.
4. **Don't use `transform-origin` set to the ring center** for card rotation. It works for the simple case but breaks when combined with other transforms, card tilt, or face-camera. Use `transform-origin: 50% 50%` and compute final position mathematically.
5. **Don't guess rotations — verify with screenshots.** When the user says "this is wrong," take Puppeteer screenshots at known angles and compare to expected behavior. Don't keep trying different sign conventions hoping one works.
