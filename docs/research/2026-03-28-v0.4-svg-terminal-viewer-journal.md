## 3D Mouse Controls

Implemented full 3D viewport controls:
- **Drag:** orbit camera around look target
- **Shift+drag:** dolly X/Y (translate camera + target in screen-aligned plane)
- **Ctrl+drag:** rotate around world origin (or center of mass of focused terminals)
- **Scroll:** zoom (FOV)
- **Shift+scroll:** dolly Z (move camera along view direction)

Key implementation detail: `syncOrbitFromCamera()` derives orbitAngle/orbitPitch/orbitDist from the camera's actual position when orbit starts. Without this, starting an orbit from a focused camera position would snap the camera to a default position.

Click-vs-drag is distinguished by a 5px dead zone (`dragDistance`). Click events fire normally under the threshold.

## Scene Lighting

- Background: medium gray gradient (#a5a5ae → #c2c2c8) — chosen iteratively with user
- Specular: intensity 0.4 on card faces using panel normal dot product with light direction
- Shadows: floor at Y=-300, shadow opacity 0.9 max, radial-gradient ellipse with blur
- Shadow offset follows LIGHT_DIR for directional shadow placement

## Multi-Focus Terminals

Ctrl+click adds terminals to a focused set. Layout arranges them in a grid:
- 1 terminal: center (same as original single-focus behavior with 1:1 pixel mapping)
- 2+: grid layout, camera pulls back to fit all

Yellow neon border indicates which terminal receives keyboard input (`input-active` class). Click between focused terminals to switch input target. Esc unfocuses all.

### Event Routing Was The Hard Part

The biggest challenge was ctrl+click event routing. Three separate event paths competed:
1. **onMouseUp** (document) — `handleCtrlClick` finds terminals by bounding rect
2. **onSceneClick** (renderer.domElement) — click handler for 3D scene
3. **Thumbnail click handler** (sidebar items) — direct click on sidebar thumbnails

Problems encountered:
- `e.preventDefault()` on mousedown suppresses the subsequent click event
- CSS3DRenderer terminals have overlapping bounding rects in screen space
- mouseup fires BEFORE click, so timestamp guards between them race
- The sidebar thumbnails and 3D scene are different DOM trees but overlap spatially

Final solution:
- Ctrl+click on 3D cards: handled in `onMouseUp` via `handleCtrlClick`, skips already-focused terminals
- Ctrl+click on thumbnails: handled by thumbnail's own click handler with `stopPropagation`
- `mouseDownOnSidebar` flag prevents `handleCtrlClick` from double-firing when clicking thumbnails
- `lastAddToFocusTime` timestamp blocks `focusTerminal` from replacing focus within 200ms of `addToFocus`
- `onSceneClick` completely ignores ctrl events (returns early)

## Fly-In 3D Rotation

Cards spawn at random 3D angles and slerp toward face-camera during fly-in. Focused terminal captures its current quaternion (`focusQuatFrom`) and slerps to camera orientation over the morph duration. Previously focused terminal morphs back to ring position when a new terminal is clicked.

## Help Panel

Frosted glass help panel (backdrop-filter blur) triggered by ? button in upper-left corner. Shows all keyboard/mouse controls. Toggle with ? key or button click.

## Text Crispness

Discussed extensively but not yet solved to full satisfaction. Key findings:
- Chrome rasterizes DOM elements into bitmaps before applying 3D transforms
- HTML text is sharper than SVG text under 3D transforms
- The Nx scale trick (4x CSS size, 0.25 3D scale) improves but doesn't eliminate softening
- User rejected HTML overlay crossfade approach — SVG is the target format for universality
- Rendering differences between Chrome and Edge are significant
- Current state: acceptable for overview, could be better for focused view
