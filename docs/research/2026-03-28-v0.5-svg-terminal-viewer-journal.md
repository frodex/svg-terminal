# SVG Terminal Viewer Journal v0.5
**Date:** 2026-03-28
**Session:** Interactive Terminals — WebSocket Streaming
**Status:** In Progress
**Preceding:** 2026-03-28-v0.4-svg-terminal-viewer-journal.md
**Changes from v0.4:** WebSocket streaming, direct keystroke capture, async tmux

## Architecture Decision: WebSocket over SSE

Considered two approaches for real-time terminal streaming:

**SSE + POST:** Server-Sent Events for output, HTTP POST for input. Zero new dependencies. Trivially simple server (30 lines). But requires two channels and POST has per-request HTTP overhead for keystrokes.

**WebSocket (ws package):** Single bidirectional connection. The ws package was already in node_modules as a transitive dependency of puppeteer — no new dependency added to the tree. Cleaner single-connection model. Ready for future features (resize, mouse events) without adding endpoints.

Chose WebSocket because ws was already installed and the bidirectional channel is cleaner.

## Server Changes

### Async tmux execution
All tmux commands converted from `execFileSync` (blocking) to async `execFile` wrapped in `tmuxAsync()`. Without this, the 30ms WebSocket polling would block the event loop and stall all connections.

### capturePane helper
Extracted from handlePane into a reusable function. Both the HTTP /api/pane endpoint and WebSocket handler share it. Returns { width, height, cursor, title, lines }.

### Server-side diff
Diff logic moved from client to server. `diffState(prev, curr)` compares line-by-line via JSON.stringify. Returns null (no change), full screen (first message or dimension change), or delta (changed lines only). This is the same algorithm the client used for polling, just server-side now.

### WebSocket handler
`handleTerminalWs(ws, session, pane)` — one per connection:
1. Sends full screen state immediately on connect
2. Polls tmux at 30ms via setInterval
3. Diffs against last state, pushes only changes
4. Receives input messages, forwards to tmux via send-keys
5. After input, re-captures after 5ms delay for immediate feedback
6. Cleans up poll timer on close/error

### Expanded key whitelist
`isAllowedKey()` now accepts: Enter, Tab, Escape, BSpace, DC, IC, Space, arrows, Home, End, PgUp, PgDn, F1-F12, and any C-[a-z] Ctrl combo.

## SVG Client Changes

Added WebSocket client as primary connection mode. On load, attempts WebSocket to /ws/terminal. If successful, polling stops. If WebSocket fails or disconnects, falls back to HTTP polling automatically. EventSource-style auto-reconnect after 2 seconds.

New `allLines[]` array tracks full terminal state for delta updates — needed because delta messages only include changed lines, but rebuildBgLayer needs all lines.

## Dashboard Changes

### Direct keystroke capture
Replaced the input bar text field with a document-level keydown handler. When a terminal is focused, ALL keyboard events are captured and forwarded to tmux. Browser shortcuts (Ctrl+T/W/N/R, F12, Alt+F4) are excluded.

Key translation table maps browser KeyboardEvent.key names to tmux send-keys arguments. Ctrl combos are synthesized as "C-x" format.

### Per-terminal input WebSocket
Each focused terminal opens its own WebSocket for sending keystrokes. Opened on focus, closed on unfocus. Falls back to POST /api/input if WebSocket isn't connected.

### Paste support
Document-level paste handler sends clipboard text as literal keys to the active terminal.

## Latency Budget
```
Keystroke → WebSocket send:       ~0.5ms
Server receives + tmux send-keys: ~1-2ms
tmux processes input:             ~1-5ms
5ms setTimeout:                   ~5ms
Server captures pane:             ~1-3ms
Diff + WebSocket push:            ~0.5ms
Browser receives + renders:       ~1-5ms
                                  --------
Total:                            ~10-22ms (was 150ms)
```
