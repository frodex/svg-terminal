## Problem

After implementing cross-browser resize sync changes (guard removal, re-layout trigger, resize lock), the dashboard started showing "Connection lost — retrying" on all cards, cycling between 500/502/530 errors every few seconds.

## Timeline

1. Session started with brainstorming, design docs, implementation plan
2. Tasks 1-4 implemented: guard removal, re-layout trigger, server resize lock, PRD update
3. All 57 automated tests passed (19 server + 16 auth + 22 E2E)
4. Puppeteer tests confirmed guard removal fixed the resize bugs
5. User tested in browser — cards started dying
6. Multiple debug cycles: reverted guard, added debug logging, restarted server
7. Cards continued cycling even with guard restored
8. Investigated WebSocket message flood (claude-proxy sending all-53-lines deltas at 15Hz)
9. Investigated Chrome re-compositing theory
10. Eventually found: server was at 55% CPU, 170 TCP connections, 32 to claude-proxy
11. 4 different LAN IPs connected, 24 connections each
12. User closed browser — connections didn't drop
13. Moved server to port 3201 — clean start, 27 connections, stable

## Root Cause

**Version mismatch between stale browser clients and updated server.**

User had 2-3 browser instances (desktop + phone) running the dashboard for 4 days. When we changed server.mjs, dashboard.mjs, and terminal.svg today, those existing browser sessions continued running the OLD code. The old client code interacting with the new server code caused connection instability.

Evidence:
- 4 LAN IPs connected (user confirmed 2-3 were their own devices)
- Connections persisted even after user closed ONE browser
- Moving to a new port (3201) with fresh code = immediately stable
- 27 connections, 0% growth, no cycling

## What We Got Wrong

1. **Jumped to conclusions repeatedly** — assumed Chrome re-compositing, assumed message flood, assumed code bug. Violated the PHAT TOAD rule: "understanding before action."
2. **Made multiple code changes during debugging** — reverted guard, added debug logging, modified updateCardForNewSize. Each change added confusion about what was actually running.
3. **Didn't check who was connected** — took too long to look at `lsof` and discover 4 IPs.
4. **Didn't consider version mismatch** — serving static .mjs files without cache busting means old clients run old code indefinitely.

## Secondary Issues Discovered

### /api/pane doesn't route to claude-proxy (Task #19)
The HTTP `/api/pane` endpoint calls `capturePane()` for ALL sessions. Claude-proxy-only sessions (cp-AARON, cp-MARKTEXT-BROWSER, etc.) don't exist in local tmux, so this returns 500. The WebSocket path has an `isLocal` check — the HTTP path doesn't. This causes the fallback poll to fail permanently for proxy sessions.

### Connection leak
32 WebSocket proxy bridges to claude-proxy remained open even after all browsers disconnected. The proxy cleanup code looks correct (close events propagate both directions), but connections weren't cleaning up in practice. Needs investigation.

### No cache busting on static assets
dashboard.mjs, terminal.svg served without version hash or cache-control headers. Browsers can run arbitrarily old code against the current server.

## Decisions

- **Server-pushed reload** (Task #18) — when code changes, push reload to all connected browsers
- **Move to port 3201** temporarily to isolate from stale clients
- **The focused-card guard** — currently restored (local stash). The committed code has it removed. Need to reconcile in next session.
