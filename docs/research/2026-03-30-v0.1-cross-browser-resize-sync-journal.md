## Problem Statement

When two browsers are viewing the same svg-terminal dashboard, and one user resizes a terminal (changes rows/cols via +/-, alt+scroll, optimize-fit), the other browser needs to instantly reflect the new terminal dimensions. Currently there's a race condition where both browsers can send resize commands that overwrite each other.

---

## Current Architecture — How Resize Propagates

### The resize chain (single browser):

```
User action (+/-, alt+scroll, optimize-fit)
  → dashboard.mjs sends { type: 'resize', cols, rows } via WebSocket
  → server.mjs handleTerminalWs receives it
  → server calls `tmux resize-window -t session -x cols -y rows`
  → tmux session changes size
  → server re-captures pane after 10ms delay
  → server pushes `screen` message with new width/height to THIS WebSocket only
  → dashboard._screenCallback calls updateCardForNewSize()
  → card DOM reshapes
```

### How a second browser currently discovers the resize:

**Path A — WebSocket screen data (fast, ~30ms):**
Each browser has its own WebSocket to `/ws/terminal?session=X&pane=0`. The server polls tmux at 30ms intervals per connection. After User 1 resizes, the next poll for User 2's WebSocket will capture the new tmux dimensions. The `screen` message includes `width` and `height`. User 2's `_screenCallback` calls `updateCardForNewSize()`.

**Path B — Session list poll (slow, 5 seconds):**
`refreshSessions()` polls `/api/sessions` every 5 seconds. The session list includes `cols` and `rows` from `tmux list-sessions`. This calls `updateCardForNewSize()` for any changed dimensions.

### So User 2 DOES get the resize within ~30ms via Path A. What actually breaks?

---

## Question 1: What Is Actually Breaking?

The 30ms WebSocket path should propagate dimension changes quickly. So what's the real failure mode?

### Hypothesis A: Latency is the problem
User 2 takes too long to see User 1's resize. But 30ms should be fast enough — that's imperceptible. **Unlikely to be the core issue.**

### Hypothesis B: Fighting / race condition
Both browsers can send `{ type: 'resize' }` messages. If User 1 resizes to 100x30 and User 2 resizes to 80x24 within the same 30ms window, they'll overwrite each other. The tmux session oscillates. Both browsers see the other's resize, potentially triggering their own corrective resize, creating a feedback loop.

**This is the most likely failure mode.**

### Hypothesis C: Authority problem
No browser knows it should defer. When User 2's `updateCardForNewSize()` fires from an incoming `screen` message, does it trigger a new resize command back to the server? If so, that's the feedback loop.

### Hypothesis D: Focused-card guard blocks the update
`updateCardForNewSize()` at line 1587: `if (t.dom.classList.contains('focused')) return;` — this means if User 2 has the same terminal focused, the card DOM WON'T update even though the tmux dimensions changed. The base values update silently, but the visual stays stale until unfocus.

**Needs user input: which of these is the observed behavior?**

## Code Path Analysis

### What triggers a resize command FROM the browser?

Scanning dashboard.mjs for outbound resize messages:

1. **+/- header buttons** — change cols/rows by ±4/±2, send resize via WebSocket
2. **Alt+scroll (font zoom)** — changes cols/rows, sends resize
3. **Optimize-fit (⊡ button)** — calculates cols/rows to fill card, sends resize
4. **Alt+drag resize** — changes card DOM size, recalculates cols/rows, sends resize

All of these send `{ type: 'resize', cols, rows }` through `sendInput()` which routes through the SVG `<object>` contentWindow WebSocket.

### What RECEIVES a resize and could re-trigger?

1. `_screenCallback` receives `screen` messages with `width`/`height` → calls `updateCardForNewSize()`
2. `updateCardForNewSize()` updates `screenCols`, `screenRows`, reshapes card DOM
3. `updateCardForNewSize()` does NOT send a resize command back — it only updates the local card

**Finding: There is no automatic feedback loop in the current code.** `updateCardForNewSize` is receive-only. The problem is NOT a code feedback loop.

### Then what fights?

The fight happens when TWO HUMANS are both resizing the same terminal simultaneously. Each sends resize commands. The tmux session flip-flops between their desired sizes. Each browser sees the other's resize come in via the 30ms poll, reshapes its card, but the user is still dragging/scrolling to resize — so they send another resize command overwriting what just arrived.

**This is a human-level race, not a code bug.**

---

## Question 2: What Should the Resolution Model Be?

Given that the problem is concurrent human resize intent, we need a conflict resolution strategy:

### Option A: Last-write-wins (current behavior)
The most recent resize command wins. Both browsers converge to whatever the last command was. Simple. But frustrating when two users fight.

### Option B: Authority model
One browser is the "owner" of a terminal's size. Only the owner can resize. Other browsers are read-only for resize (they still see the terminal content, just can't change its dimensions). Ownership could be:
- First-to-focus owns resize
- Explicit claim ("take control" button)
- Parent/child role (ties into follow-along mode, Task #8)

### Option C: Lock-on-resize
When a user starts resizing (mousedown on alt+drag, or first +/- click), the server locks that terminal's dimensions for that browser. Other browsers' resize commands are rejected until the lock releases (mouseup, or timeout). The lock holder's resize propagates instantly to all others.

### Option D: Optimistic with conflict notification
Last-write-wins, but when Browser B's resize is overwritten by Browser A within a short window, Browser B gets a notification: "Terminal resized by another user." No lock, no authority — just visibility.

**Awaiting user input on which direction feels right.**

## Related: The Focused-Card Guard Problem

Separate from the multi-user race, there's a single-user issue:

`updateCardForNewSize()` line 1587 skips DOM updates when the card is focused. This was intentional — "the card is the user's chosen window, +/- changes font size inside it." But when ANOTHER user resizes the tmux session, the focused card should probably update to reflect the new dimensions, since the terminal content is now different.

This is a case where the focused-card guard was designed for single-user and needs to be reconsidered for multi-user.
