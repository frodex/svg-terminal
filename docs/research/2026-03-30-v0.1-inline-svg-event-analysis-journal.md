**Date:** 2026-03-30
**Session:** Terminal architecture decision — inline SVG vs `<object>` embed
**Status:** Direction Confirmed — contentWindow bridge approach selected after testing all three options and two rounds of architectural review.
**Preceding:** none
**Changes from v0.0:** Initial analysis of all event handlers and three architecture options

---

## Why This Analysis Exists

We have a dual-WebSocket bug: when a terminal is focused, the dashboard opens its own WebSocket (`t.inputWs`) for input/scroll, while terminal.svg inside the `<object>` opens a separate WebSocket for rendering. For local tmux sessions this works via shared server-side state. For proxied claude-proxy sessions, the two connections are independent — scroll sent on one doesn't affect the other.

Three approaches to fix this. Each changes how the terminal content lives in the DOM, which affects all event routing. This document traces every interaction through the current system and evaluates how each approach changes or breaks it.

```
│  │  │  └─────────────────────────────────────────────┘   │  │    │
│  │  └────────────────────────────────────────────────────┘  │    │
│  └──────────────────────────────────────────────────────────┘    │
│                                                                 │
│  Dashboard JS (document-level event listeners)                  │
│    - keydown → sendInput via WS #2 (inputWs)                   │
│    - wheel → scrollBy via WS #2                                 │
│    - mousedown/up/move → drag, hit test via getBoundingClientRect│
│    - text selection → screenToCell via <object>.contentDocument  │
│                                                                 │
│  WebSocket #2 (inputWs) ← opened on focus, closed on unfocus   │
│    - sends: input, scroll, resize                               │
│    - receives: screen/delta for copy/paste screenLines          │
└─────────────────────────────────────────────────────────────────┘
```

**The core constraint:** `<object>` has `pointer-events: none` (dashboard.css line 143). Without this, the `<object>` element swallows mouse events before the CSS3DRenderer container can intercept them. The 3D scene's click/drag/orbit handling requires that ALL pointer events reach the renderer container.

**The dual-WebSocket consequence:** Because the `<object>` creates an isolated document with its own WebSocket, the dashboard must open a SECOND WebSocket to send input/scroll. These two connections are independent.

## User Interactions Traced End-to-End

### 1. TYPING (keydown → terminal)

```
User presses 'a'
  → document keydown handler (line 2268)
  → checks: has activeInputSession? has focusedSessions?
  → not in SPECIAL_KEY_MAP, not ctrl, not alt
  → t.sendInput({ type: 'input', keys: 'a' })
  → sends via t.inputWs (WS #2)
  → server.mjs receives on WS #2
  → LOCAL: tmuxAsync('send-keys', ...) → tmux processes → WS #1 gets new screen
  → PROXY: bridge forwards to claude-proxy → response comes back on WS #2's bridge
           BUT terminal.svg is on WS #1's bridge — sees nothing ◄── THE BUG
```

**Wait — this reveals something.** Typing works on proxied sessions (user confirmed). That means WS #1 (terminal.svg) IS seeing updates. How?

Because when you type, the terminal output changes, and claude-proxy sends the update on ALL connections to that session. The `input` message goes on WS #2's bridge, but the `delta` response goes to ALL connected clients — including WS #1's bridge. Claude-proxy broadcasts to all subscribers.

**So why doesn't scroll work?** Because `scroll` changes what PART of the buffer is shown. It's connection-specific — claude-proxy sends the scrolled viewport only to the connection that requested it, not to all subscribers.

### 2. SCROLL (wheel → terminal)

```
User scrolls wheel on focused card
  → renderer.domElement wheel handler (line 1095)
  → getScrollAction() returns 'scrollContent'
  → t.scrollBy(step)
  → t.scrollOffset += step
  → t.sendInput({ type: 'scroll', offset: N })
  → sends via t.inputWs (WS #2)
  → server.mjs receives on WS #2
  → LOCAL: setScrollOffset() ← SHARED STATE, affects WS #1 too ← WORKS
  → PROXY: bridge forwards to claude-proxy
           → claude-proxy returns scrolled screen on WS #2's bridge
           → WS #1's bridge still shows live (unscrolled) view ← BROKEN
           → terminal.svg renders the unscrolled view ← USER SEES NO SCROLL
```

### 3. PAGE UP/DOWN (keydown → scroll)

```
User presses PageUp
  → document keydown handler (line 2233)
  → e.key === 'PageUp' matches BEFORE SPECIAL_KEY_MAP check
  → t.scrollBy(24)
  → same path as wheel scroll → same bug on proxied sessions
```

### 4. CLICK TO FOCUS (click → focus terminal)

```
User clicks a card in the 3D scene
  → renderer.domElement click handler (onSceneClick, line 1006)
  → wasDrag() check — skip if was a drag
  → iterate ALL terminal rects via getBoundingClientRect()
  → find closest by Z-depth (worldPos.project(camera))
  → focusTerminal(clicked)
  → opens WS #2 (inputWs) for the focused terminal
```

**No contentDocument access.** Hit testing is entirely coordinate-based. Works identically with `<object>` or inline SVG.

### 5. DRAG TITLE BAR (mousedown → mousemove → mouseup)

```
User mousedowns on a card header
  → document mousedown handler (line 698)
  → disables iframe pointer-events (line 703)
  → iterates focused terminal headers via getBoundingClientRect()
  → finds header under click, sets _moveCardSession
  → mousemove: updates card.targetPos in 3D space
  → mouseup: clears drag state, re-enables iframe pointer-events
```

**No contentDocument access.** Works identically with `<object>` or inline SVG.

### 6. TEXT SELECTION (Alt+mousedown → mousemove → mouseup)

```
User Alt+mousedowns on a focused terminal
  → document capture-phase mousedown (line 2611)
  → checks: altHeld? activeInputSession? not on header?
  → screenToCell(e, t) ← ACCESSES contentDocument
    → finds <object> element
    → reads obj.getBoundingClientRect() (screen-space rect)
    → reads obj.contentDocument.getElementById('root').viewBox
    → reads obj.contentDocument.getElementById('r0'/'r1').y positions
    → maps screen position → proportional → viewBox → row/col
  → sets selStart, selTerminal
  → mousemove: screenToCell again, updates selEnd, draws highlight
  → mouseup: getSelectedTextFromSvg() ← READS SVG TEXT CONTENT
    → reads obj.contentDocument.getElementById('r0').textContent per row
```

**This is the main contentDocument dependency.** Selection needs to:
1. Map screen coordinates to terminal cells (needs SVG geometry)
2. Draw selection rectangles in the SVG (needs SVG DOM access)
3. Read displayed text for copy (needs SVG text elements)

### 7. URL CLICK (click on focused terminal)

```
User clicks on a URL in a focused terminal
  → onSceneClick (line 1006)
  → coordinate hit test finds the focused card
  → screenToCell(e, t) ← ACCESSES contentDocument
  → getUrlAtCell(t, row, col) — checks t.screenLines[row].spans for .url
  → if URL found: opens in new tab or creates browser card
```

**Uses `screenToCell` (contentDocument) for position, but reads URL from `t.screenLines` (dashboard's copy from WS #2).** With inline SVG, screenToCell would be simpler.

### 8. RESIZE (Alt+drag body)

```
User Alt+drags the card body
  → mousedown: detects 'resize' action
  → mousemove: changes card DOM width/height
  → mouseup: calculates new cols/rows from size ratio
  → t.sendInput({ type: 'resize', cols, rows })
  → sent via WS #2
```

**No contentDocument access except in `optimizeTermToCard()` which reads cell dimensions.**

## The `contentDocument` Problem in Detail

### Current: 6 functions access `obj.contentDocument`

With `<object>`, `contentDocument` provides cross-document DOM access. It works because the SVG is same-origin. The functions read:
- SVG `viewBox` attribute (for coordinate mapping)
- `#measure` element's `getBBox()` (for cell dimensions)
- `#r0`, `#r1` text elements (for row Y positions and text content)
- `#sel-layer` (to draw selection rectangles inside the SVG)

### Approach 1: No change
Still uses `<object>`, still uses `contentDocument`. Same code, same access patterns.

### Approach 2: No change for reading
Still uses `<object>`. Reading contentDocument works the same. Writing selection layer also unchanged.

### Approach 3: Direct DOM access
The SVG elements are in the same document. Instead of:
```javascript
const obj = t.dom.querySelector('object');
const svgDoc = obj.contentDocument;
const root = svgDoc.getElementById('root');
```
You write:
```javascript
const root = t.dom.querySelector('svg.terminal-svg');
```
No `contentDocument`. No cross-frame access. Standard DOM queries. `screenToCell` gets simpler because `getScreenCTM()` works directly (same document).

**However:** The SVG elements need unique IDs per terminal. Currently `#root`, `#r0`, `#r1` etc. are fine because each `<object>` is its own document. With inline SVG, you'd need `#root-${sessionName}`, `#r0-${sessionName}` or use `t.dom.querySelector` scoped queries instead of `getElementById`.

## The WebSocket Problem in Detail

### The Root Cause

```
                     Local tmux sessions          Proxied (claude-proxy) sessions
                     ──────────────────           ──────────────────────────────
WS #1 (terminal.svg) → handleTerminalWs()        → proxy bridge A to claude-proxy
WS #2 (inputWs)      → handleTerminalWs()        → proxy bridge B to claude-proxy

Shared state?         YES — paneScrollOffsets     NO — bridges are independent
                      in server.mjs memory

Scroll visible to     YES — shared offset         NO — only bridge B sees scrolled
terminal.svg?         affects both connections     content; bridge A sees live view

Input visible to      YES — tmux processes it     YES — claude-proxy broadcasts
terminal.svg?         and WS #1 captures new      output to all connections
                      output on next poll
```

### How Each Approach Fixes It

**Approach 1:** terminal.svg's WebSocket becomes the only connection. Dashboard sends input/scroll via postMessage to terminal.svg, which forwards on its WebSocket. One connection, scroll and render on the same stream.

**Approach 2:** Dashboard's WebSocket becomes the only connection. Dashboard pushes screen data to the SVG renderer via postMessage. One connection, scroll and render on the same stream. But postMessage overhead for screen data (15KB every 30ms).

**Approach 3:** Dashboard's WebSocket becomes the only connection. Dashboard calls renderer functions directly to update the SVG DOM. One connection, no cross-frame anything. No postMessage overhead.

## The Click-Drag-on-Any-Card-Selects-Focused-Card Bug

The user also reported: "if you click and drag on ANY card, not just the one that's focused, it selects text on the card that's focused."

**Root cause:** The text selection mousedown handler (line 2611, capture phase) fires on ALL mousedowns. It checks `activeInputSession` but doesn't verify that the click is actually ON the active terminal's card. So a drag on any card in the scene triggers selection on the focused card.

**Fix (independent of architecture choice):** In the capture-phase mousedown handler, verify the click coordinates are within the focused card's bounding rect before starting selection. Currently it only checks "not on header" — it also needs "is on this card."

This bug exists in all three approaches and would be fixed the same way in all three.

---

## Recommendation

All three approaches fix the dual-WebSocket bug. The tradeoffs:

- **Approach 1** is the safest (least change) but adds bridge complexity that grows with every new feature
- **Approach 2** has the worst runtime overhead (postMessage screen data at 33Hz)
- **Approach 3** has the best long-term position (simplest code, most adaptable) but requires extracting terminal.svg's renderer into a module

The event routing analysis shows that **none of the approaches break existing event handlers.** The `pointer-events: none` constraint that protects CSS3DRenderer events works identically whether applied to an `<object>` element or a `<div>` element. The 4x scale trick, hit testing, drag, and all 21 event handlers survive all three approaches.

The four `contentDocument` functions would be simpler with Approach 3 (direct DOM) but are not broken by any approach.

## Approaches Attempted and Results

### Approach 3: Inline SVG — FAILED
- Created `terminal-renderer.mjs` (322 lines) on `ws_test` branch
- Font metrics differ: inline SVG measures 9.13x21.66 vs `<object>` 8.5x25.5 for same font at same size
- Root cause: HTML and SVG rendering contexts produce different `getBBox()` results
- Result: cursor drifts, backgrounds misaligned, text spacing wrong
- Reverted

### HTML Text Overlay — FAILED
- Added transparent `<div>` on top of `<object>` with same TermFont
- Flashing red text test showed character-level drift: 0.248px/char, ~20px by column 80
- Root cause: HTML `getBoundingClientRect()` and SVG `getBBox()` measure same font differently
- Confirmed at 4x scale — still misaligned
- Even with binary-searched font size, horizontal drift is fundamental
- Reverted

### Approach 1 variant: contentWindow bridge — SELECTED
- Proof of concept verified in puppeteer:
  1. `obj.contentWindow.someFunction()` works on SVG `<object>` ✓
  2. `parent.postMessage` from SVG context works ✓
  3. Direct callback (parent sets fn on SVG window) works ✓
- First reviewer recommended direct contentWindow calls over postMessage (simpler, synchronous)
