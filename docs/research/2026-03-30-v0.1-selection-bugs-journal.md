## Bugs Reported by User

1. **Drag on thumbnails triggers selection on focused card.** When a card is focused, clicking and dragging on ANY thumbnail in the sidebar creates a text selection on the focused terminal.

2. **Highlight stays after mouse release.** (User likes this behavior.) After selecting text, the blue highlight remains visible until the next keystroke or Ctrl+C copy.

3. **Highlight doesn't scroll with content.** When the terminal scrolls (PageUp, wheel), the highlight stays at its viewport-relative position. It now highlights different text than what was selected. If the selection persists, it should remain valid — either scrolling with the content or disappearing when the selected rows scroll off screen.

---

## Bug 1 Analysis: Sidebar Triggers Selection

### Root Cause

The capture-phase mousedown handler at `dashboard.mjs:2593` starts text selection when:
1. Left click (`e.button !== 0`)
2. `matchBinding(KEYBINDINGS.selectText, ...)` — context is `'focused'`, modifier is `null`
3. `activeInputSession` exists
4. Click is NOT on a focused card's header (coordinate check against header rects)

**Missing check:** The handler does NOT verify the click is on the focused card's body. Clicks anywhere on the page (including the sidebar) pass all four checks when a terminal is focused.

Then `screenToCell(e, t)` maps the click coordinates to the focused terminal's grid. Even though the click is on the sidebar (completely different screen region), the proportional mapping produces clamped row/col values:
```javascript
fracX = (e.clientX - objRect.left) / objRect.width;
// If click is to the LEFT of the card, fracX is negative → clamped to col 0
// If click is BELOW the card, fracY > 1 → clamped to last row
```

The `Math.max(0, Math.min(row, rows - 1))` clamping means ANY screen coordinate produces a "valid" cell — it never returns null for out-of-bounds clicks.

### Fix

Add a bounds check before starting selection. After getting the focused card, verify the click is within the card's `<object>` bounding rect:

```javascript
// After line 2610: const t = terminals.get(activeInputSession);
const obj = t.dom ? t.dom.querySelector('object') : null;
if (obj) {
  const objRect = obj.getBoundingClientRect();
  if (e.clientX < objRect.left || e.clientX > objRect.right ||
      e.clientY < objRect.top || e.clientY > objRect.bottom) return;
}
```

This is the same pattern used everywhere else in the dashboard — coordinate-based hit testing because CSS3D makes `e.target.closest()` unreliable (PRD §6.3).

Also fix `screenToCell` to return `null` when coordinates are outside the card bounds instead of clamping:

```javascript
// After computing fracX, fracY:
if (fracX < 0 || fracX > 1 || fracY < 0 || fracY > 1) return null;
```

## Bug 2 Analysis: Highlight Stays (Feature, Not Bug)

### Current Behavior

Line 2670: `selTerminal = null` stops the mousemove from updating `selEnd`, but `selStart`/`selEnd` and the SVG highlight rects persist. The highlight is cleared on the next keystroke (line 2677 keydown handler) unless it's Shift+Arrow, Ctrl+C, or a bare modifier.

This means:
- Select text → blue highlight stays visible
- Ctrl+C → copies text, highlight stays
- Any other key → highlight clears
- Click elsewhere → starts new selection (clears old one implicitly)

**User says they like this.** Keep it.

## Bug 3 Analysis: Selection Doesn't Survive Scroll

### Current Behavior

Selection coordinates (`selStart`, `selEnd`) are stored as viewport-relative `{ row, col }`:
```javascript
selStart = { row: 5, col: 10 };  // viewport row 5, column 10
selEnd = { row: 8, col: 30 };    // viewport row 8, column 30
```

When the terminal scrolls, viewport row 5 now shows different content. The highlight stays at row 5-8 visually (it's drawn in the SVG at fixed positions), but it's highlighting the wrong text.

### What "Ideal" Looks Like

Two options:

**Option A: Selection scrolls with content (like a real terminal)**
- Store selection as absolute buffer positions (row + scrollOffset at time of selection)
- On scroll, recalculate viewport-relative positions: `displayRow = absoluteRow - currentScrollOffset`
- If selected rows scroll off screen, hide highlight but keep selection valid
- On copy (Ctrl+C), read from the correct absolute positions
- This requires knowing the scroll offset, which we now have via `_screenCallback`

**Option B: Clear selection on scroll (simpler)**
- When `scrollBy` is called, call `clearSel()`
- Selection is only valid for the current viewport
- Simpler but less useful

**Recommendation: Option A** — the user specifically said "if it stays when scrolling, it needs to stay a valid selection, this would be ideal." This means absolute buffer positions.

### Implementation Sketch for Option A

```javascript
// When selection starts:
selStart = { row: cell.row, col: cell.col, scrollOffset: t.scrollOffset };
selEnd = { row: cell.row, col: cell.col, scrollOffset: t.scrollOffset };
// Absolute row = row + scrollOffset

// When drawing highlight:
const currentOffset = t.scrollOffset;
const startAbsRow = selStart.row + selStart.scrollOffset;
const endAbsRow = selEnd.row + selEnd.scrollOffset;
// Convert back to viewport rows:
const startViewRow = startAbsRow - currentOffset;
const endViewRow = endAbsRow - currentOffset;
// Only draw rects for rows that are in the viewport (0 to rows-1)

// When copying:
// Read text from the absolute positions
// Need access to the full buffer, not just current viewport screenLines
```

**Challenge:** The dashboard only has `screenLines` for the current viewport. To copy text from scrolled-away positions, it needs to either:
- Request the specific lines from the server (new API or scroll to position + read)
- Keep a local buffer of all seen lines (grows unbounded)
- Accept that copy only works when the selection is visible

For the initial implementation, Option A with "copy only works when visible" is simplest — still better than current behavior where the highlight misleads the user.
