# Startup Errors Journal v0.1
**Date:** 2026-03-30
**Session:** HTTP 404/500 errors on dashboard load
**Status:** In Progress
**Preceding:** none
**Changes from v0.0:** Initial discovery and root cause analysis

---

## Errors Observed

On dashboard load, puppeteer captures these failed HTTP requests:

```
404 /favicon.ico
500 /api/pane?session=cp-SVG-Terminal_CLAUD-PROXY_integration_01&pane=0
500 /api/pane?session=cp-SVG-TERM_CLAUD-SOCKET_session_01&pane=0
(repeated ~4x per cp-* session)
```

### Error 1: 404 `/favicon.ico`

**Cause:** Browser automatically requests `/favicon.ico`. server.mjs has no route for it. Returns 404.

**Impact:** None — cosmetic. Appears in browser console as a failed resource load.

**Fix:** Add route returning 204 No Content, or serve a simple SVG icon.

### Error 2: 500 `/api/pane` for `cp-*` sessions

**Cause:** This is a WebSocket initialization race condition.

**Sequence:**
1. Dashboard creates terminal card with `<object data="terminal.svg?session=cp-*">`
2. terminal.svg loads inside the `<object>`
3. terminal.svg starts BOTH a WebSocket connection AND an HTTP poll fallback simultaneously (line 591-592):
   ```javascript
   connectWebSocket();
   schedulePoll(pollInterval);  // safety net — WebSocket onopen will stop it
   ```
4. The poll fires before the WebSocket connects (150ms poll interval vs WebSocket handshake time)
5. Poll hits `GET /api/pane?session=cp-SVG-Terminal...&pane=0`
6. server.mjs `handlePane` calls `tmuxAsync('capture-pane', ...)` on the session name
7. `cp-*` sessions are NOT on the local tmux server — they use custom sockets managed by claude-proxy
8. `tmux capture-pane -t cp-SVG-Terminal...` fails → 500 error
9. WebSocket connects → `onopen` fires → `clearTimeout(pollTimer)` stops polling
10. Errors stop. Terminal renders via WebSocket. Everything works.

**The race window:** ~150ms between `<object>` script execution and WebSocket `onopen`. During this window, 1-2 HTTP polls fire and fail for `cp-*` sessions.

**Impact:**
- Terminal content briefly shows error state (line 329: `showError()` called on failed poll)
- Error state hidden when WebSocket connects (~200ms later)
- User may see a brief flash of "Connection lost" text
- No data loss — WebSocket sends full screen on connect
- Local tmux sessions are NOT affected (poll works for them)

**Why local sessions don't error:** `handlePane` calls `tmuxAsync('capture-pane', '-t', session)` which works for sessions on the default tmux server. `cp-*` sessions are on custom sockets (`-S /var/run/claude-proxy/sessions/...sock`) which plain `tmux` can't see.

---

## Root Cause

The poll fallback in terminal.svg was designed as a safety net for when WebSocket fails. It always fires on startup because the WebSocket hasn't connected yet. For local sessions, the poll returns valid data so the error is invisible. For proxied sessions, the poll hits an endpoint that doesn't know about custom-socket sessions.

## Possible Fixes

### Option A: Proxy `/api/pane` for cp-* sessions
Add the same proxy logic to `handlePane` that exists for WebSocket and session discovery. When session name starts with `cp-`, forward to `http://localhost:3101/api/session/:id/screen`.

**Pros:** Poll fallback works for all sessions. If WebSocket fails permanently, cp-* sessions still render via polling.
**Cons:** More proxy plumbing. The HTTP API formats differ slightly (server.mjs returns sgr-parser spans, claude-proxy returns screen-renderer spans).

### Option B: Skip poll for cp-* sessions in terminal.svg
Pass a URL parameter like `?nopoll=1` for proxied sessions. terminal.svg checks it and doesn't start the poll fallback.

**Pros:** Simple. No server changes.
**Cons:** Loses the poll fallback for cp-* sessions. If WebSocket fails, no rendering at all.

### Option C: Delay poll start until WebSocket fails
Change terminal.svg to only start polling if the WebSocket fails to connect within N seconds.

```javascript
if (SESSION) {
  connectWebSocket();
  // Only start poll fallback if WS doesn't connect within 2s
  setTimeout(function() {
    if (!useWebSocket) schedulePoll(pollInterval);
  }, 2000);
}
```

**Pros:** Eliminates the race entirely. Poll only fires when actually needed.
**Cons:** 2s delay before fallback kicks in if WebSocket is truly broken.

### Option D: Suppress 500 errors in handlePane for unknown sessions
Return an empty/default response instead of 500 when `tmux capture-pane` fails.

**Pros:** No console errors. terminal.svg handles empty response gracefully.
**Cons:** Masks real errors. If a local session genuinely fails, the error is swallowed.

### Recommendation

**Option C** is cleanest — it addresses the root cause (poll fires before WS connects) without adding proxy complexity or masking errors. The 2s delay is acceptable because the WebSocket is the primary path and connects within ~200ms.

**Option A** should be done eventually for full proxy parity, but it's not needed to fix the startup race.

---

## Priority

Low — these errors don't affect functionality. The terminal renders correctly once the WebSocket connects. But they clutter the console and may confuse developers or users who have dev tools open.
