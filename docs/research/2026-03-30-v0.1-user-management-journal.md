## What Already Exists (claude-proxy)

### Auth Infrastructure (complete, not wired)
- **SQLiteUserStore**: users table (linux_user PK, display_name, email, groups_json), provider_links table (provider + provider_id → linux_user)
- **OAuthManager**: Google/Microsoft OIDC via openid-client
- **GitHubAdapter**: GitHub OAuth (separate implementation)
- **Session cookies**: HMAC-SHA256 signed (JWT-like, no dependency)
- **Auth middleware**: decorator for routes
- **LinuxProvisioner**: useradd, groupadd, installClaude — auto-creates system accounts
- **resolveUser**: SSH/OAuth identity → Linux user mapping (email match, provider link, or new provision)

### UGO Security Model (complete, working)
- **SocketManager**: tmux sessions use custom sockets with Unix permissions
  - private (0700), shared (0770 + session group), team (0770 + team group), public (0770 + cp-users)
- **GroupWatcher**: inotify on /etc/group — reactive revocation when group membership changes externally
- Groups prefixed `cp-` in Linux (cp-users, cp-sess-{name}, cp-{team})
- Session-specific groups auto-created for shared mode

### What's NOT Built Yet
- No web login page
- No user management UI
- OAuth providers not configured with real credentials
- OAuth not wired into index.ts startApiServer call
- No invite/enrollment system

## Classroom Platform Research

### Universal Patterns Across Google Classroom, Canvas, Moodle, GitHub Classroom, CodeHS

**Roles:** Owner/Teacher, Co-teacher/TA, Student, Observer (read-only)

**Enrollment methods (every platform supports at least 2):**
1. Join code / invite link (self-service, no email needed)
2. CSV roster import (bulk, standard format)
3. Direct invite by email
4. SIS integration (enterprise)

**Organization:** Courses/Classes contain Sections/Groups. Teachers manage their own classes.

**Onboarding UX:** Role selection at entry → join code or roster linking → confirmation

### Key Insight: Map to Existing UGO

The claude-proxy UGO model maps naturally to classroom concepts:
- **cp-users** = all enrolled users (public sessions)
- **cp-{className}** = a class group (team sessions visible to class)
- **cp-sess-{name}** = per-session group (shared sessions with specific students)
- **cp-admins** = teachers/instructors
- Session access modes (private/shared/team/public) already express classroom needs

What's missing is the **management layer** — creating classes, enrolling students, assigning roles. The OS-level plumbing exists.
