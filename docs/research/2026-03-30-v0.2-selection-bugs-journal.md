**Status:** In Progress
**Preceding:** 2026-03-30-v0.1-selection-bugs-journal.md
**Changes from v0.1:** User rejected re-read approach. Added state mapping for selection vs terminal content independence.

---

## User Feedback

"When the text is initially highlighted, we capture the text. We can assume that text does not change when scrolled in this situation. No re-read is required unless a new selection is made."

This means: **capture once, store the text, never re-read.** Ctrl+C uses the stored text. The highlight is a visual indicator of what was captured, not a live pointer into the buffer.

## The Deeper Question

"Are the highlighted section and the keyboard buffer the 'same' thing when we have something highlighted, or are they different? Can they mutate independently of each other?"

### Two Independent Things

1. **The visual highlight** — SVG rects drawn in the sel-layer at specific viewport row/col positions
2. **The captured text** — the actual string content that was under the highlight when selection was made

These CAN mutate independently:

- **Terminal content changes** (new output, scrollback): The text under the highlighted rows changes. The highlight still shows at the same viewport position but covers different text now.
- **User scrolls**: The highlighted rows move off screen (or different content scrolls into the highlighted position). The visual highlight stays at fixed viewport coordinates.
- **Terminal resizes**: Row/col positions shift. Highlight may cover different or partial text.

### State Map: All Possible Situations

| State | Highlight | Captured Text | Terminal Content | User Intent |
|-------|-----------|--------------|-----------------|-------------|
| **A. Just selected** | Visible, correct position | Matches highlight | Unchanged | Copy what I selected |
| **B. New output appears** | Visible, same position | Still valid (captured at selection time) | Changed — new lines pushed old content up | Copy what I originally selected (not the new text at those positions) |
| **C. Scrolled away** | At original viewport rows (wrong content underneath) | Still valid | Same content, different viewport | Copy what I originally selected. Highlight should either move with content or disappear. |
| **D. Scrolled back** | Back at original rows | Still valid | Back to original view | Copy what I originally selected. Highlight should be back in correct position. |
| **E. Terminal resized** | Position may be wrong (cols/rows changed) | Still valid | Reflowed | Copy what I originally selected. Highlight is best-effort. |
| **F. User starts new selection** | Old highlight replaced | New capture | Irrelevant | New selection replaces old |
| **G. User types (non-selection key)** | Cleared | Discarded | Will change | Done with selection, typing now |

### What the User Expects (Observer Perspective)

**State A:** "I highlighted some text. I can see it. I'll Ctrl+C to copy." → Copy the highlighted text. ✓

**State B:** "I highlighted text, then the terminal kept printing. My blue highlight is still there but the text moved up. If I Ctrl+C, I want the text I ORIGINALLY highlighted, not whatever is there now." → Copy from capture, not re-read. The highlight is stale but the captured text is correct.

**State C:** "I highlighted text, then I scrolled up to read something. My blue highlight is still at the old viewport position but I'm looking at older content now. This is confusing — the highlight should either follow the text I selected (scroll with it) or disappear until I scroll back." → Ideal: highlight moves with content. Minimum: highlight disappears when scrolled away.

**State D:** "I scrolled back to where I was. The text I selected is back in view. The highlight should reappear at the correct position." → Highlight reappears.

**State E:** "Terminal resized. Whatever, I'll just re-select if I need to." → Clear selection on resize is acceptable.

**State F:** "I'm making a new selection." → Replace old with new. ✓ (Already works.)

**State G:** "I'm done looking at the selection, I'm typing now." → Clear. ✓ (Already works.)

### Recommended Solution

**Capture immediately, store as string:**
```javascript
// On mouseup (selection complete):
selCapturedText = getSelectedTextFromSvg(t);  // capture once
```

**Ctrl+C reads from capture, not SVG:**
```javascript
if (selCapturedText) {
  copyToClipboard(selCapturedText);
}
```

**Highlight behavior on scroll:**
- Store absolute buffer position (row + scrollOffset at time of selection)
- On scroll, recalculate viewport rows for drawing
- If selection scrolled off screen, hide highlight (but keep `selCapturedText` valid)
- If selection scrolled back into view, redraw highlight
- Ctrl+C always works regardless of scroll position (reads from `selCapturedText`)

**Highlight behavior on new output (State B):**
- Don't try to track content movement (complex, unreliable)
- Keep highlight at original absolute position
- If new output pushes selected content up, highlight becomes stale visually
- Ctrl+C still copies the correct text (from `selCapturedText`)
- A simple improvement: clear highlight after N seconds of inactivity, or when content changes significantly

**Clear selection on:**
- Any non-modifier keystroke (except Ctrl+C, Shift+Arrow) — already works
- Terminal resize — new addition
- New selection — already works

**Don't clear selection on:**
- Scroll (highlight repositions or hides, but captured text stays valid)
- New terminal output (highlight may become stale, but captured text stays valid)

### Summary of Changes to Plan

1. **Task 1 (bounds check):** Unchanged — still needed.
2. **Task 2 (absolute positions):** Still needed — but now also store `selCapturedText`.
3. **Task 3 (highlight scrolls):** Still needed — draw at viewport-adjusted positions.
4. **Task 4 (copy):** Simplified — always read from `selCapturedText`, never re-read SVG. Remove the visibility check — copy works from any scroll position.

### New Variable

```javascript
let selCapturedText = null;  // string, captured on mouseup when selection is real

// Set in mouseup handler:
selCapturedText = getSelectedTextFromSvg(t);

// Used in Ctrl+C handler:
if (selCapturedText) copyToClipboard(selCapturedText);

// Cleared in clearSel():
selCapturedText = null;
```
