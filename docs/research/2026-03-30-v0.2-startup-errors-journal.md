## Two Issues Reported

### Issue 1: "Connection lost — retrying" overlay appears on working terminal

**What user sees:** A semi-transparent dark overlay with "Connection lost — retrying" text appears on top of a terminal that IS working (content visible underneath, typing works).

**What the overlay is:**
- SVG element `#error-overlay` inside terminal.svg (line 57)
- A `<g>` group containing a dark rect (85% opacity) and text
- Controlled by `showError()` / `hideError()` which toggle `visibility` attribute
- `showError()` called when: HTTP poll fails (line 408), WebSocket receives `type: 'error'` message (line 333)
- `hideError()` called when: WebSocket `onopen` fires (line 292), successful screen message (line 312), successful HTTP poll (line 404)

**Why it appears intermittently:**
The startup race condition from v0.1 — terminal.svg starts BOTH a WebSocket AND an HTTP poll simultaneously. The poll fires first (~150ms), hits `/api/pane` for a `cp-*` session, gets a 500 error, and calls `showError()`. The overlay appears. Then ~50ms later the WebSocket connects, receives a screen message, calls `hideError()`. The overlay disappears.

**But sometimes the overlay persists.** This happens when:
1. The HTTP poll fires and fails → `showError()` → overlay visible
2. WebSocket connects → `hideError()` → overlay hidden
3. WebSocket drops (network blip, server restart, proxy timeout) → `onclose` fires
4. `onclose` doesn't call `showError()` directly, but starts poll fallback: `schedulePoll(pollInterval)`
5. Poll fires → hits `/api/pane` for cp-* session → 500 → `showError()` → overlay visible
6. WebSocket reconnects (2s delay) → `onopen` → `hideError()`

**The window where the overlay shows:** between poll failure and WebSocket reconnect. Could be 50ms (fast reconnect) or 2+ seconds (slow reconnect).

**Why it looks persistent in the screenshot:** The screenshot was taken during one of these windows. The terminal IS working underneath (content visible through 85% opacity overlay) because the WebSocket reconnected and started sending screen data, but `hideError()` may not have been called yet, or another poll failure re-triggered `showError()`.

### Issue 2: The overlay layer itself — is it needed?

**What it is:** An SVG error indicator embedded in terminal.svg. Dark background + "Connection lost" text. Covers the entire terminal viewport.

**Why it was created:** To show the user when the terminal connection drops. In the original design (HTTP polling only), this was the primary error indicator. When polling failed, the terminal went dark with this message.

**Is it still needed?**

With the single-WebSocket architecture:
- The WebSocket has auto-reconnect (2s delay)
- During reconnect, the SVG shows the last rendered content (stale but visible)
- The overlay blocks the view of the stale content with a dark rect

**Arguments to KEEP it:**
- User knows the terminal lost connection (not just frozen/stale)
- Clear visual state: "this isn't live right now"

**Arguments to REMOVE or CHANGE it:**
- 85% opacity is too aggressive — hides content the user may want to read
- The intermittent flash (show/hide in rapid succession during race) is jarring
- With WebSocket auto-reconnect, the "lost" state lasts 2 seconds max
- A subtle indicator (border color change, small icon) would be less intrusive

**Arguments to FIX the trigger:**
- The overlay triggers on HTTP poll failure, but the HTTP poll is a fallback — the WebSocket is the primary path
- The overlay should only show when the WebSocket is down, not when a fallback poll fails
- Fix: `showError()` only called from WebSocket close/error, not from poll failure

## Are They Related?

**Yes.** Both issues stem from the same root cause: the startup race condition where the HTTP poll fires before the WebSocket connects.

```
Timeline for cp-* session:
t=0ms     <object> loads terminal.svg
t=0ms     connectWebSocket() called — handshake begins
t=0ms     schedulePoll(150) called — poll timer starts
t=150ms   Poll fires → GET /api/pane?session=cp-*
t=150ms   server.mjs: tmux capture-pane fails (session on custom socket) → 500
t=150ms   terminal.svg: showError() → OVERLAY VISIBLE
t=200ms   WebSocket onopen fires → hideError() → OVERLAY HIDDEN
t=200ms   WebSocket sends screen → terminal renders → hideError() again
```

The 500 error (Issue 2 from v0.1) directly causes the overlay flash (Issue 1 here).

## Debug Paths

### If they ARE related (most likely):
Fix the startup race — **Option C from v0.1**: delay poll start until WebSocket fails to connect within 2 seconds. This eliminates the 500 errors AND the overlay flash because the poll never fires during normal startup.

### If the overlay appears WITHOUT a 500 error (separate issue):
The WebSocket itself dropped. Debug path:
1. Check server logs for WebSocket close events
2. Check if the proxy bridge to claude-proxy is timing out
3. Add ping/pong keepalive to the WebSocket connection
4. Check if Chrome's background tab throttling is pausing the WebSocket

### If the overlay persists after WebSocket reconnects (separate issue):
The `hideError()` call in `onopen` or `onmessage` isn't firing. Debug path:
1. Add `console.log` in `hideError` to confirm it's called
2. Check if the `error-overlay` element's `visibility` attribute is actually being set
3. Check if there's a CSS rule overriding the SVG visibility attribute

## Recommended Fix (addresses both issues)

**Change terminal.svg startup from:**
```javascript
if (SESSION) {
  connectWebSocket();
  schedulePoll(pollInterval);  // fires immediately, races with WS
}
```

**To:**
```javascript
if (SESSION) {
  connectWebSocket();
  // Only start poll fallback if WS doesn't connect within 2s
  setTimeout(function() {
    if (!useWebSocket) {
      showError();  // show overlay only when WS truly failed
      schedulePoll(pollInterval);
    }
  }, 2000);
}
```

This:
1. Eliminates the 500 errors (poll never fires for cp-* sessions during normal startup)
2. Eliminates the overlay flash (showError only called after 2s WS failure)
3. Preserves the poll fallback for when WebSocket genuinely fails
4. Preserves the error overlay for genuine connection loss

**Additionally:** Consider reducing overlay opacity from 85% to 50% or replacing with a less intrusive indicator (colored border, small "disconnected" badge in corner).

## UPDATE: Why Overlay Persists After Connection (user reported ctrl+shift+r required)

User reported the overlay stayed permanently, even though the terminal was working underneath. Required a hard refresh to clear.

### Root Cause: Race Between onopen hideError and poll showError

```
t=0ms     connectWebSocket() + schedulePoll(150)
t=150ms   Poll fires → 500 → showError() → OVERLAY ON
t=200ms   WS onopen → hideError() → OVERLAY OFF
t=200ms   WS sends screen → hideError() → still off
          (everything looks fine so far)

BUT sometimes:
t=0ms     connectWebSocket() + schedulePoll(150)
t=180ms   WS onopen → hideError() → OVERLAY OFF
t=180ms   WS onopen clears pollTimer (lines 288-290)
t=150ms   BUT poll was already scheduled and ALREADY FIRED before clearTimeout ran
          (JS event loop: the setTimeout callback was already queued)
t=181ms   Poll callback executes → 500 → showError() → OVERLAY ON (after hideError!)
t=200ms   WS sends first delta (not screen) → NO hideError() call
t=???     OVERLAY STAYS ON FOREVER
```

### The Two Bugs

**Bug A:** `hideError()` is called on `screen` messages (line 312) but NOT on `delta` messages. If the first message after the race is a delta, the overlay persists.

**Bug B:** The poll timer fires even after `clearTimeout` in `onopen` because the callback was already queued in the event loop before `onopen` ran. `clearTimeout` only prevents future fires — if the callback is already in the microtask queue, it still executes.

### Fix

**Fix A (immediate):** Add `hideError()` to the delta handler:
```javascript
} else if (msg.type === 'delta') {
    // ... existing delta code ...
    hideError();  // ← ADD THIS
}
```

**Fix B (root cause):** Add a guard in the poll callback:
```javascript
function poll() {
  pollTimer = null;
  if (useWebSocket) return;  // ← this line already exists (line 351)
  // ... rest of poll
}
```

Wait — line 351 already has `if (useWebSocket) return;`. But `useWebSocket` is set to `true` in `onopen` (line 286). If `onopen` ran before the poll callback, `useWebSocket` is true and the poll should bail. Unless the poll callback was queued BEFORE `onopen` set the flag...

Actually in JS, `setTimeout` callbacks and WebSocket `onopen` are both macrotasks. They execute one at a time. If `onopen` runs first, `useWebSocket = true`, and the poll callback (running next) sees it and returns. If the poll callback runs first, `useWebSocket` is still false, the poll executes, 500 → `showError()`, then `onopen` runs → `hideError()`. In this case the overlay is hidden correctly.

The problem must be when `onopen` → `hideError()` runs, then a SUBSEQUENT poll fires (re-scheduled by the `catch` handler at line 409: `schedulePoll(RETRY_MS)`). The failed poll re-schedules itself!

```
t=150ms   First poll fires → 500 → catch → showError() + schedulePoll(2000)
t=200ms   WS onopen → hideError() + clearTimeout(pollTimer)
          BUT pollTimer was already set to a NEW timeout by schedulePoll(2000)
          AND clearTimeout in onopen clears the ORIGINAL timer, not the retry timer
```

No wait — `pollTimer` is a single variable. `schedulePoll(2000)` sets `pollTimer = setTimeout(poll, 2000)`. Then `onopen`'s `clearTimeout(pollTimer)` clears THAT timer. So the retry should be cancelled.

Unless the `catch` → `schedulePoll` and `onopen` → `clearTimeout` happen in the same microtask tick and the order is wrong. But they're separate macrotasks...

**Most likely cause:** The poll fires, fails (500), calls `showError()` AND `schedulePoll(RETRY_MS)`. Before the retry fires, WS connects, `onopen` clears the timer and calls `hideError()`. No problem. But if the poll fires AFTER `onopen` (timer was queued before `onopen` but executes after), `showError()` runs after `hideError()`. And the next WS message is a delta with no `hideError()` → stuck.

**The simplest fix for both:** Add `hideError()` to the delta handler AND delay poll start by 2s.
