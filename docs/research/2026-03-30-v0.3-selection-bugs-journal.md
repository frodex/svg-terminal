## Updated Model

Selection capture is per-terminal, stored on the terminal object:

```javascript
terminals.set(sessionName, {
  // ... existing fields ...
  textCaptureActive: false,    // is there a captured selection?
  textCaptureValue: null,      // the captured text string
  textCaptureStart: null,      // { row, col, scrollOffset } — for highlight drawing
  textCaptureEnd: null,        // { row, col, scrollOffset } — for highlight drawing
});
```

### State Transitions

```
IDLE (no capture)
  │
  ├── Mouse drag (> 1.5 cell widths) on focused terminal body
  │   └── textCaptureActive = true
  │       textCaptureValue = extracted text
  │       textCaptureStart/End = cell positions with scrollOffset
  │       clipboard = textCaptureValue (auto-copy)
  │       highlight drawn in SVG sel-layer
  │
  ├── Ctrl+C while textCaptureActive
  │   └── clipboard = textCaptureValue (re-push to clipboard)
  │       state unchanged (stays active)
  │
  ├── Click (not drag) on terminal body
  │   └── textCaptureActive = false
  │       textCaptureValue = null
  │       highlight cleared
  │       (click-to-move-cursor fires instead)
  │
  ├── Any non-modifier keystroke (except Ctrl+C, Shift+Arrow)
  │   └── textCaptureActive = false
  │       textCaptureValue = null
  │       highlight cleared
  │
  ├── New selection drag
  │   └── old capture replaced with new
  │
  ├── Escape
  │   └── If textCaptureActive: clear capture (don't unfocus)
  │       If not textCaptureActive: unfocus terminal (existing behavior)
  │
  └── Scroll (PageUp, wheel)
      └── textCaptureActive unchanged
          textCaptureValue unchanged
          highlight redrawn at scroll-adjusted position (or hidden if off screen)
```

### Focus/Unfocus Behavior

- **Unfocus:** Capture persists. Highlight stays in the SVG DOM (sel-layer isn't cleared by render cycles). When the user refocuses and presses Ctrl+C, it re-pushes the captured text.
- **Focus different terminal:** Each terminal has its own `textCaptureActive`/`textCaptureValue`. Switching focus doesn't clear other terminals' captures.
- **Multiple captures:** Theoretically possible (one per terminal), but only the active terminal's Ctrl+C fires. The user would need to refocus the terminal with the capture to re-push it.

## Minimum Drag Distance

### Problem

Clicking on a terminal card often produces a 1-character "selection" because:
1. Mousedown at cell (5, 10)
2. Tiny mouse movement (1-2 pixels) during click
3. Mouseup at cell (5, 11) — different cell!
4. `isRealSelection` returns true → accidental selection

### Fix

Track pixel distance, not just cell difference. Require > 1.5 cell widths of drag before starting capture.

```javascript
// On mousedown:
selDragStartX = e.clientX;
selDragStartY = e.clientY;

// On mouseup — check pixel distance:
const dx = e.clientX - selDragStartX;
const dy = e.clientY - selDragStartY;
const pixelDist = Math.sqrt(dx * dx + dy * dy);

// Get cell width in screen pixels from the terminal's render info
const renderInfo = getTermRenderInfo(t);
const minDragPx = renderInfo ? renderInfo.cellW * 1.5 : 15;

const isRealSelection = pixelDist > minDragPx;
```

This means:
- Click (no/tiny movement) → click-to-move-cursor → clears any existing capture
- Short drag (< 1.5 cells) → treated as click → no capture
- Real drag (> 1.5 cells) → capture + highlight

## Clear Conditions Summary

| Action | Result |
|--------|--------|
| Click on terminal body (not drag) | Clear capture, move cursor |
| Start new selection | Replace old capture |
| Any keystroke (except Ctrl+C, Shift+Arrow, modifiers) | Clear capture |
| Escape (with active capture) | Clear capture, stay focused |
| Escape (without active capture) | Unfocus terminal (existing) |
| Scroll | Keep capture, reposition highlight |
| Focus different terminal | Keep capture on original terminal |
| Unfocus all | Keep capture |
| Terminal resize | Clear capture (positions invalid) |
| Terminal removed from scene | Clear capture |

## Global Variables to Terminal Properties Migration

Current global state:
```javascript
let selTerminal = null;
let selStart = null;
let selEnd = null;
let selOverlay = null;
let selCapturedText = null;  // proposed in v0.2
```

New per-terminal state:
```javascript
// On each terminal object:
textCaptureActive: false,
textCaptureValue: null,
textCaptureStart: null,    // { row, col, scrollOffset }
textCaptureEnd: null,      // { row, col, scrollOffset }
```

Still need one global to track which terminal is being actively dragged:
```javascript
let selDragging = null;     // sessionName of terminal being drag-selected, null when not dragging
let selDragStartX = 0;      // pixel coords at mousedown
let selDragStartY = 0;
```

The highlight drawing reads from the terminal object, not globals. Ctrl+C reads from the focused terminal's `textCaptureValue`.

## Workflow Verification

User's workflow:
1. **Select text in terminal** → mouseup triggers capture → `t.textCaptureValue = "selected text"`, clipboard = "selected text", highlight drawn
2. **Switch to Word, Ctrl+V** → pastes "selected text" ✓
3. **Do work in Word, Ctrl+C something** → clipboard now has Word content
4. **Come back to terminal** → highlight still visible on the terminal card
5. **Ctrl+C** → reads `t.textCaptureValue`, pushes "selected text" back to clipboard
6. **Switch to Word, Ctrl+V** → pastes "selected text" ✓

Works because `textCaptureValue` never mutated — only cleared by explicit user action.
