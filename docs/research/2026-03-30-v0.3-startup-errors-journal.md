## Problem Statement

The "Connection lost — retrying" overlay (`#error-overlay` in terminal.svg) appears on top of a working terminal and persists until hard refresh (Ctrl+Shift+R). The terminal content is visible underneath and interactive — typing works. The overlay is cosmetically blocking the view and misleading.

## Root Cause: Poll/WebSocket Race + Missing hideError on Delta

### The Race Sequence

```
t=0ms     terminal.svg loads, calls connectWebSocket() + schedulePoll(150)
t=150ms   Poll timer fires → GET /api/pane?session=cp-*
          For cp-* sessions: tmux capture-pane fails (custom socket) → 500 error
          Poll catch handler: showError() → OVERLAY ON
          Poll catch handler: schedulePoll(RETRY_MS) → schedules another poll

t=180ms   WebSocket onopen fires
          useWebSocket = true
          clearTimeout(pollTimer) → cancels pending retry poll
          hideError() → OVERLAY OFF
          ... looks fine so far ...

BUT: If the poll callback was already queued in the JS event loop before
onopen ran, the execution order can be:

t=180ms   WebSocket onopen → hideError() → OVERLAY OFF
t=181ms   Poll callback (already queued) executes → 500 → showError() → OVERLAY ON
t=200ms   WebSocket sends first message → it's a DELTA (not screen)
          Delta handler does NOT call hideError()
          → OVERLAY STAYS ON FOREVER
```

### Why Delta Doesn't Hide Error

In `ws.onmessage` handler:
- `screen` message → line 312: `hideError()` ✓
- `delta` message → NO `hideError()` call ✗
- `error` message → line 333: `showError()`

The first WebSocket message is usually a `screen` (full state on connect). But if the connection was already established and sending deltas, the race means `showError()` fires between screen and delta messages, and only a screen message would clear it.

### Why It Persists Until Hard Refresh

Once the overlay is stuck:
- Deltas keep arriving (terminal updates underneath) → no `hideError()`
- The WebSocket IS connected → no `onclose` → no reconnect cycle
- The poll was cancelled by `onopen` → no more poll attempts
- Nothing in the code path calls `hideError()` again until the next `screen` message
- `screen` messages only arrive on: initial connect, resize, or scroll
- If the user just types normally → only deltas → overlay stays forever

### Self-Healing: Already Exists But Blocked

WebSocket has auto-reconnect:
```javascript
ws.onclose → useWebSocket = false → schedulePoll() → setTimeout(connectWebSocket, 2000)
```

If the WebSocket genuinely drops and reconnects:
1. `onclose` → starts poll + reconnect timer
2. Reconnect → `onopen` → `hideError()` → cleared
3. New connection sends `screen` → `hideError()` → belt and suspenders

But in the bug scenario, the WebSocket never dropped — it's working fine. The overlay is a false alarm from a poll that raced with the connection. The self-healing reconnect never triggers because there's nothing to reconnect.

## Fix: Two Changes

### Fix 1: Guard showError in poll catch

The poll's error handler should not show the overlay if the WebSocket is already connected. The WebSocket is the primary path — if it's working, poll errors are irrelevant.

**In terminal.svg, poll catch handler (~line 407-409):**

FROM:
```javascript
.catch(function (err) {
    showError();
    schedulePoll(RETRY_MS);
});
```

TO:
```javascript
.catch(function (err) {
    if (!useWebSocket) showError();
    schedulePoll(RETRY_MS);
});
```

**Why this works:** `useWebSocket` is set to `true` in `onopen` (line 286). If the WebSocket connected between the poll firing and the poll failing, `useWebSocket` is already true. The poll error is irrelevant — don't show the overlay.

### Fix 2: Add hideError to delta handler

Any successful data from the WebSocket means the connection is working. Both `screen` and `delta` messages should clear the error overlay.

**In terminal.svg, delta handler (~line 330, after the _screenCallback call):**

Add:
```javascript
hideError();
```

**Why this works:** If `showError()` somehow fired after the WebSocket connected (the race), the very next delta message clears it. Belt and suspenders — even if Fix 1 misses an edge case, Fix 2 self-heals within 30ms (the delta polling interval).

## Why Not a Delay?

User asked: "is there a better option than delay?"

A 2-second delay before starting the poll was proposed in v0.1. It works but is arbitrary:
- What if WebSocket takes 3 seconds to connect? Poll starts, same race.
- What if WebSocket connects in 50ms? 2 seconds of unnecessary wait before fallback.

The connection state check (`if (!useWebSocket)`) is the correct signal:
- WebSocket connected → poll errors are noise, suppress overlay
- WebSocket NOT connected → poll errors are real, show overlay

No timing assumptions. The state IS the answer.

## What Changes for Users

- The "Connection lost" overlay no longer flashes on startup for cp-* sessions
- If the WebSocket genuinely fails, the overlay still shows correctly
- If the WebSocket reconnects, any message (screen or delta) clears the overlay
- No behavioral change for local tmux sessions (their polls succeed, no overlay)
