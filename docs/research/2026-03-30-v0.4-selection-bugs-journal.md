## Issue: Typing Doesn't Clear Selection

### Observed

User types while selection highlight is visible. The keystroke goes to the terminal but the highlight stays.

### Root Cause Investigation

There are TWO separate `document.addEventListener('keydown')` handlers:

1. **Main input handler** (line 2135) â€” sends keystrokes to terminal, handles PageUp/scroll, Ctrl+C copy, etc. Calls `e.preventDefault()` at line 2156 and `return` after processing each key.

2. **Selection clear handler** (line 2677) â€” checks for non-modifier keystrokes and calls `clearSel()`.

Both listeners fire for every keydown event â€” `addEventListener` doesn't prevent other listeners from the same event type. The main handler calls `e.preventDefault()` and `return`, but `return` only exits its own callback, not the event. The clear handler should still fire.

BUT â€” looking at the clear handler:
```javascript
document.addEventListener('keydown', function(e) {
  if (e.shiftKey && ['ArrowUp', ...].includes(e.key)) return;
  if (e.ctrlKey && e.key.toLowerCase() === 'c') return;
  if (['Control', 'Shift', 'Alt', 'Meta'].includes(e.key)) return;
  if (selStart && !e.altKey) {
    clearSel();
  }
});
```

It checks `selStart` â€” the global variable. If `selStart` is set, it should clear. But wait â€” the mouseup handler (line 2670) sets `selTerminal = null` but keeps `selStart`/`selEnd`. So `selStart` should be non-null and `clearSel()` should fire.

**Possible cause:** The order of `addEventListener` registrations matters. If the input handler is registered first and calls `e.stopPropagation()` or `e.stopImmediatePropagation()` somewhere... Let me check.

Line 2616-2617 in the selection mousedown handler:
```javascript
  e.preventDefault();
  e.stopPropagation();
```

That's in the mousedown handler (capture phase), not keydown. The keydown handlers don't call `stopImmediatePropagation`. So both should fire.

**Alternative cause:** The `clearSel()` function clears the globals (`selStart = null`, `selEnd = null`) and clears the SVG overlay (`selOverlay.innerHTML = ''`). But in the v2 plan, we're moving capture to per-terminal objects. The `clearSel()` might clear the globals but not the terminal's `textCaptureActive` flag.

**Resolution:** In the updated plan (v2), the keystroke clear handler must clear BOTH the globals (for highlight drawing) AND the terminal's capture fields. Task 5 in the plan addresses this but the current (unpatched) code only clears globals. This explains the current bug â€” `clearSel()` is being called but something is keeping the highlight visible.

Actually wait â€” let me re-read `clearSel()`:
```javascript
function clearSel() {
  selTerminal = null;
  selStart = null;
  selEnd = null;
  if (selOverlay) selOverlay.innerHTML = '';
}
```

It clears `selOverlay.innerHTML` â€” but `selOverlay` might not be the right element. The highlight is drawn in the SVG's sel-layer via `contentDocument`, not in a dashboard DOM element. Let me check `getSelOverlay`:

The `drawSelHighlight` function accesses `obj.contentDocument.getElementById('sel-layer')`. But `clearSel` uses `selOverlay` which is initialized to `null` (line 2275) and... I need to check if it's ever set. If `selOverlay` is always null, `clearSel` never actually clears the SVG highlight rects!

**This may be the actual bug.** `clearSel()` sets the globals to null but never removes the SVG rects from the sel-layer. The rects persist in the SVG DOM until `drawSelHighlight` is called again (which clears before redrawing) or the SVG re-renders.

## Issue: New Terminal Output Doesn't Move Selection

### The Problem

When new output appears in the terminal:
- Content scrolls up (new lines push old lines into scrollback)
- The viewport always shows the latest content (scrollOffset = 0)
- Selection was made at viewport row 5 when that row showed "foo"
- After 10 new lines of output, row 5 now shows something else
- "foo" has moved to scrollback (10 lines above the viewport)
- The highlight sits at viewport row 5 showing the wrong text

### Why This Is Different From User-Scroll

**User scroll:**
- User explicitly changes `scrollOffset` from 0 to N
- We know the offset changed
- We can recalculate: `viewportRow = absoluteRow - scrollOffset`
- Content at a given absolute position doesn't change, just the viewport window moves

**New output:**
- `scrollOffset` stays at 0 (live view)
- But the mapping of "absolute buffer row â†’ viewport row" changes
- Specifically: `baseY` in the xterm buffer increases with each new line
- If we captured at `absoluteRow = baseY + viewportRow` at time T, then at time T+N, `baseY` has increased by N, so the same absoluteRow is now at `viewportRow - N`

### What We Need

To track selection position through new output, we need to know how many lines have scrolled since the capture was made. Two approaches:

**Approach A: Track line count via screen callbacks**

The `_screenCallback` fires on every screen/delta update. On a `screen` message, we get `msg.height` (rows). We could track `baseY` if the server included it in the response. Currently it doesn't â€” we only get viewport content.

**Approach B: Content-based tracking**

Store the first line of text at the time of capture. On each update, scan for that text in the current viewport. If found at a different row, the selection shifted by the difference. Fragile â€” text can repeat, and scanning is expensive.

**Approach C: Server-side baseY**

Have the server include `baseY` (or a line counter) in screen/delta messages. The dashboard tracks the delta: `lineShift = currentBaseY - captureBaseY`. Highlight draws at `capturedViewportRow - lineShift`.

For local tmux sessions: `baseY` isn't available from tmux directly (it's not in capture-pane output). We'd need to track it from the `#{history_size}` tmux format variable, which IS included in the server's metadata response.

For claude-proxy sessions: `baseY` is available from the xterm buffer dims. The claude-proxy agent's scroll fix already accesses `dims.baseY`.

**Approach D: Accept the limitation for now**

New output makes the highlight stale. The captured text is still correct (stored on the terminal object). The highlight is visual-only and becomes misleading. Clear it when content changes significantly.

### Recommendation

**For initial implementation: Approach D with a refinement.** Track whether the terminal has received new content since the capture. If it has, fade or dim the highlight (change opacity from 0.3 to 0.1) to signal staleness. The captured text remains valid for Ctrl+C.

**For future:** Approach C (server includes baseY/lineCount) is the clean solution. It allows exact repositioning of the highlight as content scrolls. This requires a small server-side change in both server.mjs (add `historySize` to delta messages) and claude-proxy (add `baseY` to delta messages).
