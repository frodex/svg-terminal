# Text Selection Bugs Journal v0.5
**Date:** 2026-03-30
**Session:** Selection design — simplified to fade-on-release
**Status:** Direction Confirmed
**Preceding:** 2026-03-30-v0.4-selection-bugs-journal.md
**Changes from v0.4:** Abandoned persistent capture. Selection clears on mouseup with visual fade.

---

## Design Decision

Tracking selection position through live terminal output (new lines pushing content up) is not feasible without server-side line counters. Rather than add complexity for diminishing returns, the user chose:

**Select → auto-copy → fade → clear.**

### Behavior

1. User drags to select text on a focused terminal
2. On mouseup: text copied to clipboard immediately
3. Highlight flashes bright (white/bright gray) to confirm capture
4. Highlight fades to transparent over 2 seconds
5. After fade completes, selection state cleared entirely
6. No persistent highlight, no stale state, no tracking needed

### Why This Is Better

- No stale highlights misleading the user
- No per-terminal capture state to manage
- No scroll/output tracking complexity
- No ambiguity about what Ctrl+C does (always sends C-c to terminal — no selection interference)
- Clipboard has the text from the moment of release
- Visual feedback (flash + fade) confirms the capture happened

### What We Still Fix

1. **Bounds check** — prevent selection starting from sidebar thumbnails
2. **Minimum drag distance** — prevent accidental 1-character selections from clicks
3. **Fade animation** — flash bright on mouseup, fade to transparent over 2 seconds

### What We Drop From Previous Plans

- Per-terminal `textCaptureActive` / `textCaptureValue` — not needed
- Scroll-aware highlight repositioning — not needed
- Escape to clear capture — not needed (nothing to clear)
- Ctrl+C re-copy from stored text — not needed
- `historySize` in WebSocket messages — not needed for this feature

### Implementation Summary

Changes to `dashboard.mjs` only:
1. Add bounds check in mousedown (prevent sidebar selection)
2. Add pixel distance check in mouseup (minimum drag > 1.5 cell widths)
3. On real selection mouseup: copy to clipboard, trigger fade animation
4. Fade animation: CSS transition on sel-layer rects opacity (1.0 → 0.8 bright flash → 0.0 over 2s)
5. After fade: clearSel() removes rects and nulls globals
6. Remove the keystroke-clear handler (no persistent selection to clear)

### Fade Animation Detail

On mouseup with real selection:
```
t=0ms:     Selection complete. Copy to clipboard.
t=0ms:     Set sel-layer rects fill to rgba(200, 200, 255, 0.5) (bright flash)
t=100ms:   Begin fade transition
t=2100ms:  Opacity reaches 0. Call clearSel().
```

Implementation options:
- **CSS transition on SVG rects** — set opacity via setAttribute, use setTimeout for staged changes. SVG rects support `opacity` attribute and CSS transitions.
- **JS animation loop** — requestAnimationFrame to interpolate opacity. More control but more code.
- **CSS `@keyframes` on a class** — add a `.fading` class to the sel-layer `<g>`. But this is inside the `<object>` contentDocument — CSS classes injected from parent may not work.

Simplest: set opacity on each rect via `setAttribute('opacity', ...)` in a setTimeout chain.

```javascript
// On mouseup with real selection:
const layer = getSelOverlay(t);
if (layer) {
  // Flash bright
  for (const rect of layer.children) {
    rect.setAttribute('fill', 'rgba(200, 200, 255, 0.6)');
  }
  // Fade out over 2 seconds
  let fadeStart = performance.now();
  function fade() {
    const elapsed = performance.now() - fadeStart;
    const progress = Math.min(1, elapsed / 2000);
    const opacity = 0.6 * (1 - progress);
    for (const rect of layer.children) {
      rect.setAttribute('opacity', String(opacity));
    }
    if (progress < 1) {
      requestAnimationFrame(fade);
    } else {
      clearSel();
    }
  }
  requestAnimationFrame(fade);
}
```
