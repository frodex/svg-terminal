**Status:** In Progress

**Changes from v0.5:** Keyboard (Shift+Arrow) selection broken after mouse selection rework

---

## What Changed That Broke Keyboard Selection

The v3 plan focused on mouse selection and made these changes:
1. Removed the keystroke-clear handler (no more `clearSel()` on typing)
2. Changed Ctrl+C to always send C-c to terminal (no selection check)
3. Mouse selection now fades on mouseup â€” ephemeral by design

But keyboard selection (Shift+Arrow) was not addressed. It uses the SAME global variables (`selStart`, `selEnd`) and the SAME `drawSelHighlight` function. The changes broke it:

---

## Error States Found

### Error 1: No way to clear keyboard selection
**Before:** Keystroke-clear handler called `clearSel()` on non-modifier/non-selection keys. Typing would clear the Shift+Arrow selection.
**After:** Keystroke-clear handler removed. Keyboard selection persists forever. No mouseup (it's keyboard-driven). No fade timer. The highlight stays until... nothing clears it.

### Error 2: Ctrl+C doesn't copy keyboard selection
**Before:** `if (e.ctrlKey && e.key === 'c' && selStart)` checked for active selection and copied it.
**After:** That check was removed. Ctrl+C always sends C-c to terminal. If user Shift+Arrow selects text then presses Ctrl+C, the process gets interrupted instead of copying.

### Error 3: Selection and cursor are at different positions
Keyboard selection starts from `t._lastCursor` position. But `_lastCursor` is updated from `_screenCallback` messages (screen/delta). If the callback hasn't fired recently, `_lastCursor` may be stale. The selection starts at the wrong position.

Also: after selection, if the user types (which now doesn't clear selection), the cursor moves but the selection stays. The selection visually covers text that no longer relates to the cursor position.

### Error 4: Mouse and keyboard selection share globals
If a mouse selection is fading (2s animation in progress) and the user starts a Shift+Arrow selection, the globals get overwritten. The fade animation's `clearSel()` call at the end will clear the keyboard selection. Or vice versa â€” starting a mouse selection during a keyboard selection overwrites it.

---

## Two Types of Selection â€” Different Behaviors

The fundamental issue: **mouse selection and keyboard selection have different lifecycles.**

| | Mouse Selection | Keyboard Selection |
|---|---|---|
| **Start** | Mousedown + drag | Shift+Arrow |
| **Extend** | Mousemove | Shift+Arrow (repeated) |
| **End/Confirm** | Mouseup | No natural "end" event |
| **Copy** | Auto-copy on mouseup | Ctrl+C |
| **Visual feedback** | Flash + fade | Persistent highlight |
| **Clear** | Fade timer (2s) | ??? |

They CANNOT share the same lifecycle. Mouse selection is ephemeral (auto-copy + fade). Keyboard selection needs to persist until explicitly copied or dismissed.

---

## Proposed Resolution: Two Modes

### Mouse Selection (unchanged from v3)
- Drag to select â†’ auto-copy â†’ flash â†’ fade â†’ clear
- No persistent state
- Ctrl+C sends C-c (no selection to check)

### Keyboard Selection (new behavior)
- Shift+Arrow extends selection from cursor position
- Highlight persists (drawn in sel-layer, stays visible)
- Ctrl+C with active keyboard selection â†’ copy to clipboard + clear selection
- Ctrl+C without active selection â†’ send C-c to terminal
- Escape with active keyboard selection â†’ clear selection (don't unfocus)
- Any non-selection keystroke (without shift) â†’ clear selection + send key
- Mouse click â†’ clear keyboard selection

### How to distinguish them

Add a flag: `selMode = null | 'mouse' | 'keyboard'`

- Mousedown sets `selMode = 'mouse'`
- Shift+Arrow sets `selMode = 'keyboard'`
- `selMode` determines behavior in Ctrl+C and clear logic

### Ctrl+C behavior:
```javascript
if (e.ctrlKey && e.key === 'c') {
  if (selMode === 'keyboard' && selStart) {
    // Keyboard selection active â€” copy and clear
    const text = getSelectedTextFromSvg(t);
    if (text) copyToClipboard(text);
    clearSel();
    selMode = null;
    return;
  }
  // No keyboard selection â€” send C-c to terminal
  t.sendInput({ type: 'input', keys: 'c', ctrl: true });
  return;
}
```

### Clear logic for keyboard selection:
```javascript
// Any non-modifier, non-selection, non-Ctrl+C keystroke clears keyboard selection
if (selMode === 'keyboard' && selStart) {
  // But still send the key to terminal
  clearSel();
  selMode = null;
  // fall through to send the key
}
```

### Escape behavior:
```javascript
if (e.key === 'Escape') {
  if (selMode === 'keyboard' && selStart) {
    clearSel();
    selMode = null;
    e.preventDefault();
    return;
  }
  return; // existing unfocus behavior
}
```

## Race Condition: Mouse Fade vs Keyboard Selection

If mouse selection is fading and user starts Shift+Arrow:
- Shift+Arrow sets `selMode = 'keyboard'`, sets `selStart`/`selEnd`
- The mouse fade's `requestAnimationFrame` loop is still running
- When fade completes, it calls `clearSel()` which clears the keyboard selection's globals

**Fix:** The fade's clearSel should check `selMode`. If `selMode === 'keyboard'`, don't clear â€” a new keyboard selection has started during the fade.

```javascript
// In fade completion:
if (progress >= 1) {
  // Only clear if no new selection has started
  if (selMode !== 'keyboard') {
    clearSel();
  } else {
    // Just clear the fading rects, don't touch selStart/selEnd
    while (layer.firstChild) layer.removeChild(layer.firstChild);
    // Redraw the keyboard selection
    drawSelHighlight(fadeTerminal);
  }
}
```

## Summary of Changes Needed

1. Add `let selMode = null;` global â€” `'mouse'` or `'keyboard'`
2. Mousedown handler sets `selMode = 'mouse'`
3. Shift+Arrow handler sets `selMode = 'keyboard'`
4. Ctrl+C: if `selMode === 'keyboard'` â†’ copy + clear. Otherwise â†’ send C-c.
5. Escape: if `selMode === 'keyboard'` â†’ clear selection, stay focused
6. Any non-selection keystroke: if `selMode === 'keyboard'` â†’ clear selection, send key
7. Fade completion: check `selMode` before clearing
8. Mouse click: clear keyboard selection if active
