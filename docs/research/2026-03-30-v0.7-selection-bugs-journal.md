## Current State of Keyboard Selection

### What happens right now when user presses Shift+Arrow:

1. Dashboard intercepts keydown (line 2162)
2. Checks `e.shiftKey && arrow key` — matches
3. If no existing selection, initializes `selStart` from `t._lastCursor` (which may be stale)
4. Extends `selEnd` by one cell in the arrow direction
5. Draws highlight rects in SVG sel-layer
6. Calls `return` — **key never reaches the terminal**
7. Cursor in terminal does NOT move

### What doesn't happen:

- No auto-copy (removed in v3 mouse rework)
- No way to dismiss (keystroke-clear handler removed in v3)
- No Ctrl+C copy (handler removed in v3 — Ctrl+C always sends C-c)
- Shift release does nothing (not tracked)

### Result: orphaned highlight rects that sit there forever with no way to use or clear them.

## What Does Shift+Arrow Mean to tmux?

```
tmux list-keys | grep S-Up/Down/Left/Right
```

With prefix (Ctrl+B): `refresh-client` (viewport panning). Not relevant.
Without prefix: **passes through to application.** For shell/readline: usually same as plain arrow. For Claude Code TUI: varies (may do nothing, may navigate).

**Key insight:** Shift+Arrow is not meaningful to most terminal applications. Our interception is fine — we're not stealing a useful keystroke.

## The Cursor Problem

Keyboard selection does NOT advance the terminal cursor because we `return` before sending any key. This means:

- Selection start position comes from `_lastCursor` (stale server data)
- The selection and cursor are completely independent
- After selection, the cursor is still wherever it was — typing resumes there, not at the selection end
- If the user wanted to "select and delete" (like in a text editor), they can't — the cursor didn't move

**In a normal text editor:** Shift+Arrow moves the cursor AND extends selection. The cursor IS the selection endpoint.

**In a terminal:** The cursor belongs to the running process (shell, vim, Claude Code). The dashboard can't move it arbitrarily — it can only send keystrokes. And plain Arrow keys have application-specific meaning (not always "move cursor right by one character").

**Conclusion:** Terminal selection is fundamentally a read-only overlay. It can't interact with the cursor. This is the same as how iTerm2, Windows Terminal, etc. handle it — selection is visual, cursor is independent, they exist in different layers.

## What About Just Sending Arrow Keys?

User asked: "shift-arrow remove the shift — regarding the commands that get back to the terminal tmux so it's advancing the arrow."

This would mean: Shift+Arrow sends a plain Arrow to the terminal AND extends selection. The cursor moves and the selection tracks it.

**Problems:**
1. Arrow keys don't always mean "move cursor one position" — in vim it navigates, in less it scrolls, in Claude Code's TUI it does menu navigation
2. The selection would be based on assumptions about what the terminal DID with the arrow key, but we can't know for sure
3. If the arrow key causes output (like scrolling a page in less), the selection coordinates become meaningless
4. We'd need to wait for the terminal to respond and re-read the cursor position to know where it actually went

**This approach is fragile.** It assumes cursor-in-line-editor behavior which isn't always true.

## Shift Release as "Selection Complete"

Browser fires `keyup` for Shift. We already track `keyup` for Control and Alt (lines 927-929). Adding Shift tracking is trivial.

**Proposed flow:**
```
Shift+Arrow (hold) → extend selection, draw highlight
Shift+Arrow (hold) → keep extending
Shift released → "selection complete":
  1. Auto-copy selected text to clipboard
  2. Flash highlight bright
  3. Fade out over 2 seconds
  4. clearSel()
```

This makes keyboard selection behave identically to mouse selection:

| | Mouse | Keyboard |
|---|---|---|
| Start extending | Mousedown + drag | Shift + Arrow |
| Keep extending | Mousemove (held) | Shift + Arrow (repeated) |
| Complete | Mouseup | Shift release |
| Auto-copy | On mouseup | On Shift release |
| Visual feedback | Flash + 2s fade | Flash + 2s fade |
| Ctrl+C | Sends C-c (no selection) | Sends C-c (no selection) |

Same lifecycle. Same visual feedback. Same code path after the "complete" trigger.

## Ctrl+C Question

User asked: "Ctrl+C is a dangerous terminal command, we may block it actually."

Options:
1. **Block Ctrl+C entirely** — user can't interrupt processes from the web dashboard. They'd need SSH. Safest but limiting.
2. **Require double Ctrl+C** — first press shows "Press Ctrl+C again to interrupt." Second press sends C-c. Prevents accidents.
3. **Keep as-is** — single Ctrl+C sends SIGINT. Power users expect this. Claude Code handles interrupts gracefully.
4. **Ctrl+C = copy, Ctrl+Shift+C = interrupt** — browser convention for copy, explicit interrupt requires shift. Breaks terminal muscle memory.

With the Shift-release auto-copy model, Ctrl+C for copy becomes unnecessary — the clipboard already has the text when Shift releases. This removes the primary reason to intercept Ctrl+C for copy, leaving it free for its terminal purpose (SIGINT).

## Edge Cases to Handle

### Shift released without any Shift+Arrow selection
User just pressed and released Shift (no arrow). `selStart` is null. Don't trigger auto-copy. This is the normal case — Shift alone does nothing.

### Shift+Arrow during mouse fade
Mouse selection is fading (rAF loop running). User starts Shift+Arrow. New selection starts, old fade is still running. The `selMode` flag from v0.6 solves this — fade checks `selMode !== 'keyboard'` before clearing.

### Shift+Tab (cycle terminals)
Shift+Tab is already handled separately (line 2154: `if (e.key === 'Tab' && e.shiftKey && focusedSessions.size > 1) return;`). It returns before reaching the Shift+Arrow check. No conflict.

### Tab/Window loses focus during Shift+Arrow
`window.blur` fires. Shift is "released" but not via keyup. Need to clear keyboard selection state on blur, similar to how we clear `ctrlHeld`/`altHeld`.

### Shift+Home / Shift+End
Already handled — these are in the `includes` check (line 2162). Home selects to column 0, End to last column. Same lifecycle as Shift+Arrow.

---

## Implementation Outline

1. Add `let shiftHeld = false;` and `let selMode = null;` globals
2. Track Shift keydown/keyup/blur (same as Ctrl/Alt tracking)
3. Shift+Arrow handler: set `selMode = 'keyboard'`, extend selection as before
4. Shift keyup handler: if `selMode === 'keyboard'` and `selStart`, trigger auto-copy + fade
5. Mouse selection mousedown: set `selMode = 'mouse'`
6. Fade completion: check `selMode` before clearing
7. Ctrl+C: no selection check needed — always sends C-c
