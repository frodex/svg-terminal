# Event-Driven Terminal Updates Journal v0.1
**Date:** 2026-03-31
**Session:** Brainstorming replacement for 30ms polling
**Status:** In Progress
**Preceding:** none
**Changes from v0.0:** Initial journal

## Problem

svg-terminal server polls every tmux session at 30ms intervals using `execFile('tmux', ['capture-pane', ...])`. With 12 sessions, that's ~400 child process spawns per second. Idle terminals consume 30-36% CPU doing nothing useful — `diffState` returns null every time.

## What We Want

- Zero CPU when terminals are idle
- Same or better latency when terminals are active
- Scrollback must work (browser requests historical lines, server provides them)
- No loss of existing features (SGR colors, cursor, title, selection, copy/paste)

## What Claude-Proxy Already Does

Claude-proxy solved this for its own sessions using `@xterm/headless`:

```
┌─────────────────────────────────────────────────────┐
│ claude-proxy (working today)                        │
│                                                     │
│  PTY (node-pty)                                     │
│    │                                                │
│    │ raw bytes (program output)                     │
│    ▼                                                │
│  @xterm/headless (vterm)                            │
│    │  ├── maintains screen buffer in memory         │
│    │  ├── scrollback buffer (configurable depth)    │
│    │  └── fires onRender() when screen changes     │
│    │                                                │
│    ▼                                                │
│  screen-renderer.ts                                 │
│    │  reads vterm.buffer cells → Span[]             │
│    │  (colors, bold, italic already resolved)       │
│    │                                                │
│    ▼                                                │
│  api-server.ts                                      │
│    │  30ms batch timer (only when dirty)            │
│    │  diffState → delta or full screen              │
│    │                                                │
│    ▼                                                │
│  WebSocket → browser                                │
│    { type: "screen", lines, width, height }         │
│    { type: "delta", changed: { idx: spans } }       │
└─────────────────────────────────────────────────────┘
```

Key: `onRender()` is the event that replaces polling. It only fires when something actually changes on screen. No polling loop needed.

## Open Questions (unresolved)

1. **How do we get raw bytes from local tmux sessions?**
   - tmux control mode (`tmux -C attach`) sends `%output` with base64-encoded bytes
   - node-pty to own the PTY directly (bypasses tmux entirely — loses tmux features)
   - tmux pipe-pane (pipes pane output to a file/command)
   - Something else?

2. **Do we need @xterm/headless in svg-terminal?**
   - If yes: add as dependency, maintain vterm per session, use screen-renderer
   - If no: need another way to get event-driven updates without polling

3. **Scrollback on server restart:**
   - xterm-headless buffer is lost on restart
   - Could hybrid: vterm for live screen, tmux capture-pane for deep scrollback
   - Could pre-load vterm with tmux history on startup

4. **Can we use option B instead?**
   - Control mode as change DETECTOR only
   - `%output` notification → trigger one `capture-pane` call
   - No xterm-headless needed, no new dependency
   - Keeps sgr-parser.mjs, capturePane(), entire existing pipeline
   - Just eliminates blind polling — only capture when something changed
   - Scrollback unchanged (still tmux)

5. **Should local sessions just go through claude-proxy?**
   - claude-proxy already solved all of this
   - svg-terminal becomes a pure frontend, no tmux interaction
   - But: tight coupling, single point of failure, architecture change
