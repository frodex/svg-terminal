## Context

Coming in after a server meltdown caused by stale browser clients + updated server code. 170 TCP connections, 55% CPU, connection cycling. Immediate fix: isolated server to port 3201. Previous agent (v0.1 journal author) proposed xterm/headless adoption — that analysis contained unverified claims that would have degraded the product.

## What v0.1 Got Wrong

### 1. "xterm/headless provides scrollback"
**Wrong.** Claude-proxy commit `7aa1323` ("fix: scroll uses tmux capture-pane instead of xterm buffer") proves xterm/headless has NO scrollback in alternate screen mode. Claude-proxy had to fall back to `tmux capture-pane -S -E` — the exact same mechanism svg-terminal already uses.

Adopting xterm/headless (v0.1 Option A) would have:
- Added a dependency to a zero-dep server
- Required maintaining a second buffer per session
- Lost scrollback in alternate screen mode
- Required a fallback to tmux capture-pane anyway (defeating the purpose)

### 2. "Option B needs validation"
Option B (tmux control mode as change detector) was presented as speculative. It was never researched, never tested, and the open questions were left unanswered. The entire v0.1 journal was brainstorming, not research.

### 3. Missing the real structural problem
v0.1 identified the 30ms polling as the problem but missed the critical multiplier: **each WebSocket connection runs its own independent poll loop**. N browsers viewing the same session = N independent `tmux capture-pane` calls every 30ms. The per-connection architecture is the actual scaling bottleneck, not just the poll interval.

## What We Actually Verified

### Scrollback architecture (WORKING, DO NOT CHANGE)
- Browser sends `{ type: 'scroll', offset: N }` via WebSocket
- Server calls `capturePaneAt(session, pane, offset)` → `tmux capture-pane -p -e -S {-offset} -E {-offset+height-1}`
- tmux provides full history (depth = tmux `history-limit`, default 2000 lines)
- Survives server restarts because tmux is a separate daemon
- Scroll position is ephemeral in-memory state (resets on disconnect) — this is correct behavior

### Rendering pipeline (FRAGILE, DO NOT CHANGE)
```
tmux capture-pane -p -e → raw ANSI lines
  → sgr-parser.mjs parseLine() → Span[] per line
  → diffState() → screen or delta message
  → WebSocket → terminal.svg
  → updateLine() creates SVG <tspan> elements with x-positioned spans
  → rebuildBgLayer() creates <rect> elements for background colors
```
Every stage has specific data shape contracts. The Span format includes: text, cls, fg, bg, bgCls, bold, italic, underline, dim, strikethrough, url. Standard colors (0-15) use CSS classes, extended (16-255) use hex strings. Breaking any of these contracts breaks rendering.

### Claude-proxy's actual approach
Claude-proxy uses xterm/headless as a **viewport renderer + change detector**, NOT as a scrollback store:
- `onWriteParsed` fires when xterm finishes parsing ANSI data → sets `dirty = true`
- 30ms timer checks dirty flag → only renders and sends if dirty
- For scrollback: falls back to `tmux capture-pane` (same as svg-terminal)
- Has its own `parseAnsiLine()` which is basically sgr-parser.mjs in TypeScript

The key insight: the only thing xterm/headless provides that svg-terminal doesn't have is the **event-driven dirty detection**. We can get the same effect without xterm/headless.

## Legacy Code Audit: New Sessions vs Old Sessions

### Discovery: Two eras of sessions
- **Old sessions:** Pre-WebSocket tmux sessions. Still alive, running old agents as reference sources. Will be terminated soon. No new ones will be created.
- **New sessions:** WebSocket-enabled. All future sessions. Two types: local tmux (via `handleTerminalWs`) and claude-proxy (via WS proxy bridge).

### What new sessions actually need (complete path)

**Discovery:** `GET /api/sessions` → dashboard polls 5s → `addTerminal()`
**Connection:** `WebSocket /ws/terminal?session=X&pane=Y`
  - Local tmux → `handleTerminalWs()` → capture + diff + push via WS
  - Claude-proxy → WS proxy bridge to `localhost:3101`
**Input:** `ws.send({ type: 'input', ... })` → routed to tmux via WS handler
**Scroll:** `ws.send({ type: 'scroll', offset: N })` → capture at offset via WS handler
**Resize:** `ws.send({ type: 'resize', cols, rows })` → `tmux resize-window` via WS handler
**Titles:** Arrive in every screen/delta WS message via `title` field

Everything flows through the single WebSocket per terminal card.

### Legacy code identified (only needed for old sessions)

**terminal.svg — HTTP polling fallback:**
- `poll()` function (line 354-410) — fetches `/api/pane` over HTTP
- `schedulePoll()` / `startPolling()` (line 144-156, 429-437)
- `pollInterval` / `pollTimer` state (line 120-122)
- Tier measurement / IntersectionObserver for adaptive poll rates (line 415-465)
- Safety net `schedulePoll(pollInterval)` on startup (line 608) — THIS causes 404/500 errors on new sessions because it fires before WebSocket connects

**server.mjs — HTTP data endpoints:**
- `handlePane()` (line 202-214) — `GET /api/pane` for HTTP polling
- `handleInput()` (line 241-286) — `POST /api/input` for HTTP keystroke input

**dashboard.mjs — HTTP title polling:**
- `fetchTitle()` (line 1518-1525) — hits `GET /api/pane` for every session including claude-proxy ones → generates 500 errors
- `refreshTitles()` + `setInterval(refreshTitles, 10000)` (line 510-515, 563)

### Source of console errors (404/500/502)
1. `refreshTitles()` calls `fetchTitle()` which hits `GET /api/pane` for EVERY session — claude-proxy sessions return 500 because `capturePane()` can't find them in local tmux
2. terminal.svg fires `schedulePoll(pollInterval)` as "safety net" before WebSocket connects — races with WS, hits `/api/pane` for sessions that only work over WebSocket

Both are legacy code paths that new sessions don't need. Titles already arrive via WebSocket.

## Corrected Architecture Direction

### Phase 1: Shared capture per session (biggest win, smallest change)
Replace per-connection poll loops with one capture loop per session:pane. All WebSocket clients for the same session:pane share the captured state. Diff once, broadcast to many.

Impact: 30 browsers × 1 session goes from 30 capture-pane/30ms to 1.

### Phase 2: Adaptive backoff (stop burning CPU on idle)
When `diffState` returns null N times in a row, back off: 30ms → 100ms → 500ms → 1000ms. On any input from any client, snap back to 30ms.

Impact: Idle sessions approach zero CPU.

### Phase 3: Deprecate HTTP polling path
Mark legacy code, remove safety-net poll in terminal.svg, remove `refreshTitles()`. Titles from WebSocket only.

Impact: Eliminates console errors, removes dead code paths.

### Phase 4 (future, optional): True event-driven
Replace adaptive polling with tmux control mode or pipe-pane change detection. Only worth doing if Phase 1+2 don't reach 30×30 target.

### What does NOT change
- sgr-parser.mjs
- terminal.svg rendering (updateLine, rebuildBgLayer, rebuildLinkLayer)
- WebSocket message format (screen/delta)
- Scrollback via tmux capture-pane -S -E
- SSE command channel
- Claude-proxy WS proxy bridge
- Session discovery
- tmux sessions (sacred)
- claude-proxy server (stable)

## Open Questions

1. **Shared capture refactor:** Should the SessionWatcher be a Map of active watchers keyed by `session:pane`, with a subscriber list? Or simpler — just dedup at the `setInterval` level?
2. **Adaptive backoff curve:** What's the right progression? 30ms → 100ms → 500ms → 1000ms? Or more aggressive?
3. **Input snap-back:** When any client sends input, ALL clients for that session should get fast polling. How to propagate?
4. **Legacy code timeline:** Mark deprecated now, remove when old sessions are gone? Or feature-flag?
