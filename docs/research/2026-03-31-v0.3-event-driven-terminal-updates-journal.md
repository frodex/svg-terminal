# Event-Driven Terminal Updates Journal v0.3
**Date:** 2026-03-31
**Session:** First-principles redesign
**Status:** In Progress
**Preceding:** 2026-03-31-v0.2-event-driven-terminal-updates-journal.md
**Changes from v0.2:** Dropped adaptive backoff as a phase. Reframed from "optimize polling" to "replace polling with change detection." Identified the actual broken step.

## What Changed From v0.2

v0.2 was still thinking inside the existing architecture — "how do we poll less?" with shared captures and backoff curves. That's optimizing a workaround instead of fixing the actual problem.

This version starts from first principles: what does the system actually need to do?

## First Principles Analysis

The system does five things:

1. **Detect that a terminal has new output**
2. **Capture the screen state** (`tmux capture-pane`)
3. **Parse it into structured data** (sgr-parser → Spans)
4. **Send it to browsers** (WebSocket)
5. **Render it** (terminal.svg → SVG tspans)

Steps 2-5 work correctly. Step 1 doesn't exist. The current system substitutes a blind 30ms poll loop for step 1. That was the first thing that worked and it was never rethought.

## The Fix

**tmux control mode** provides step 1.

Tested on this system (tmux 3.5a):
```
tmux -C attach-session -t SESSION
> refresh-client -A '%PANE_ID:on'

Receives:
%output %818 echo FINAL_TEST_OUTPUT\015\012
%output %818 FINAL_TEST_OUTPUT\015\012
```

- `%output` fires when a pane produces output
- Sub-millisecond latency
- Zero CPU when idle
- One control client can watch multiple panes via multiple `-A` commands
- Output must be enabled per-pane with `refresh-client -A 'PANE:on'`
- stdin must stay open (control client dies if stdin closes)
- Escape sequences in output are octal-encoded (need unescaping)

## Target Architecture

```
tmux control mode
  │
  │  %output PANE_ID DATA
  ▼
server: "pane X changed"
  │
  │  capturePane(session, pane)  ← one capture
  │  sgr-parser → Spans
  │  diffState()
  ▼
broadcast to all WebSocket clients watching session:pane
  │
  ▼
terminal.svg renders (unchanged)
```

No polling. No per-connection loops. No backoff curves. Capture on change, broadcast to subscribers.

Scrollback unchanged — browser sends `{ type: 'scroll', offset: N }`, server calls `capturePaneAt()` with tmux `-S`/`-E` flags. tmux is the scrollback store.

## What Gets Removed

All of this was a substitute for not having step 1:

- `setInterval(captureAndPush, 30)` per WebSocket connection
- `poll()` / `schedulePoll()` / `startPolling()` / tier measurement in terminal.svg
- `handlePane()` HTTP endpoint in server.mjs
- `handleInput()` HTTP endpoint in server.mjs
- `refreshTitles()` / `fetchTitle()` in dashboard.mjs (titles arrive via WebSocket already)

## What Stays

- `capturePane()` / `capturePaneAt()` — still how we get screen data from tmux
- `sgr-parser.mjs` — still how we parse ANSI → Spans
- `diffState()` — still how we compute deltas
- `handleTerminalWs()` — refactored from "poll loop owner" to "subscriber to shared watcher"
- terminal.svg WebSocket receiver + SVG rendering — unchanged
- Claude-proxy WS proxy bridge — unchanged (already event-driven on that side)
- SSE command channel — unchanged, available for future use
- `refreshSessions()` 5s poll — stays for now (session discovery, not terminal data)

## Open Questions

1. **One control client per session, or one global control client?** tmux control mode attaches to a session. Can we share one and watch panes across sessions? Or do we need one per session?

2. **Debouncing:** High-throughput output (e.g. `cat bigfile.txt`) will fire many `%output` events. We need to coalesce these — capture once after a burst, not once per `%output`. The existing 30ms batch window is roughly right. But now it's "capture 30ms after last %output" not "capture every 30ms regardless."

3. **Startup/reconnect:** When a browser connects to a session that's been idle, there's no `%output` pending. Need to do one initial capture on connect (same as today's `await captureAndPush()` on line 441).

4. **Control client lifecycle:** What happens when a tmux session is created or destroyed? The control client needs to track active sessions and subscribe/unsubscribe panes.

5. **SSE role:** The SSE channel exists and works. Does it play a role here, or is the WebSocket sufficient for terminal data delivery? SSE could be useful for session-level notifications ("new session created", "session terminated") separate from terminal content.

## Rejected Approaches (With Reasons)

### xterm/headless (v0.1 proposal)
- Loses scrollback in alternate screen mode
- Claude-proxy proved this — had to fall back to tmux capture-pane
- Adds dependency, adds complexity, solves a problem tmux control mode solves better

### Adaptive backoff (v0.2 proposal)
- Optimizes polling instead of eliminating it
- Still burns CPU proportional to session count when idle
- Adds complexity (backoff curves, snap-back logic) for an incomplete fix
- If control mode works (it does), backoff is unnecessary

### inotify on PTY devices
- Tested: inotify doesn't monitor PTY buffers
- /dev/pts/N is a character device, not a regular file

### PTY file descriptor watching
- tmux already owns the PTY — reading from it would race with tmux
- Can't have two readers on the same PTY

### pipe-pane
- Works (tested), but gives raw bytes not change notifications
- Would need a parser to determine "something changed" — that's what control mode already does
- Could be complementary for raw byte forwarding in future
