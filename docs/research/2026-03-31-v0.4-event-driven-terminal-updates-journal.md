## How We Got Here

v0.2 (first pass): Proposed shared capture + adaptive backoff. Optimizing the wrong thing.
v0.3 (first principles attempt): Proposed tmux -C as event trigger. Sounded clean but added complexity for minimal gain.
v0.4 (this version): Worked through the implications of tmux -C with the user. Benchmarked actual costs. Arrived at a simpler answer.

## The Real Problem (Verified)

The server melted not because polling is inherently wrong, but because every WebSocket connection runs its own independent poll loop. N browsers × M sessions = N×M captures every 30ms. 30 browsers × 30 sessions = 900 captures every 30ms.

The fix: one capture per session, broadcast to all browsers. 900 → 30.

## Why tmux -C Was Dropped

### What tmux -C does

`tmux -C attach-session -t SESSION` opens a persistent control mode connection. When a pane produces output, tmux sends `%output %PANE_ID escaped-data` on stdout. The data is the raw bytes the program wrote — cursor movement, scroll regions, colors, everything.

### The tension

`capture-pane -e` was chosen on day one (2026-03-27, see bibliography) specifically because it gives **pre-rendered lines with only SGR codes** — tmux does the terminal emulation and hands back clean text. The alternative — interpreting raw terminal bytes — requires a terminal emulator (xterm/headless), which we rejected.

`%output` sends raw bytes (before tmux processes them). `capture-pane` reads the rendered screen (after tmux processes them). They're the same data at different stages of tmux's pipeline:

```
Program writes bytes → tmux receives raw bytes
                         │
                         ├─→ %output: raw bytes BEFORE processing
                         │   (cursor movement, scroll, clear, color — everything)
                         │
                         ▼
                       tmux internal screen buffer
                       (tmux interprets all control codes here)
                         │
                         ├─→ capture-pane: rendered lines AFTER processing
                         │   (clean text with SGR only, no cursor movement)
```

### What we'd actually be doing with tmux -C

Using `%output` purely as a trigger — "something happened" — then throwing away the data and calling `capture-pane` anyway for the rendered screen. Two connections to tmux where we had one. A persistent process per session to manage (lifecycle, errors, stdin kept open). All to avoid a timer.

### The math doesn't justify it

Benchmarked on this system (see below). Shared polling at 30ms handles 30 sessions in 16ms with room to spare. The problem was 900 polls, not 30.

## Benchmarks (Measured on This System)

### capture-pane cost

30 sequential calls:
```
real 0m0.058s → ~2ms each
```

30 concurrent calls:
```
real 0m0.013s → parallelizes well, ~0.4ms effective per call
```

### sgr-parser cost

30 screens (41 lines each) parsed:
```
2.6ms total → 0.09ms per screen
```

Parsing is essentially free. The cost is dominated by child process spawn.

### Total per poll cycle (30 sessions, shared capture)

- Child process spawns: ~13ms (concurrent)
- SGR parsing: ~2.6ms
- **Total: ~16ms per 30ms cycle**
- Leaves 14ms headroom for diff, serialization, WebSocket send

### Comparison

| Architecture | Captures per 30ms | Estimated wall time |
|---|---|---|
| Current (per-connection, 30 browsers × 30 sessions) | 900 | server melts |
| Current (per-connection, 1 browser × 25 sessions) | 25 | ~40% CPU (measured) |
| Shared capture (30 sessions) | 30 | ~16ms |
| Shared + tmux -C trigger | 0-30 (event-driven) | 0-16ms |

Shared capture alone gets us from 900 to 30. tmux -C gets us from 30 to 0-30. The marginal gain doesn't justify the added complexity.

## Architecture Direction (Revised)

### What changes

**Server: One poll loop per session, not per connection.**

When a browser's WebSocket subscribes to a session (via the shared WebSocket), the server ensures a capture loop exists for that session. The loop captures once every 30ms, diffs once, broadcasts the result to all subscribers. When the last subscriber disconnects, the loop stops.

**Single WebSocket per browser, server broadcasts everything.**

Browser authenticates on connect. Server sends all session data the user can see. Browser renders what it has cards for, ignores the rest. No per-session WebSocket connections. No subscription management — server pushes everything.

**Server-side parsing may move to browser (open question).**

Server currently runs `parseLine()` on every line before sending. The raw ANSI strings from `capture-pane` are actually smaller than the JSON Span objects. Browser could parse instead. This would make the server a pure pass-through: capture → tag with session → send raw lines. Needs more analysis — terminal.svg currently expects pre-parsed Spans.

**Legacy HTTP polling paths removed.**

`poll()`, `handlePane()`, `handleInput()`, `refreshTitles()`, `fetchTitle()` — all built before WebSocket, all dead code for new sessions.

### What doesn't change

- `capture-pane` as the data source (tmux does the terminal emulation)
- `sgr-parser.mjs` (wherever it runs — server or browser)
- terminal.svg SVG rendering
- Scrollback via `tmux capture-pane -S -E`
- tmux sessions (never touched)

### Open questions

1. **Where does parsing happen?** Server (current) or browser (possible)? Raw ANSI strings are smaller than JSON Spans. Browser is a rendering engine. But terminal.svg expects Spans. Needs investigation.

2. **Is diffState needed?** With shared capture, the diff saves bandwidth (only send changed lines). But on a LAN, sending full screens (~5KB each) for 30 sessions to 30 browsers is ~4.5MB/sec. Trivial. Could simplify by dropping diff entirely.

3. **Raw string diff instead of Span diff?** If keeping diff, compare raw ANSI strings before parsing. Only parse changed lines. Cheaper than parse-everything-then-JSON.stringify-to-compare.

4. **Single WebSocket subscription model:** Browser authenticates, server pushes everything. How does auth integrate with the WebSocket upgrade? What happens when sessions are created/destroyed — server pushes session-add/session-remove events?

5. **What else depends on the 30ms poll?** Cursor position, title, dimensions, dead flag, historySize all come from the capture. If we slow the poll or make it event-driven later, those updates slow down too. Is that OK?

## What v0.1-v0.3 Got Wrong

### v0.1: xterm/headless
Wrong data source. Loses scrollback. Adds dependency. Claude-proxy proved it doesn't work for scroll (commit 7aa1323).

### v0.2: Adaptive backoff
Optimizing the wrong dimension. The problem was N×M captures, not the interval. Backoff adds complexity (curves, snap-back) for an incomplete fix.

### v0.3: tmux -C as trigger
Sounded clean in theory. In practice: adds a persistent process per session, throws away the data it receives, still calls capture-pane. Two tmux connections where one suffices. The benchmarks show shared polling handles the load fine.

### Common thread
Each version tried to make the polling "smarter" without questioning whether the actual bottleneck was polling frequency. It wasn't. It was duplication. The simplest fix — share the capture — was always the right first step.

## Bibliography Findings

From project bibliography (2026-03-27):

> [tmux Control Mode wiki] — `%output` notifications for real-time streaming, `pause-after` flow control. **Not adopted for POC — capture-pane polling is simpler**

> [tmux man page — capture-pane] — **`capture-pane -p -e` gives SGR-only re-serialized screen state (no cursor movement codes)**

> [termtosvg] — `CELL_WIDTH=8 CELL_HEIGHT=17`, line-by-line SVG rendering

The line-oriented architecture was a deliberate day-one decision. `capture-pane` gives pre-rendered lines. SVG renders one `<text>` per row. Server parses per line. Diff compares per line. Everything is line-oriented because `capture-pane` outputs lines. This was the right call for the POC — no terminal emulator needed.

Server-side parsing was natural because the original system was HTTP (`GET /api/pane` returning JSON). When WebSocket replaced HTTP, nobody revisited whether the server should still be parsing. The transport changed, the pipeline didn't.
