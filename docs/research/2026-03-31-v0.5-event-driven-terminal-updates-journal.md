# Event-Driven Terminal Updates Journal v0.5
**Date:** 2026-03-31
**Session:** Final synthesis before PRD amendment
**Status:** Direction Confirmed
**Preceding:** 2026-03-31-v0.4-event-driven-terminal-updates-journal.md
**Changes from v0.4:** Consolidated all findings, tradeoffs, and decisions from user discussion. Added throttle via SSE. Confirmed direction: shared capture + single WebSocket + tunable poll + server-side throttle.

## Decision Summary

### The Problem
Server melted because each browser terminal card opens its own WebSocket, each WebSocket triggers its own 30ms poll loop on the server. 30 browsers × 30 sessions = 900 independent captures every 30ms. The problem is per-connection duplication, not polling itself.

### The Fix (Three Changes)
1. **Single WebSocket per browser** — dashboard.mjs owns one WebSocket, routes messages to cards by session tag. 30 browsers = 30 connections (not 900).
2. **Shared capture per session** — one poll loop per session on the server, broadcasts to all connected browsers. 30 sessions = 30 captures per cycle (not 900).
3. **Tunable poll interval with server-side throttle** — default ~100ms (not hardcoded 30ms). Server monitors its own load and pushes throttle commands via SSE to slow browsers under pressure.

### What Was Considered and Rejected

**xterm/headless (v0.1):** Loses scrollback in alternate screen mode. Claude-proxy proved this — commit 7aa1323 fell back to tmux capture-pane. Adds dependency for no benefit.

**Adaptive backoff (v0.2):** Optimizes polling frequency instead of fixing the duplication problem. Adds complexity (backoff curves, snap-back logic) for an incomplete fix.

**tmux -C control mode as trigger (v0.3):** Adds a persistent process per session that sends raw terminal bytes (cursor movement, scroll regions, everything). We'd throw away the data and call capture-pane anyway for rendered lines. Two connections to tmux where one suffices. Day-one bibliography entry: "Not adopted for POC — capture-pane polling is simpler." The day-one reasoning still holds.

**tmux -C as data source:** Would require a terminal emulator to interpret raw bytes into a screen. That's what xterm/headless is. Already rejected.

**inotify on PTY devices:** Tested on this system. Doesn't work — inotify doesn't monitor character devices. tmux screen state lives in process memory, not on disk.

### Key Benchmarks (Measured 2026-03-31)

**capture-pane cost:**
- 30 sequential: 58ms (~2ms each)
- 30 concurrent: 13ms (~0.4ms each effective)
- Parallelizes well via async execFile

**sgr-parser cost:**
- 30 screens (41 lines each): 2.6ms total
- Per screen: 0.09ms
- Parsing is essentially free

**Total per shared poll cycle (30 sessions):**
- Captures + parsing: ~16ms
- Fits inside a 30ms window (14ms headroom)
- At 100ms interval: ~16% duty cycle (84ms idle per cycle)

### Key Design Decisions With Reasoning

**Why capture-pane and not raw bytes:**
`capture-pane -e` gives pre-rendered lines with only SGR codes. tmux has already interpreted cursor movement, scroll regions, clear commands. No terminal emulator needed. This was a deliberate day-one decision (2026-03-27 bibliography: termtosvg pattern, tmux man page reference). Raw bytes (from %output or pipe-pane) require a terminal emulator to interpret into a screen.

**Why server broadcasts everything (no per-user filtering):**
Simpler. No subscription management, no per-user tracking on the server. Browser renders what it has cards for, ignores the rest. Bandwidth is trivial on LAN (~5KB per screen × 30 sessions = 150KB per update cycle). Security is enforced at the session discovery level (server only reports sessions the user can access) and at tmux Unix permissions, not at the data delivery level.

**Why diffState stays (for now):**
tmux has no incremental API — capture-pane always returns the full screen. diffState prevents sending unchanged screens. With 30 sessions at 100ms intervals, most diffs return null most of the time. Without diff, we'd send ~150KB × 10/sec = 1.5MB/sec to each browser. With diff, we send only changed lines — typically a few KB/sec total. The diff cost (JSON.stringify per line) is negligible compared to capture-pane.

**Why 30ms was probably wrong from the start:**
No documentation exists for why 30ms was chosen. 30ms = 33fps, appropriate for video, overkill for terminal text. 100ms = 10fps, still feels responsive for typing. Terminal output appears in bursts (command finishes, Claude responds) not continuous animation. 100ms default with server-side throttle is more appropriate.

**Why server-side parsing may move to browser:**
Server currently runs parseLine() on every line. Raw ANSI strings from capture-pane are smaller than JSON Span arrays. Browser is a rendering engine — parsing is what it's for. Moving parsing to browser makes the server a pure pass-through: capture → tag → send. But terminal.svg currently expects pre-parsed Spans, so this requires browser-side changes. Open question — not blocking for Phase 1.

**Why SSE throttle instead of server-side backoff:**
Server knows its own load. SSE channel already exists. One message (`{ type: "throttle", interval: N }`) and browsers adjust. Server-side backoff (varying the capture interval) works too, but the SSE throttle gives the server explicit control over browser behavior during overload. Belt and suspenders: server can both slow its own captures AND tell browsers to poll less.

### Data Flow Through tmux

```
Program writes bytes → tmux receives raw bytes
                         │
                         ├─→ %output: raw bytes BEFORE processing
                         │   (cursor movement, scroll, clear, color)
                         │   NOT USED — would need terminal emulator
                         │
                         ▼
                       tmux internal screen buffer
                       (tmux interprets all control codes)
                         │
                         ├─→ capture-pane -e: rendered lines AFTER processing
                         │   (clean text with SGR only)
                         │   THIS IS WHAT WE USE
```

### Wire Format (Single WebSocket)

Every message is self-contained JSON. Session field identifies which terminal it's for. Messages for different sessions arrive interleaved on the same connection. No combining, no packaging, no demuxing — just tagged messages on a shared pipe.

**Server → Browser:**
- `{ session, pane, type: "screen", width, height, cursor, title, lines, scrollOffset }` — full screen
- `{ session, pane, type: "delta", cursor, title, changed: { "lineIdx": {spans} } }` — changed lines
- `{ type: "session-add", session, cols, rows, title, source }` — new session available
- `{ type: "session-remove", session }` — session gone
- `{ session, type: "error", message }` — error on specific session

**Browser → Server:**
- `{ type: "auth", token }` — first message, identifies user
- `{ session, pane, type: "input", keys|specialKey|ctrl }` — keystroke
- `{ session, pane, type: "scroll", offset }` — scrollback request
- `{ session, pane, type: "resize", cols, rows }` — terminal resize

**Via SSE (server push, separate channel):**
- `{ type: "throttle", interval: N }` — adjust poll/refresh rate
- `{ type: "reload" }` — force page reload (stale client recovery)

### Legacy Code Removal

All built before WebSocket existed or as safety nets during transition:

**terminal.svg:** poll(), schedulePoll(), startPolling(), pollInterval, pollTimer, RETRY_MS, tier measurement, IntersectionObserver, safety-net schedulePoll on startup (causes 404/500 on new sessions)

**server.mjs:** handlePane() GET /api/pane, handleInput() POST /api/input, per-connection setInterval in handleTerminalWs

**dashboard.mjs:** fetchTitle() via /api/pane, refreshTitles() + setInterval (titles come via WebSocket), HTTP POST fallback in sendInput()

### Open Questions for Implementation

1. **Where does parsing happen?** Server (current, works) or browser (simpler server, needs terminal.svg changes). Not blocking for Phase 1.
2. **Raw string diff:** Could compare raw ANSI strings before parsing, only parse changed lines. Optimization, not blocking.
3. **Auth integration:** Browser sends token on WebSocket connect. Server looks up user, determines visible sessions. How does this wire into existing OAuth/session-cookie system?
4. **Session lifecycle events:** Server pushes session-add/session-remove. Replaces the 5s refreshSessions HTTP poll. Dashboard creates/removes cards reactively.
5. **Reconnection:** Single WebSocket failure = all terminals dark. Dashboard handles reconnection with exponential backoff. On reconnect, server sends full screen for all sessions (re-sync).
