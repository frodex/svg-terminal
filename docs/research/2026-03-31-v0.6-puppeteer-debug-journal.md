# v0.6 Puppeteer Debug Journal -- Shared WebSocket Architecture

**Date**: 2026-03-31  
**Scope**: Automated testing of the dual WebSocket architecture (shared dashboard WS + per-card WS)  
**Branch**: camera-only-test  

## Context

The system runs two WebSocket architectures simultaneously during transition:
1. **Shared dashboard WS** (`/ws/dashboard`): single multiplexed connection per browser, server-side `SessionWatcher` captures and broadcasts to all subscribers
2. **Per-card WS** (`/ws/terminal?session=X`): legacy one-connection-per-card, each SVG `<object>` opens its own WS

Reports of empty cards, broken typing, server crashes under load, and excessive CPU from dual capture.

## Test Results

### Test 1: Card Content Loading -- PASS
All 16 cards loaded with content after 6 seconds. The shared WS `routeDashboardMessage()` correctly routes `screen`/`delta` messages to each card's SVG via `renderMessage()`. Pending message queue (`_pendingMessages`) handles the race between WS data arrival and SVG `<object>` load.

### Test 2: Typing on Local Sessions -- PASS (after test fix)
Typing via `page.keyboard` correctly flows through: `keydown` handler -> `t.sendInput()` -> `sendDashboardMessage()` -> shared WS -> server `handleDashboardWs` -> `tmux send-keys`. Screen updates arrive via the shared watcher within ~100ms.

Initial test failure was a false negative: the test was reading text content from the thumbnail SVG frame (which showed only measure text "MMMMMMMMMM") instead of the main card frame.

### Test 3: Typing on cp-* Sessions -- PASS
No errors when typing on claude-proxy sessions. Input routes through the shared WS to the `_cpUpstream` bridge WebSocket.

### Test 4: Connection Count -- FIXED
**Before fix**: 33 WebSocket connections (32 per-card + 1 dashboard)  
**After fix**: 17 WebSocket connections (16 per-card + 1 dashboard)

**Root cause**: Each card creates a sidebar thumbnail via `createThumbnail()`, which loads `terminal.svg?session=X` as an `<object>`. The SVG's startup code calls `connectWebSocket()` for any non-empty SESSION parameter. With 16 sessions, this created 16 extra per-card WS connections from thumbnails that received no data (server's capture loop is skipped when shared watcher exists).

### Test 5: Race Condition (Screen Before SVG Load) -- PASS
The pending message queue in `routeDashboardMessage()` correctly handles early data. Messages queue until SVG `<object>` fires its `load` event, then flush.

### Test 6: Server Health -- PASS
Server remains stable, no errors in logs.

### Test 7: Detailed Per-Card Audit -- PASS (after fix)
All 32 SVG frames (16 main + 16 thumbnail) show content.

## Bug Found and Fixed

### Thumbnail SVGs Opening Unnecessary WebSocket Connections

**File**: `dashboard.mjs` (createThumbnail), `terminal.svg` (startup)

**Problem**: `createThumbnail()` creates an `<object>` loading `terminal.svg?session=X`. The SVG startup code sees a non-empty SESSION and calls `connectWebSocket()`, opening a per-card WS to `/ws/terminal?session=X&pane=0`. The server's `handleTerminalWs()` connects but skips the capture loop (shared watcher exists), so these thumbnail WS connections receive zero data -- pure waste.

With 16 sessions, this doubled the connection count from 17 to 33.

**Fix (commit a15b01a)**:
1. Added `&thumb=1` parameter to thumbnail SVG URLs
2. Added `IS_THUMB` flag in `terminal.svg` that suppresses `connectWebSocket()`
3. Thumbnails now receive content via `renderMessage()` from the shared dashboard WS
4. Added pending message queue (`_thumbPendingMessages`) for thumbnail SVG load timing
5. `routeDashboardMessage()` now routes to both main card and thumbnail SVGs

## Remaining Observations

1. **Per-card WS connections from main cards still exist** (16 connections). These serve as the `sendToWs()` input fallback path. They could be eliminated entirely once the shared WS input path is confirmed stable, but during transition they provide redundancy.

2. **Dashboard WS "connected" message appears multiple times** across test runs. This is expected -- each `page.goto()` creates a new page load and new WS connection. Not a bug.

3. **The `_sharedWsActive` gate works correctly**: per-card WS `onmessage` is suppressed once `renderMessage()` is called, preventing dual-render from two capture streams.

4. **Server CPU**: The shared watcher capture loop runs at 100ms intervals per session regardless of subscriber count. With 16 sessions, that is 160 tmux captures/second total. The per-card WS capture loop (30ms interval) is correctly skipped when shared watcher exists.
