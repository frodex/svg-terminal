## Issues Found and Fixed During Live Testing

### 1. Claude-proxy input routing (commit 63b5733)
**Problem:** Typing into cp-* sessions failed with "can't find session" — `handleDashboardWs` sent all input to local tmux via `send-keys`, but cp-* sessions aren't in local tmux.
**Fix:** Check for `_cpUpstream` WebSocket on the session watcher. If present, forward input through the upstream bridge instead of local tmux.

### 2. Safety-net HTTP poll flooding (commits 677297f, 013a374)
**Problem:** terminal.svg had THREE paths that started HTTP polling: startup safety-net, WebSocket onclose fallback, IntersectionObserver. All hit `/api/pane` for sessions that don't exist locally → 500 errors → server overload → crash.
**Fix:** Disabled all HTTP poll start paths. Shared WS handles data delivery.

### 3. Dual-capture rendering conflict (commit a1facc4)
**Problem:** Both per-card WS (30ms) and shared WS (100ms) sent screen data to the same terminal.svg. Two capture streams at different cadences = cursor jumping, characters out of order.
**Fix:** Added `_sharedWsActive` flag in terminal.svg. Once `renderMessage()` is called (shared WS active), `ws.onmessage` returns early (per-card WS rendering suppressed).

### 4. Per-card capture loop doubling CPU (commit 96dd4ed)
**Problem:** Even with shared watchers running, per-card `handleTerminalWs` still started 30ms capture loops for each WebSocket connection. Doubled the capture work.
**Fix:** Skip `setInterval(captureAndPush, 30)` in `handleTerminalWs` when `sessionWatchers` already has a watcher for the session. Keep the message handler for input fallback.

### 5. Empty cards on load (commit 1f4ee31)
**Problem:** Shared WS sends `screen` data immediately after `session-add`, but the SVG `<object>` hasn't loaded yet. `renderMessage` doesn't exist. Message silently dropped. Card loads empty.
**Fix:** Queue messages in `_pendingMessages` array. Flush when SVG object fires `load` event.

### 6. Thumbnail WebSocket waste (commit a15b01a, from puppeteer debug agent)
**Problem:** 16 sidebar thumbnails each loaded terminal.svg which opened its own per-card WebSocket. 16 extra connections doing nothing (capture loop skipped).
**Fix:** Added `&thumb=1` URL parameter. terminal.svg skips `connectWebSocket()` in thumb mode. Dashboard routes shared WS data to thumbnails via `renderMessage()`.

### 7. Cursor lag during typing (commit d2394b5)
**Problem:** `triggerCapture` nulled `lastState` on every keystroke, forcing a full `screen` message instead of a small `delta`. During fast typing, multiple full screen dumps queued up, making the cursor visibly lag behind the text.
**Fix:** Removed `lastState = null` from `triggerCapture`. Delta messages are sufficient — they include cursor position and changed lines.

### 8. Capture interval too slow (commit 121d337)
**Problem:** CAPTURE_INTERVAL defaulted to 100ms. Interactive typing felt sluggish compared to old 30ms per-card polling.
**Fix:** Reduced default to 30ms. The real win is shared capture (M captures not N×M), not slower interval. Can tune upward via env or SSE throttle once stable.

## Stale Client Issue

Port 3200 had stale browser clients from other devices (running 4+ day old code). They flooded the server with failing HTTP requests on every restart. No Cloudflare `CF-Connecting-IP` header — traffic came from LAN IP 192.168.23.70 directly, not through Cloudflare tunnel.

Resolved by: stale clients eventually stopped (user may have closed them). Server stable on 3200 after that.

## Final State

- Server on port 3200, stable, 37% CPU with 1 browser
- Typing works through shared WS → server → tmux
- Scrolling works through shared WS
- Claude-proxy sessions work (input routed through upstream bridge)
- CPU will stay flat with additional browsers (shared capture)
- Per-card WebSocket connections still exist but capture loops skipped
- 30ms capture interval, tunable via CAPTURE_INTERVAL env
