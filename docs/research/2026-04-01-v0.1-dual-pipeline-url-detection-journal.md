# Dual Pipeline URL Detection — Journal v0.1
**Date:** 2026-04-01
**Session:** 0317c840 (svg-terminal lead)
**Status:** Resolved
**Preceding:** none (new topic)
**Changes from v0.0:** Initial journal documenting the dual-pipeline URL detection gap

## The Problem

URLs displayed in claude-proxy sessions (e.g., `http://192.168.22.56:55601`) were not being highlighted or clickable in the SVG terminal viewer, even though the same URLs worked correctly in local tmux sessions.

## Root Cause: Two Rendering Pipelines

svg-terminal has TWO distinct data pipelines, each producing span objects differently:

### Pipeline 1: Local tmux (sgr-parser.mjs)
```
tmux capture-pane -e → raw ANSI text → sgr-parser.mjs → spans with url property → SVG
```
`sgr-parser.mjs` has `tagPlainUrls()` (line 188) that scans each span's text for `http://` and `https://` patterns and sets `span.url = "http://..."`. The SVG viewer reads `span.url` to draw underlines and the dashboard reads it for click-to-open.

### Pipeline 2: claude-proxy (screen-renderer.ts)
```
xterm/headless buffer → screen-renderer.ts → spans WITHOUT url property → WebSocket → SVG
```
`screen-renderer.ts` converts xterm cells to spans using `lineToSpans()` and `cellColorToHex()`. It groups adjacent cells with the same attributes into text runs. **It does NOT scan for URLs.** No `span.url` is ever set.

### Why Pipeline 2 Exists
claude-proxy creates sessions on custom tmux sockets (`-S /var/run/claude-proxy/sessions/...sock`). These sessions are invisible to plain `tmux list-sessions` on the default server. The `screen-renderer.ts` pipeline reads from the xterm/headless buffer instead of using `tmux capture-pane`.

svg-terminal discovers these sessions via `GET http://127.0.0.1:3101/api/sessions` (claude-proxy's API) and proxies WebSocket connections to `ws://127.0.0.1:3101/api/session/:id/stream`.

## The Fix: Client-Side URL Detection

Rather than modifying `screen-renderer.ts` (which lives in claude-proxy's codebase and is owned by another agent), we added client-side URL detection that works regardless of which pipeline produced the spans.

### Fix 1: terminal.svg — `findUrlsInText()`
Added a function in the SVG viewer that scans span text for URL patterns:
```js
function findUrlsInText(text) {
  // Searches for http:// and https:// patterns
  // Walks forward to find URL end (whitespace or terminating chars)
  // Strips trailing punctuation
  // Returns array of {start, end, url}
}
```
`rebuildLinkLayer()` now:
1. First checks `span.url` (server-tagged, Pipeline 1)
2. If not tagged, runs `findUrlsInText()` on the span text (client-side detection)
3. Both paths draw blue underlines with `data-url` attribute

### Fix 2: dashboard.mjs — `getUrlAtCell()` fallback
The click handler that detects URL clicks also needed the same fallback:
1. First checks `span.url` on the stored spans
2. If not tagged, builds the full line text and runs a regex to find URLs at the clicked column
3. Both paths return the URL for `addBrowserCard()` or `window.open()`

## What Other Features Are Affected

This dual-pipeline gap affects ANY feature that depends on server-side span properties that only exist in one pipeline:

### Definitely Affected
1. **URL highlighting** — Fixed (this journal)
2. **URL click-to-open** — Fixed (this journal)

### Potentially Affected
3. **OSC 8 hyperlinks** — `sgr-parser.mjs` handles OSC 8 sequences (`\e]8;params;URL\e\\`) and sets `span.url`. `screen-renderer.ts` may or may not handle these. If xterm/headless processes OSC 8 internally, the cell attributes might include the hyperlink — needs investigation.

4. **Color class names** — `sgr-parser.mjs` maps the first 16 ANSI colors to CSS classes (`c0`-`c7`, `cb0`-`cb7`). `screen-renderer.ts` uses hex RGB for all colors. The SVG viewer handles both (`span.cls` for classes, `span.fg` for hex), so this works. But any feature that depends on `span.cls` specifically (e.g., CSS theming) might behave differently between pipelines.

5. **Background color classes** — Same as #4 but for `span.bgCls` vs `span.bg`.

6. **Text selection** — `getSelectedTextFromSvg()` reads from the SVG's DOM, which is pipeline-independent. Should be fine.

7. **Scrollback** — Pipeline 1 uses `tmux capture-pane -S/-E` offsets. Pipeline 2 reads xterm buffer line ranges. Different scrollback depths and behaviors. Currently the proxy path doesn't have scrollback implemented.

8. **Terminal title** — Pipeline 1 reads from `tmux display-message '#{pane_title}'`. Pipeline 2 gets title from claude-proxy's composed title (`session name | *sizeowner | uptime`). Different formats.

### Architecture Lesson
When two pipelines produce the same output format (spans) but with different properties filled in, every consumer needs to handle BOTH variants. The client-side fallback pattern (check server-tagged property first, fall back to client-side detection) is the right approach — it's defensive and works regardless of which pipeline produced the data.

The long-term fix is to unify the pipelines in claude-proxy v2 integration, where `screen-renderer.ts` would be the only pipeline and could include URL tagging. But until then, client-side fallbacks are necessary.
