## Understanding So Far

From reading the PRD (v0.4.0), dashboard.mjs, and the user's screenshot + description, here's my model of the system:

### The Rendering Stack
- tmux sessions → server.mjs (WebSocket) → SVG terminal rendering (inline SVG)
- Cards are DOM elements wrapped in CSS3DObject (Three.js CSS3DRenderer)
- 4x scale trick: card DOM is oversized, CSS3DObject scale = 0.25, forces Chrome to rasterize text at high resolution
- Camera-only focus model: cards never change DOM size on focus/unfocus, camera moves instead

### Card/Terminal Size Model
- **Card size (x, y):** physical DOM dimensions in pixels (e.g., 1290×1056)
- **Terminal size (cols, rows):** character grid dimensions of the tmux session
- **Cell size:** fixed at SVG_CELL_W = 8.65px, SVG_CELL_H = 17px
- **Apparent size to user:** function of card's position in 3D space (Z depth) and camera perspective
- calcCardSize(cols, rows) derives card DOM from terminal aspect ratio
- Two optimize directions: fit terminal→card, fit card→terminal

### Current Layout System (calculateFocusedLayout)
- Allocates screen area proportional to cell count (cols × rows)
- Masonry bin-packing into 1-4 columns, picks best coverage
- Projects screen positions into 3D world coordinates via frustum math
- Camera pulls back to accommodate all focused cards
- Cards at different Z depths based on their world size vs allocated screen fraction

### What the Screenshot Shows
- 4 main terminal cards (STEWARD 01, MARKTEXT-BROWSER, STEWARD 02, claude-proxy sessions)
- Right sidebar with thumbnail cards (smaller, stacked vertically)
- Cards have different sizes based on their terminal dimensions
- Different text sizes visible across cards — text unification is a real concern

### What We're Designing
The user wants named layout presets (2-up, 3-up, 2-column, 3-column, etc.) that:
1. Define 2D screen-space arrangements (like CSS grid/flexbox layouts)
2. Position cards into those arrangements
3. Mutate card size (x,y) and/or terminal size (cols/rows) as needed
4. Optionally unify text size across all cards in a layout (requires calculating whether font sizes as-seen-by-user match, and adjusting card or terminal dimensions to achieve uniformity)

### Key Insight — Two Layers of Mutation
A layout isn't just "put card here at this screen position." It may need to:
- **Mutate card (x,y):** change the physical DOM size to fit the allocated rectangle
- **Mutate terminal (cols/rows):** resize the tmux session to match the new card shape
- **Unify text:** ensure all cards show text at the same apparent size to the user, which means calculating the effective font-zoom per card (function of card size, terminal size, and Z depth) and adjusting until they match

### Confirmed: Layout System Operates at Multiple Levels
Per user clarification (2026-04-01):
- **Position-only layouts:** frustum + Z-depth placement only, no mutations
- **Mutating layouts:** also resize card (x,y) and/or terminal (cols/rows) to achieve desired visual result
- **Usable space:** viewport minus 3D UI elements (sidebar, menus, status bar). Cards fill remaining space, NOT centered to full browser. UI elements are themselves 3D objects that constrain the available viewport.
- The system works in 3D, not just 2D screen-space projection

### Confirmed: Usable Space = Simple Rectangle Subtraction (For Now)
Per user (2026-04-01): Keep UI elements (sidebar, status bar) as CSS fixed overlays. Usable space = viewport minus their pixel footprint. This design is intentionally flexible — if we later want to promote thumbnails into 3D frustum-positioned cards, we just shrink the usable-space rectangle and place them via the same frustum projection as any other card. The layout system should not assume UI elements are permanently CSS overlays — it should work with either approach.

### Confirmed: Independent Composable Mutation Operations
Per user (2026-04-01): Layout mutations are independent operations, not a sequential pipeline. Any can be applied in any order, any combination:
- **Position into layout** — place cards in layout slots via frustum projection
- **Maximize card→layout** — resize card (x,y) to fill its allocated layout slot
- **Maximize terminal→card** — resize terminal (cols/rows) to fill its card
- **Font-size mutation** — change SVG cell size to fit more cols/rows in same card at smaller font
- **Unify text size** — equalize apparent character size in screen pixels across all cards
Examples: position → unify text. Or: position → maximize card → unify text. Or: position → maximize terminal. No operation requires another as a prerequisite. Text unification target: identical apparent character size across all cards.

### Confirmed: Font Size as a Layout Lever
Per user (2026-04-01): "Usable" is about apparent font size as seen by user, not just cols/rows. If a layout slot is too small for 80x24 at current font, shrinking the font could exceed minimum constraints for both col and row — making the layout viable. Need a metric for minimum usable apparent font size (screen pixels per character as seen by user). This is a tunable threshold, exact value TBD from usage data. Font-size mutation sits between card mutation and terminal mutation in cost — it's per-browser (doesn't touch tmux) but changes readability.

### Confirmed: Mutation Cost Hierarchy
Per user (2026-04-01): Operations should prefer cheapest mutation first:
1. **3D relocation** (Z-depth / frustum positioning) — free, no state change, affects no one
2. **Card mutation** (x,y DOM resize) — per-browser only, saved in profile JSON
3. **Font-size mutation** (SVG cell size) — per-browser, doesn't touch tmux, but changes readability
4. **Terminal mutation** (cols/rows tmux resize) — global, propagates to all connected browsers
Default to relocation if it achieves the desired effect. Only escalate when it can't.

### Edge Case: Card Mutation and Co-Browsers
Per user (2026-04-01): Card-only mutations (x,y without terminal cols/rows change) don't trigger `updateCardForNewSize` on co-browsers — their cards stay as-is. But:
- The 3D space is now divergent between viewers (already the case with per-browser profiles)
- If a co-browser's layout system auto-repositions on card-size changes, it could trigger a reposition event that mutates their card without them seeing a visible change — cards shift for no apparent reason
Both are true. This is documented in case we need to address reposition behavior for co-browsers. May need a "reposition reason" flag or suppression logic.

### Design Principle: Be Polite to Co-Browsers
Per user (2026-04-01): When one user's layout triggers a terminal mutation that propagates to co-browsers, the co-browser should absorb the change gracefully:
1. Accept the new terminal size (cols/rows) — shared tmux reality, unavoidable
2. Recalculate card size to match (calcCardSize)
3. Compensate via Z-depth repositioning to keep the card's apparent screen size/position stable
4. Do NOT rerun the co-browser's layout — let the frustum absorb the change
The co-browser sees content reflow (terminal resized) but their layout doesn't jump. This is a proposal — needs verification that Z-compensation alone can absorb arbitrary terminal size changes without visible disruption.

### Confirmed: Multi-User Context
Per user (2026-04-01): Multiple users on different browsers view the same cards. Terminal (tmux) is shared — if one user mutates terminal cols/rows, `updateCardForNewSize` fires on all connected browsers and overwrites their `baseCardW/baseCardH`. This is the origin of that function. Layout mutations that touch terminal size (cols/rows) will propagate to all viewers. Card-only mutations (x,y) are per-browser (stored in profile JSON).

### Confirmed: VT100 as Minimum Usable Terminal
Per user (2026-04-01): 
- VT100 (80x24) is the reasonable minimum terminal size
- At minimum size, card aspect is ~1.44:1 (close to 4:3 landscape)
- Small terminals don't work in portrait — VT100 is near minimum usable height AND width
- As terminal grows (more total cells = rows*cols), aspect can expand: landscape → ultra-wide, or landscape → portrait
- Ultra-wide stacked rows is a valid special case but not the default
- Layouts must respect this: don't force a slot shape that would require a terminal below VT100 dimensions to be usable
- Card aspect ratio constrained between 16:9 (widest landscape, ~1.78:1) and 9:16 (tallest portrait, ~0.56:1) — universal, no exceptions. Even in N-stacked-rows, cards should be narrower at a comfortable aspect rather than letterboxed to fill the full slot width. Cards center within their slot if the aspect constraint prevents filling it. Usability beats space optimization.

### Future: Smart Scaling (Out of Scope)
Per user (2026-04-01): Once building blocks are built, add a "smart scaling" one-button feature that analyzes the current group and makes a reasonable layout + mutation decision automatically. User tunes by toggling individual layer options. This is why operations must be independent and composable — smart scaling orchestrates them. Not in scope for this design.

### Open Questions
- Are layouts static (fixed positions) or responsive (adapt to viewport resize)?
- Do layouts persist across sessions (saved to profile JSON)?
- When switching layouts, do we animate the transition?
- What happens to cards that don't fit the layout? (e.g., 5 cards in a 4-up layout)
- How is "usable space" calculated? Is it derived from the 3D UI element positions, or is it a simpler rectangle subtraction like the current SIDEBAR_WIDTH / STATUS_BAR_H approach?
