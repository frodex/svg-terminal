## Tier 2 overview visibility

### v0.1 stated

`tier2CardShouldShow(name)` uses `focusedSessions.size === 0 || focusedSessions.has(name)` — always show all cards in overview.

### Actual implementation

```javascript
function tier2CardShouldShow(sessionName) {
  if (perfTier < 2) return true;
  if (focusedSessions.has(sessionName)) return true;
  if (focusedSessions.size > 0) return false;
  var coarse = typeof matchMedia !== 'undefined' && matchMedia('(pointer: coarse)').matches;
  return !coarse || window.innerWidth <= window.innerHeight;
}
```

In overview (no focus), cards are shown on desktop or portrait touch devices. On coarse + landscape, all overview cards are hidden. This is a deliberate trade-off: the mobile compositor cannot handle the full CSS3D ring in landscape, and showing cards it cannot render produces frozen/glitched visuals worse than an empty ring. Sidebar thumbnails remain functional for navigation.

## Proactive helpers

### v0.1 stated

Named functions `maybeProactiveAutoTier1()`, `maybeProactiveAutoTier2()`, `perfCoarsePointer()`, `perfApproxPixelLoad()` called from `init`, `onResize`, `addTerminal`.

### Actual implementation

None of those functions exist. Their intent is absorbed into two places:

1. **`init`**: if `_cachedGPU.isSoftware`, applies tier 1 immediately. Otherwise starts at tier 0.
2. **`onResize`**: runs an 8-frame compositor test (see below) that cascades degradation as needed.

Gap: `addTerminal` does not re-evaluate tier. New terminals increase compositor load but nothing triggers re-assessment until the next resize or until the background FPS sampler re-arms. This is a known gap (tracked in sessions.md Pending Items).

## onResize — compositor-safe resize + 8-frame test

### v0.1 stated

`onResize` updates camera/renderer, optionally resets FPS sampling, runs lightweight proactive tier hooks.

### Actual implementation

`onResize` is the primary performance adaptation mechanism. The architecture:

1. **Hide all cards** (`css3dObject.visible = false`) BEFORE `renderer.setSize()` — prevents the compositor from choking on CSS3D transforms at the new viewport size.
2. **Resize renderer**, render empty scene.
3. **Manual mode**: restore visibility per current tier, return.
4. **Auto mode**: reset to tier 0 (restores RENDER_SCALE, shadows, ring spin), show all cards, render.
5. **8-frame rAF test**: measure `requestAnimationFrame` inter-frame delays over 8 frames. Take the 90th percentile worst frame time.
6. **Threshold**: 20ms for coarse pointer, 35ms for fine pointer.
7. **Cascading degradation**: if p90 > threshold, apply tier 1, re-render, re-test. If still over, apply tier 2.
8. **Re-up**: because step 4 resets to tier 0, conditions that improve (e.g., rotate back to portrait) allow re-promotion without page reload.

The hide-before-resize step is the critical architectural insight. CSS3DRenderer compositing is asynchronous — the browser compositor thread handles the actual painting of CSS-transformed DOM elements independently of the JS thread. `requestAnimationFrame` timing measures JS thread performance, not compositor stress. Hiding cards eliminates compositor work during the layout change, then the 8-frame test measures compositor performance at the new viewport size with cards visible.

## Per-frame syncPerfTier2Visibility

### v0.1 stated

`syncPerfTier2Visibility()` called each frame in `animate` so tier 2 visibility cannot drift.

### Actual implementation

`syncPerfTier2Visibility` is called from event paths only: `applyPerfTier`, `onResize` (manual branch), `focusTerminal`, `unfocusTerminal`. Not called per-frame. This avoids evaluating `matchMedia` + viewport dimensions 60 times per second. Correctness relies on all mutation points calling the function — the current call sites cover focus/unfocus, tier changes, and resize, which are the known mutation events.

## RENDER_SCALE pattern

### v0.1 stated

`_savedRenderScale` saves the original scale when lowering for tier ≥ 1.

### Actual implementation

Uses `RENDER_SCALE_DEFAULT` constant (value: 2). `applyPerfTier` checks `RENDER_SCALE !== RENDER_SCALE_DEFAULT` to decide whether to restore. No mutable saved state. `resizeRenderer()` helper centralizes `renderer.setSize` + `renderer.domElement.style.transform` logic, called by `applyPerfTier`, `init`, and `onResize`.

## Background FPS sampler

Both v0.1 and current code use a background FPS sampler in `animate`. Current implementation:

- 3-second sampling window (`_perfCheckPhase 0→1→2`)
- Skips first 33% of frame times (warm-up)
- Coarse-aware thresholds: avg > 42ms (coarse) or > 50ms (fine) → tier 2; avg > 33ms → tier 1
- Fires once per page load, stops at `_perfCheckPhase = 2`
- Re-armed on resize (`_perfCheckPhase` reset to 0)
- Not re-armed on `addTerminal` (gap noted above)

## Thumbnail data path (new — not in v0.1)

`routeDashboardMessage` populates `t.screenLines` directly from incoming WebSocket `screen`/`delta` messages. This decouples thumbnail content from the SVG `<object>` load state. When `CSS3DObject.visible = false` sets `display: none` on the card DOM, the embedded SVG `<object>` stops loading — the old `_screenCallback` path would never fire, leaving thumbnails empty. The WebSocket path ensures thumbnails always have content regardless of card visibility.

---

## Compositor repaint helpers (new — not in v0.1)

`scheduleTerminalSurfaceRepaint`, `bumpTranslateZ`, outline nudge, and `routeEmbedMessageToSvg` guards address CSS3D / `<object>` compositing issues and delta-before-screen race conditions. These are a separate track from performance tiers but interact with the visibility system.

## Test file status

### v0.1 stated

`test-performance-mode.mjs` Test 3 updated to expect zero hidden cards in overview.

### Actual test file (before this session's fix)

Test 3 expected `hiddenCount === terminalCount` (all hidden) or `hiddenCount > 0`. On Puppeteer (non-coarse pointer), `tier2CardShouldShow` returns `true` for all overview cards, so `hiddenCount` would be 0 — both assertions fail. The test was never updated to match either the v0.1 proposal or the current code.

### Fix applied

Test 3 now expects `hiddenCount === 0` on non-coarse Puppeteer (all overview cards visible at tier 2 on desktop). This matches the actual `tier2CardShouldShow` behavior: `!coarse` → `true` → all cards shown.

## Summary of what survived from v0.1 and what changed

**Survived in spirit:**
- Reduced-tier animation cost (`lowMotion` / `perfTier >= 1` skips float, drift, shadow, specular)
- Coarse vs fine FPS threshold split for tier 2
- Sidebar layer promotion (`translateZ(0)`, `isolation: isolate`)
- Central tier 2 visibility helper (`syncPerfTier2Visibility`)
- `unfocusTerminal` delegates to `syncPerfTier2Visibility` instead of hand-written loop

**Changed in substance:**
- Tier 2 overview: conditional on coarse+landscape, not unconditional "show all"
- Proactive helpers: replaced by compositor-safe resize + 8-frame rAF test
- onResize: hide→resize→test architecture instead of lightweight hooks
- animate: no per-frame tier 2 sync (event-driven instead)
- Re-up capability: resize resets to tier 0 and re-tests, allowing promotion when conditions improve
- Render scale: `RENDER_SCALE_DEFAULT` constant instead of `_savedRenderScale` mutable

**Unique to current implementation (not in v0.1):**
- Compositor-safe hide-before-resize
- 8-frame cascading rAF test
- Thumbnail data from WebSocket (not SVG load)
- Re-promotion on improved conditions
