## Problem Statement

The svg-terminal dashboard displays multiple terminal sessions simultaneously. Each session generates screen data that must reach the browser. The current pipeline has no fairness guarantees, no priority system, and potential starvation scenarios. Corrupted card visuals have been observed that may stem from pipeline timing issues.

---

## Current Architecture

### Two data paths exist

```
PATH A: Claude-proxy sessions (cp-* sessions)
  tmux PTY → claude-proxy process → Unix socket → svg-terminal → WebSocket → browser

PATH B: Local tmux sessions (direct)
  tmux → execFile('capture-pane') child process → svg-terminal → WebSocket → browser
```

### Path A: Claude-proxy pipeline (single socket, multiplexed)

```
┌─────────────┐     ┌────────────────┐     ┌─────────────────┐     ┌─────────┐
│ tmux sess 1 │────▶│                │     │                 │     │         │
│ tmux sess 2 │────▶│  claude-proxy  │────▶│  svg-terminal   │────▶│ browser │
│ tmux sess 3 │────▶│  (multiplexer) │     │  (cpOnDataChunk)│     │  (WS)   │
│ ...         │────▶│                │     │                 │     │         │
│ tmux sess N │────▶│                │     │                 │     │         │
└─────────────┘     └────────────────┘     └─────────────────┘     └─────────┘
                           │                        │
                     ONE Unix socket          ONE event handler
                     ALL sessions             processes sequentially
                     multiplexed              in arrival order
```

**How data flows:**

1. Each tmux session has a PTY managed by claude-proxy
2. PTY fires callbacks when screen changes: `pty.onScreenChange(startLine, endLine)`, `pty.onCursorChange()`, `pty.onTitleChange()`
3. Callbacks set `dirty = true` and record which lines changed in `dirtyLines` Set — they do NOT send data
4. Claude-proxy's `TerminalMirror` class polls each session at 30ms intervals (`setInterval(() => {...}, 30)`)
5. On each poll: if `dirty === false`, skip (no work). If dirty, read ONLY the changed lines from the PTY buffer, build ONE coalesced delta, emit it
6. This is **natural coalescing** — if 10 screen changes happen in 30ms, ONE delta message is sent containing all dirty lines
7. Delta is serialized to JSON + newline delimiter and written to Unix socket
8. All sessions share ONE Unix socket — messages interleaved
9. svg-terminal's `cpOnDataChunk` receives raw bytes, splits on `\n`, parses JSON
10. Routes to per-session handler → broadcasts to subscribed WebSocket clients

**CORRECTION:** Previous version stated "Path A sends EVERY diff as a separate message. No coalescing." This is WRONG. Claude-proxy coalesces at 30ms per session via the dirty-flag + poll pattern. At most ~33 messages/second per session.

**Timing:**

| Stage | What happens | Duration |
|-------|-------------|----------|
| PTY callback | Sets dirty flag + records line numbers | <0.1ms (no data sent) |
| TerminalMirror 30ms poll | Checks dirty flag, reads changed lines from PTY buffer | ~1-5ms per session |
| JSON serialize + socket write | One coalesced delta per session per 30ms cycle | <1ms |
| OS kernel buffer | Holds bytes until svg-terminal reads | 128-256KB capacity |
| libuv read | Reads up to 64KB per `read()` syscall | <1ms |
| `cpOnDataChunk` | Parse JSON, route to handler | <1ms per message |
| Handler broadcast | `ws.send(json)` to each subscriber | <1ms per client |

**Maximum message rate:** 20 sessions × 33 messages/sec = 660 messages/sec on the socket. Each message is a coalesced delta (only changed lines), not a full screen.

### Path B: Local tmux pipeline (per-session child processes)

```
┌─────────────┐     ┌─────────────────┐     ┌──────────────────┐     ┌─────────┐
│ tmux sess 1 │     │ execFile pool   │     │                  │     │         │
│             │◀───▶│ (4 threads)     │────▶│  svg-terminal    │────▶│ browser │
│ tmux sess 2 │     │                 │     │  (captureAndBroadcast)│ │  (WS)   │
│             │◀───▶│ thread 1: cap A │     │                  │     │         │
│ tmux sess 3 │     │ thread 2: cap B │     │                  │     │         │
│             │◀───▶│ thread 3: cap C │     │                  │     │         │
│ ...         │     │ thread 4: cap D │     │                  │     │         │
│ tmux sess N │     │ queue: E,F,...N │     │                  │     │         │
└─────────────┘     └─────────────────┘     └──────────────────┘     └─────────┘
                           │
                    libuv thread pool
                    default: 4 threads
                    sessions > 4 wait in queue
```

**How data flows:**

1. Each session has a `SessionWatcher` with a `setInterval` timer
2. Timer fires → calls `captureAndBroadcast()`
3. `captureAndBroadcast()` calls `capturePane()` which runs `execFile('tmux', ['capture-pane', ...])`
4. `execFile` uses libuv's thread pool (default 4 threads)
5. Child process runs, captures tmux screen, returns stdout
6. Callback fires on event loop with captured data
7. `diffState()` compares to last state → if changed, broadcasts to subscribers

**Timing:**

| Stage | What happens | Duration |
|-------|-------------|----------|
| setInterval timer fires | Queues `captureAndBroadcast` on event loop | 0ms (scheduling) |
| `execFile('tmux', ...)` | Enters libuv thread pool queue | 0ms if thread available, waits if all 4 busy |
| tmux capture-pane | Child process runs, captures screen | ~5-15ms |
| Child stdout → callback | Data enters event loop | <1ms |
| `diffState()` | Compare old vs new screen | <1ms |
| `ws.send(json)` | Broadcast diff to subscribers | <1ms per client |

**Polling intervals:**

| State | Interval | Why |
|-------|----------|-----|
| Focused session | 20ms | Low latency for active terminal |
| Unfocused session | 500ms | Reduce load for background sessions |
| After input | 5ms (one-shot via `setTimeout`) | Immediate feedback after keystroke |

## The Problems

### Problem 1: Thread pool starvation (Path B — local tmux)

**4 threads, 20 sessions.** When all 20 timers fire (they drift to similar phase over time), 4 sessions capture simultaneously, 16 wait in the libuv thread pool queue. FIFO order — no fairness guarantee.

**Scenario:**
```
Timer fires for sessions A through T (20 sessions)
Thread pool: [A, B, C, D] running, [E, F, G, H, I, J, K, L, M, N, O, P, Q, R, S, T] queued
Session A has large scrollback → capture takes 50ms
Sessions B, C, D finish in 5ms → threads free
Threads pick up E, F, G from queue
Meanwhile A is STILL running → holding a thread for 50ms
Sessions H-T waiting, getting stale
```

**Impact:** Sessions later in the queue get stale data. The user sees different sessions updating at different rates.

**What guarantees we aren't going back to the same socket over and over?** Nothing in the current design. If session A's `setInterval` fires again before session T's first capture completes, A re-enters the queue. With 20ms focused interval and 50ms capture time, A could queue 2-3 times before T gets its first turn.

### Problem 2: Single-pipe multiplexing bottleneck (Path A — claude-proxy)

**One Unix socket, all sessions.** Claude-proxy writes messages for all sessions onto one pipe. svg-terminal reads 64KB at a time, processes all messages in each chunk sequentially.

**Scenario:**
```
Claude-proxy writes 20 screen diffs (one per session) to socket
svg-terminal's cpOnDataChunk receives a 64KB chunk containing ~10 messages
Processes all 10 in a synchronous loop:
  - Parse JSON (~0.1ms)
  - Find handler (~0.01ms)
  - ws.send to each subscriber (~0.1ms per client)
  Total: ~2ms for 10 messages
Meanwhile, another 10 messages arrived in the kernel buffer
Next data event fires → processes those 10
```

**In normal operation this is fine.** 2ms for 10 messages = 5000 messages/second capacity. 20 sessions at 50 updates/second = 1000/second. Well within capacity.

**Under abuse/flood:** An attacker with access to claude-proxy's socket could write continuously. svg-terminal's event loop would be dominated by `cpOnDataChunk`, starving WebSocket message handling (browser input) and timer callbacks (local session captures).

### Problem 3: No message priority

**Control messages wait behind screen data.** A `session-end` event from claude-proxy enters the same pipe as screen diffs. If 50 screen diffs are ahead of it in the buffer, all 50 process before the kill signal.

**Impact:**
- Dead session renders 50 more frames before card disappears
- Permission change (viewOnly) allows input for 50 more frames before blocking
- Session settings change is delayed by screen data volume

**Current flow:**
```
chunk: [diff-A, diff-B, diff-A, session-end-C, diff-A, settings-changed-D]
                                      ↑
                        processes 3 diffs before seeing this
```

**Desired flow:**
```
chunk: [diff-A, diff-B, diff-A, session-end-C, diff-A, settings-changed-D]
                                      ↓
Step 1: Parse all messages
Step 2: Process control first: session-end-C, settings-changed-D  
Step 3: Process screen diffs: diff-A (latest only), diff-B
Step 4: Skip diff for C (dead), coalesce A's 3 diffs to 1
```

### Problem 4: ~~No staleness coalescing~~ CORRECTED — coalescing exists at source

**CORRECTION:** Claude-proxy's TerminalMirror already coalesces at 30ms intervals. Multiple screen changes within one 30ms window produce ONE delta message. This problem was overstated.

**Remaining concern:** If svg-terminal's event loop is blocked (e.g., processing a large batch of messages), multiple 30ms deltas from the same session could queue up in the kernel buffer. When they're processed, intermediate deltas are still wasted work. But this is a minor concern at 33 messages/sec/session — not a design flaw.

### Problem 5: Chunk boundaries vs message boundaries

**64KB chunks don't align with JSON messages.** A message could be split across two chunks. The `cpBuf` newline-splitting handles this correctly for complete/incomplete messages, BUT:

```
Chunk 1 (64KB): ...valid messages...\n{"event":"terminal","sessionId":"cp-A","data":{"type":"del
Chunk 2 (64KB): ta","lines":[...]}}\n...more messages...\n
```

- Chunk 1: everything before the last `\n` processes. The partial `{"event":"terminal"...` stays in `cpBuf`.
- Chunk 2: `cpBuf` + chunk 2 data. The partial message completes. Processing continues.
- This is CORRECT — no data loss.

**Potential corruption scenario:** If a message contains a literal `\n` inside a JSON string value (e.g., terminal output with newlines), the `split('\n')` would incorrectly split it mid-message. Claude-proxy MUST ensure JSON is serialized without literal newlines (using `\n` escape instead). **Verify this assumption.**

## Event-driven vs polling comparison

| Aspect | Path A (claude-proxy) | Path B (local tmux) |
|--------|----------------------|---------------------|
| Trigger | Event-driven (data arrives on socket) | Polling (setInterval timer) |
| Fairness | None — arrival order | None — thread pool FIFO |
| Latency | Low — event fires immediately | Bounded by poll interval (20-500ms) |
| Starvation | Possible if one session floods | Possible if thread pool exhausted |
| Resource | 1 socket, no threads | 4 threads (UV_THREADPOOL_SIZE) |
| Coalescing | None | Implicit — poll captures latest state |
| Control priority | None — inline with data | N/A — control via separate events |

### Key insight: BOTH paths have natural coalescing

**Path B (local tmux):** Captures always get the LATEST screen state. If 5 updates happen between polls, the capture returns the final state — the intermediate 4 are never seen.

**Path A (claude-proxy):** TerminalMirror's 30ms poll collects ALL dirty lines since last poll into ONE delta message. If 10 screen changes happen in 30ms, one delta with all changed lines is sent. Coalescing happens at the source.

**CORRECTION:** Previous version stated "Path A sends EVERY diff as a separate message. No coalescing." This was WRONG — verified by reading `TerminalMirror.start()` in `claude-proxy/src/socket-server.ts` lines 69-172.

## Redesign Options

### Option 1: Per-session message queues with priority drain

```
cpOnDataChunk → parse all messages → route to per-session queues
                                   → route control messages to priority queue

Processing cycle (on setImmediate or timer):
  1. Drain priority queue (all control messages)
  2. Round-robin: take ONE message per session from each queue
  3. For screen diffs: only process LATEST, discard intermediates
  4. Broadcast results
  5. If any queues still have messages, schedule next cycle via setImmediate
```

**Pros:**
- Fair: every session gets attention each cycle
- Priority: control messages always first
- Coalesced: intermediate diffs skipped
- Non-blocking: setImmediate yields between cycles

**Cons:**
- Added latency (queuing + cycle time vs immediate processing)
- More memory (per-session queues)
- Complexity

### Option 2: Coalesce-on-write (claude-proxy side)

Have claude-proxy coalesce diffs BEFORE writing to the socket. If session A has 3 pending diffs, combine them into one message before writing.

```
Claude-proxy:
  Per-session write buffer
  On dirty: queue diff
  On flush timer (e.g., 16ms): send latest state per session, discard intermediates
  Control messages: write immediately (bypass flush timer)
```

**Pros:**
- Fixes the problem at the source
- svg-terminal stays simple
- Less data on the socket
- Control messages get natural priority (immediate write)

**Cons:**
- Adds latency (flush timer)
- More complexity in claude-proxy
- Doesn't help Path B (local tmux — but Path B already coalesces)

### Option 3: Increase thread pool + prioritize (Path B only)

```
UV_THREADPOOL_SIZE=20  (or match session count)
```

Eliminates thread pool starvation for local tmux sessions. Each session gets its own thread for captures.

**Pros:**
- Simple — one env var change
- No code changes for Path B fairness

**Cons:**
- 20 threads × 20 sessions = 400 concurrent child processes possible
- OS resource pressure
- Doesn't fix Path A issues
- Doesn't scale — more sessions = more threads

### Option 4: Replace execFile with tmux control mode (Path B)

Instead of spawning a child process per capture, use tmux's `-C` control mode which opens a persistent connection to tmux and receives events.

**Pros:**
- No child process spawning
- Event-driven like Path A
- Lower latency

**Cons:**
- Different output format — needs parser
- Previously investigated and rejected (see sessions.md: "Rejected tmux -C control mode (adds persistent process, throws away data, calls capture-pane anyway)")
- May not actually be better — needs re-evaluation

### Option 5: Move all sessions through claude-proxy (eliminate Path B)

All sessions — including local tmux — go through claude-proxy. One data path, one pipeline.

**Pros:**
- One architecture to optimize
- Claude-proxy already handles multiplexing
- Permission model consistent for all sessions
- Coalescing (Option 2) covers everything

**Cons:**
- Claude-proxy becomes a hard dependency (can't run svg-terminal standalone)
- More load on claude-proxy
- All the Path A problems still apply (just consolidated)

## Open questions

2. **Does claude-proxy ensure no literal `\n` in JSON messages?** If `JSON.stringify` is used (standard), literal newlines in terminal output become `\\n` in JSON — safe. But verify.

3. **What is claude-proxy's TerminalMirror poll rate?** This determines the maximum message rate on the Unix socket.

4. **Can we measure actual thread pool contention?** Add timing to `captureAndBroadcast` to see queue wait time vs capture time.

5. **Is Option 2 (coalesce at source) sufficient, or do we need Option 1 (per-session queues in svg-terminal) as well?**

6. **Should we move to Option 5 (everything through claude-proxy) as a long-term architecture?** This simplifies svg-terminal significantly but makes claude-proxy a single point of failure.

## Design Requirements (from user)

Priority order:

1. **Data integrity:** Screen diffs must not be mutated, clipped, dropped, or severed in ways that break terminal rendering. If a diff is partial or corrupt, it must be detected and recovered (request full screen), not rendered as garbage.

2. **Buffer handling:** Buffers must be emptied in a predictable, well-understood pattern. No scenarios where buffers overflow silently, data is lost, or stale data is served as current.

3. **DOS/abuse resistance:** The pipeline must be hardy against flooding, malformed data, and resource exhaustion. A misbehaving session or attacker should not be able to degrade other sessions.

4. **Control message interweaving:** Control signals (session-end, permission changes, user controls, authorization changes, timing signals) must be interleaved with terminal data in a way that is:
   - Optimized for user experience (control takes effect immediately, not after N screen frames)
   - Fast and responsive
   - Not prone to race conditions between control and data

5. **Reconnection hardiness:** The pipeline must recover cleanly from disconnects — WebSocket drops, claude-proxy restarts, tmux session loss. No stale state, no ghost cards, no blank terminals after reconnect.

6. **Not prone to corrupt data:** Under no circumstances should one session's data appear in another session's card, or a partial diff be applied to the wrong base state.

## Recommendation (preliminary)

Needs more research before recommending. The coalescing at source (already exists) and event-driven control signals (just added) cover some requirements. But data integrity verification, abuse resistance, and reconnection hardiness need deeper analysis of:

- What exactly causes the corrupted card visuals observed?
- Can a partial JSON message on the Unix socket corrupt subsequent messages?
- What happens when a WS client can't keep up with the broadcast rate?
- What happens when claude-proxy restarts mid-delta?

Next session: reproduce the card corruption, trace the exact data path that causes it, then design the fix.
