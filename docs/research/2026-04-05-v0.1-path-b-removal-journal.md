## Context

svg-terminal has two data paths:
- **Path A:** svg-terminal → claude-proxy (Unix socket JSON-RPC) → tmux PTY (event-driven, 30ms coalescing)
- **Path B:** svg-terminal → tmux directly via `execFile('tmux', ...)` (child process spawning, libuv thread pool)

Path B was supposed to be removed in the legacy cleanup but the previous agent got confused and preserved it, treating it as essential. It isn't. All production sessions use Path A. Path B bypasses the permission model and its `execFile` architecture causes chunky input (every keystroke and every screen capture spawns a child process through a 4-thread pool).

The TODO at `docs/TODO-path-b-removal.md` was committed identifying this but asking "is claude-proxy a hard dependency?" — a question that was already answered by the architecture. Claude-proxy has been a hard dependency since the WebSocket consolidation (2026-03-31). The auth model, permission model, session lifecycle, and all production sessions depend on it.

## Code blocks to remove

### 1. `tmuxAsync` function (line 24-31)
The helper itself. Zero callers after Path B removal.

### 2. `capturePane()` function (lines 396-431)
Spawns `tmux display-message` + `capture-pane` child process. Used by:
- `captureAndBroadcast()` in SessionWatcher (line 675)
- `get-screen` WS handler local branch (line 1200)

### 3. `capturePaneAt()` function (lines 433-466)
Spawns `tmux display-message` + `capture-pane` with scroll offset. Used by:
- `captureAndBroadcast()` in SessionWatcher (line 673)

### 4. `parseLine()` / SGR parsing (wherever defined)
Used only by `capturePane` / `capturePaneAt` to parse tmux SGR output. Claude-proxy does its own parsing.

### 5. `diffState()` function (lines 583-600)
Compares previous and current capture state. Used only by `captureAndBroadcast()`.

### 6. `getOrCreateWatcher()` / SessionWatcher poll loop (lines 654-702)
The entire poll-based capture system. For cp sessions, `bridgeClaudeProxySession` replaces this with event-driven data from TerminalMirror. After Path B removal, all sessions are cp sessions, so the poll loop is dead.

**IMPORTANT:** The `sessionWatchers` Map and `subscribers` pattern is still used by `bridgeClaudeProxySession` — it creates watcher objects to hold subscriber sets. This must be preserved but can be simplified (remove timer, captureAndBroadcast, lastState).

### 7. `subscribeToSession()` function (line ~705)
Adds a WS to a watcher's subscriber set AND starts the poll timer. After removal, subscription for cp sessions goes through `bridgeClaudeProxySession` which already handles this.

### 8. `triggerCapture()` function (lines 738-746)
Manually triggers a capture poll. Used after input (line 1399) and resize (line 1361). After removal, not needed — cp sessions get screen updates via TerminalMirror events.

### 9. `getScrollOffset()` / `setScrollOffset()` / `paneScrollOffsets` (lines 573-581)
Scroll state for local tmux sessions. Claude-proxy handles scroll via its own `handleScroll` method.

### 10. Local tmux branch in `sendSessionDiscovery()` (lines 895-925)
Calls `tmux list-sessions` to find local sessions. Remove — cpRequest('listSessions') is the sole source.

### 11. Local tmux fallback in `create-session` handler (lines 1101-1106)
Falls back to `tmux new-session` if cpRequest fails for shell profile. Remove — error should propagate.

### 12. Local branch in `get-screen` handler (lines 1196-1200)
Checks `has-session` to detect local, then calls `capturePane`. Remove — always use cpRequest('getSessionScreen').

### 13. Local tmux in `get-sessions` handler (lines 1226-1236)
Lists local tmux sessions alongside cp sessions. Remove.

### 14. Local tmux input handler (lines 1363-1399)
Handles input/resize/scroll for local sessions via tmux send-keys. Remove — all input goes through cpRequest.

### 15. Local tmux resize handler (lines 1349-1360)
Resize lock + tmux resize-window. Remove.

### 16. `subscribe` handler's local branch (line 1077-1079)
Calls `subscribeToSession(ws, session, '0')` for non-cp sessions. Remove — all sessions are cp.

## Risks and concerns

### 1. Sessions created outside claude-proxy
If someone creates a tmux session directly (`tmux new-session -s foo`), it won't appear in the dashboard. This is a feature, not a bug — unmanaged sessions bypass the permission model.

### 2. Claude-proxy down = no sessions
If claude-proxy crashes, svg-terminal can't show any sessions. This is already the case for all production sessions. The reconnection overlay handles this UX.

### 3. `execFile` import
After removing all tmuxAsync callers, the `execFile` import can be removed (verify no other callers).

### 4. SGR parser
`parseLine()` / `color-table.mjs` / `sgr-parser.mjs` — used only by `capturePane`. After removal, these become dead code in server.mjs. They may still be used by terminal.svg (check). The server imports should be removed.

### 5. Test files
`test-server.mjs` likely tests local tmux paths. Needs audit.

## Open questions

1. **Should `terminal.svg` standalone viewer be removed too?** It uses `startPolling(150)` which calls `GET /api/pane` (already removed). It's already broken. Removing it is cleanup, not a decision.

2. **Should the `resizeLocks` Map be preserved?** It prevents resize thrashing from multiple browsers. Claude-proxy may have its own rate limiting on resize. Check.

3. **The 5ms `triggerCapture` after input — does claude-proxy need an equivalent?** No — TerminalMirror's 30ms poll catches changes from input. The 5ms trigger was a latency optimization for Path B's slower poll. Path A doesn't need it.
