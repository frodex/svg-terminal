## Problem Statement

Security audit revealed that while svg-terminal has user-level authentication (OAuth + session cookies), it has no **per-session authorization**. Any authenticated user can:

1. Read any session's terminal content (`/api/pane?session=X`)
2. Send keystrokes to any session (`POST /api/input`, `/ws/terminal`)
3. Subscribe to screen updates for any session (`/ws/dashboard`)
4. See all sessions in the system (`/api/sessions`)

Additionally:
- `/auth/status?email=` leaks user registration status without any auth
- WebSocket connections to claude-proxy use a hardcoded `root` user identity regardless of who's actually authenticated
- The UI degrades silently on auth failure instead of redirecting to login

This is a multi-user system being deployed with OAuth (Google, Microsoft, GitHub). These gaps allow any authenticated user to read and control any other user's terminal sessions, including sessions running as root.

## Current Architecture

### Two-tier system

**svg-terminal (web UI layer):**
- Handles OAuth, session cookies, user database
- Serves the dashboard HTML/JS/CSS
- Proxies requests to claude-proxy via Unix socket JSON-RPC
- Passes `linux_user` from session cookie on every proxy call

**claude-proxy (session management layer):**
- Creates/destroys tmux sessions on custom sockets
- Stores session metadata including permission flags
- Enforces `allowedUsers`, `allowedGroups`, `public`, `hidden`, `viewOnly`
- Returns only permitted sessions on `listSessions` calls
- Socket-level Unix permissions (UGO) for direct tmux access

### What already exists in the UI forms

The session creation forms already have fields for:
- `public` checkbox (default: checked)
- `hidden` checkbox
- `viewOnly` checkbox
- `allowedUsers` multi-select (populated from claude-proxy)
- `allowedGroups` multi-select (populated from claude-proxy)
- `password` field

These fields are sent to claude-proxy on create/restart/fork. Claude-proxy enforces them on its side. But svg-terminal does NOT enforce them — it trusts the user identity but doesn't check permissions before serving data or accepting input.

### The boundary problem

Claude-proxy filters `listSessions` by user. But svg-terminal has additional data paths that bypass this:

1. **Local tmux sessions** — svg-terminal discovers these directly via `tmux list-sessions`, no filtering
2. **`/api/pane`** — reads screen content directly from tmux, no authorization check
3. **`POST /api/input`** — writes to tmux directly, no authorization check
4. **`/ws/terminal`** — establishes per-card WebSocket, no session ownership check
5. **`/ws/dashboard` subscriptions** — subscribes to any session name, no check

For claude-proxy sessions, the unix socket RPC passes the user and claude-proxy enforces access. But svg-terminal doesn't check the response — if claude-proxy returns an error, it may not be handled correctly.

## Research: How should this be done?

### Approach A: Enforce at svg-terminal layer (defense in depth)

Every endpoint in svg-terminal that touches session data checks authorization before processing.

**How it works:**
1. On every session-related request, extract `linux_user` from session cookie
2. Look up session metadata (from claude-proxy or local tmux)
3. Check: is this user the owner? Are they in `allowedUsers`? Is the session `public`?
4. Deny access if unauthorized

**For local tmux sessions:**
- Need a session metadata store in svg-terminal (SQLite or in-memory)
- When sessions are created, record owner + permissions
- Check this store on every access

**For claude-proxy sessions:**
- Already pass `user` on RPC calls — claude-proxy enforces
- But svg-terminal should also validate responses and not serve data if claude-proxy returns an error/denied

**Pros:**
- Defense in depth — even if claude-proxy has a bug, svg-terminal blocks unauthorized access
- Consistent enforcement regardless of session type (local or proxied)
- Can enforce on WebSocket subscription, not just HTTP

**Cons:**
- Duplicates logic that claude-proxy already has
- Need to keep permission state in sync between svg-terminal and claude-proxy
- More complex — two systems making authorization decisions

### Approach B: Delegate entirely to claude-proxy (single source of truth)

All session operations go through claude-proxy, which is the sole authorization authority. Svg-terminal is just a pass-through.

**How it works:**
1. Remove direct tmux access from svg-terminal
2. ALL sessions (even "local" ones) are managed by claude-proxy
3. Every data request is proxied to claude-proxy with the user identity
4. Claude-proxy returns data or error — svg-terminal forwards the result

**Pros:**
- Single source of truth for permissions
- No duplication of authorization logic
- Claude-proxy already has the UGO model
- Simpler svg-terminal code

**Cons:**
- Tight coupling — svg-terminal can't function without claude-proxy
- Performance overhead — every screen capture goes through RPC
- Local tmux sessions would need to be created through claude-proxy
- More latency on the data path

### Approach C: Hybrid — svg-terminal enforces, claude-proxy is authoritative

Svg-terminal maintains a **cached copy** of session permissions, refreshed from claude-proxy. Enforces locally for performance, but claude-proxy is the source of truth.

**How it works:**
1. On WebSocket connect / session discovery, svg-terminal fetches session permissions from claude-proxy
2. Caches them in memory (Map of session → { owner, allowedUsers, public, viewOnly, ... })
3. On every data/input request, checks the local cache
4. Cache refreshed periodically or on session-add/session-remove events
5. For local tmux sessions, svg-terminal IS the authority (stores permissions in SQLite)

**Pros:**
- Fast — no RPC on every keystroke or screen frame
- Defense in depth — svg-terminal blocks even if claude-proxy is bypassed
- Graceful degradation — if claude-proxy is down, cached permissions still enforce
- Clean separation — claude-proxy for session lifecycle, svg-terminal for web access control

**Cons:**
- Cache staleness — permissions change on claude-proxy but svg-terminal hasn't refreshed
- More state to manage
- Need to handle cache invalidation correctly

### Approach D: Token-scoped access (session tokens)

Instead of checking permissions on every request, issue session-specific tokens.

**How it works:**
1. When a user opens a session in the UI, svg-terminal requests a "session access token" from claude-proxy
2. Claude-proxy validates the user's permissions and returns a signed, time-limited token scoped to that session
3. The token is included in all subsequent WS/HTTP requests for that session
4. Server validates the token — no per-request permission lookup needed
5. Token expires after N minutes, client auto-refreshes

**Pros:**
- Extremely fast — token validation is just HMAC check, no database/RPC
- Stateless on the server side
- Natural revocation via expiry
- Works well with WebSocket (token in URL, no cookie needed)
- Already partially implemented (ws-token endpoint)

**Cons:**
- Tokens can be stolen (but same risk as session cookies)
- Revocation requires waiting for expiry (or maintaining a blacklist)
- Doesn't prevent session enumeration (user still needs to discover sessions first)
- More complex client-side token management

## Analysis: Which approach fits this system?

### Key constraints

1. **Cloudflare tunnel** — WebSocket cookies don't work through the tunnel. Tokens must be passed explicitly.
2. **SVG `<object>` sub-documents** — child documents don't share parent's session state. Must receive tokens via URL params.
3. **Performance** — 30ms screen capture polling. Authorization check on every frame is expensive if it requires RPC.
4. **Multi-session dashboard** — user views 10+ sessions simultaneously. Need efficient bulk authorization.
5. **Two session types** — local tmux (svg-terminal controls) and claude-proxy (delegated).

### Recommendation: Approach C+D hybrid

Combine cached permissions (Approach C) with session tokens (Approach D):

1. **Session discovery** — when dashboard loads, fetch permitted sessions from claude-proxy (with user identity). Cache the permission set.
2. **Session tokens** — for each permitted session, generate a short-lived HMAC token scoped to `(user, session, expiry)`. Pass to SVG `<object>` elements and WebSocket connections.
3. **Token validation** — on every WS frame and HTTP request, validate the session token. Fast HMAC check, no RPC.
4. **Cache refresh** — on session-add/session-remove SSE events, refresh the permission cache. Also periodic refresh (every 60s).
5. **Local tmux sessions** — svg-terminal is the authority. Store owner in session metadata. Default: owner-only unless explicitly made public.
6. **Fail-closed** — no token = no access. Expired token = redirect to re-authenticate.

## Open questions

1. **Should local tmux sessions be migrated to claude-proxy?** That would eliminate the "two sources of truth" problem. But it means claude-proxy must be running for any sessions to work.

2. **Token lifetime** — 5 minutes? 15 minutes? Shorter = more secure, longer = fewer refresh round-trips. Current ws-token is 5 min.

3. **Session password** — claude-proxy supports passwords on sessions. Should svg-terminal validate them, or delegate to claude-proxy? Currently the password field exists in the form but enforcement is on claude-proxy's side.

4. **View-only enforcement** — should svg-terminal block input on view-only sessions, or trust claude-proxy to reject it? Defense in depth says both.

5. **Root user sessions** — the admin (frodex310@gmail.com) maps to `root`. Root should see all sessions. How does this interact with session tokens? Maybe root gets a wildcard token or bypasses the check.

6. **Admin PIN for privileged actions** — resolved. PIN (4-20 chars, HMAC-SHA256 hashed) required for sensitive admin operations. 15-minute sudo window after entry. Lighter than full OAuth re-auth, doesn't navigate away from admin page. Applied to: flag changes, deactivate, purge, force re-login, merge, any action on root-mapped users.

7. **Rate limiting** — resolved. Tiered rate limits by real client IP (`cf-connecting-ip` for tunnel, `socket.remoteAddress` for direct):
   - Auth endpoints: 5/min, lockout after 10 failures
   - OAuth init: 10/min
   - Admin mutations: 10/min per user
   - Privileged actions: 3/min per user
   - Failed WS upgrades: 20/min per IP (prevents brute-force of API keys)
   - `/auth/api-key` has NO rate limit — already behind cookie auth, legitimate reconnect retries must not be throttled
   - Successful WS connections reset the failure counter

8. **API key duplication** — resolved. One active WS connection per API key. If a second connection attempts to use the same key, reject it. Max 10 active keys per user. 11th issuance revokes the oldest. Prevents stolen key abuse while allowing multiple browser tabs.

9. **Reconnection and re-auth distinction** — resolved:
   - Cookie valid + WS drops (network flake): transparent re-auth, fetch new API key, reconnect
   - Cookie valid + API key expired (idle/absolute): transparent, fetch new key
   - Duplicate key detected: revoke cookie + key, force full OAuth login
   - Admin force re-login: revoke all keys, send `reauth-required` WS message
   - Reconnection overlay shows "Retrying (attempt N)... countdown" then "Connection lost - Sign in again" link
   - No auto-redirect — user can see their cards (frosted) and click login when ready

10. **Dev mode hardening** — resolved. Dev mode now requires:
   - Explicit `AUTH_MODE=dev` env var (missing OAuth vars alone = server refuses to start)
   - `DEV_PASSWORD` env var (required when AUTH_MODE=dev)
   - Localhost-only by default (`DEV_LOCALHOST_ONLY=0` to opt out)
   - Deleting OAuth env vars no longer silently degrades to open access

## Additional findings: CP_DEFAULT_USER and identity passthrough

### Verified: auth IS enabled in production

`AUTH_ENABLED` is `true` because `GOOGLE_CLIENT_ID` is set in systemd env. Not a dev-mode bypass issue. The `getAuthUser()` function correctly returns the real user from the session cookie when auth is enabled.

### Hardcoded identity paths confirmed

Traced every `cpRequest` call. Results:

**Correctly passes authenticated user's `linux_user`:**
- Dashboard WS: input, resize, scroll, subscribe, session discovery
- Session create/restart/fork/dead-list
- Dashboard initial screen fetch

**Hardcodes `CP_DEFAULT_USER` (root):**
- `/api/pane` for cp-* sessions (line 431) — always `root`
- `/ws/terminal` legacy endpoint (line 2416) — always `root` via `attachCpToTerminalWs`
- Initial `listSessions` call at server startup (line 552) — uses `root`
- `unsubscribe` RPC (line 210) — doesn't pass user at all

### Why the admin never noticed

`frodex310@gmail.com` has `linux_user=root`. Both the correct path (`user.linux_user`) and the broken path (`CP_DEFAULT_USER=root`) resolve to the same value. The bug only manifests for non-root users like `cp-aaronb`.

## Credential lifecycle and forced re-auth

### Current state

- Session cookie: 24-hour expiry, HMAC-signed, HttpOnly
- No server-side session store — can't revoke individual sessions
- No forced re-auth mechanism — deactivating a user doesn't kick them out until cookie expires
- No idle timeout — cookie valid for full 24h regardless of activity
- WS token (`cp_ws_token`): mirrors session cookie, non-HttpOnly, same 24h expiry
- No way to invalidate all sessions for a user

### What's needed

**1. Server-side session/key store**
- Map of `apiKey → { userId, issuedAt, lastActivity, browserUid }`
- SQLite table or in-memory Map (loses state on restart, but restart invalidates all sessions anyway)
- Every API key validated against the store, not just HMAC

**2. Forced re-auth (admin action)**
- Admin clicks "Force re-login" on a user in admin panel
- Server deletes all API keys for that user from the store
- Next WS message or HTTP request with that key → rejected → client redirects to login
- User signs in again → new key issued

**3. Session timeout**
- Idle timeout: if no activity in N minutes, key expires
- Absolute timeout: key expires after N hours regardless of activity
- `lastActivity` updated on every message (throttled to once per minute to avoid write storms)

**4. UI state preservation on forced re-auth**
- Before redirecting to login, save to localStorage:
  - Which sessions were open
  - Layout/arrangement
  - Focused session
  - Scroll positions
- On login return, restore from localStorage
- Already partially in place — layout saves to localStorage via `_saveLayout()`

**5. Graceful forced re-auth UX**
- Server sends a WS message: `{ type: 'reauth-required', reason: 'admin-revoked' | 'expired' | 'idle-timeout' }`
- Client shows a non-dismissible overlay: "Your session has expired. Click to re-authenticate."
- Click → saves UI state → redirects to `/login?return=/`
- After login → redirects back → restores UI state
- During overlay: no data displayed, no input accepted (screen content hidden/blurred)
