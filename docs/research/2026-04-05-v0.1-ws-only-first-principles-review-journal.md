## Investigation 1: Did removing refreshSessions() lose dimension-change detection?

### Concern

The reviewer note on the legacy cleanup plan said: "Dropping 30s HTTP poll removes a fallback if WS events are missed." The old `refreshSessions()` called `updateCardForNewSize(t, session.cols, session.rows)` on every 30s poll — a silent self-healing loop for dimension changes from external clients.

### Finding: No gap exists

Both data paths send a full `screen` message (not `delta`) when dimensions change:

**Path B (local tmux) — server.mjs `diffState()` lines 581-583:**
```javascript
if (prev.width !== curr.width || prev.height !== curr.height) {
    return { type: 'screen', width: curr.width, height: curr.height,
      cursor: curr.cursor, title: curr.title, lines: curr.lines };
}
```
When tmux dimensions change (external SSH client resizes), `capturePane` returns new width/height. `diffState` detects the mismatch and sends a full `screen` message. Latency: 20ms (focused) to 500ms (unfocused).

**Path A (claude-proxy) — socket-server.ts `TerminalMirror` lines 124-142:**
```typescript
const dimsChanged = dims.cols !== this.lastCols || dims.rows !== this.lastRows;
if (dimsChanged) {
    // ... sends full screen message with new width/height
    this.emit(fullState);
    return;
}
```
Claude-proxy's 30ms poll detects dimension changes and emits a full `screen` message. Latency: <=30ms.

**Dashboard reception chain:**
1. WS message arrives → `routeDashboardMessage` (dashboard.mjs:1724)
2. → `routeEmbedMessageToSvg` (line 1731) → `obj.contentWindow.renderMessage(msg)` (line 1582)
3. terminal.svg renders, fires `window._screenCallback(msg)` (terminal.svg:616)
4. `_screenCallback` checks `msg.type === 'screen'` (dashboard.mjs:3849) → calls `updateCardForNewSize(t, msg.width, msg.height)` (line 3859)

**Conclusion:** The 30s poll was redundant. Dimension changes are detected within 20-500ms via the normal data path. No regression.

### Note on delta messages

Delta messages do NOT carry width/height and do NOT trigger `updateCardForNewSize`. This is correct — if dimensions haven't changed, `diffState` sends `delta`; if they have, it sends `screen`. The branching is explicit and intentional.

## Investigation 2: +/- button input blocking bug

### Symptom

After clicking + or - to resize a terminal card, keyboard input to that card is blocked until the user clicks elsewhere (unfocus/refocus).

### Root cause: DOM focus leak

`mkHdrBtn()` (dashboard.mjs:2766-2771) creates `<button>` elements. Browser auto-focuses the button on click. `shouldSendKeysToTerminal()` (line 4405-4422) blocks input when `document.activeElement` is a BUTTON (line 4416).

**No state corruption occurs.** `activeInputSession`, `focusedSessions`, `_suppressRelayout` are all correct throughout. The card resizes correctly, the terminal dimensions update, the DOM updates. Only the browser's `document.activeElement` pointing at the button causes the input guard to block.

### Fix applied

Added `btn.blur()` after the click handler in `mkHdrBtn()` (line 2770). This is the single factory for all 5 header buttons (−, +, ⊡, ⊞, ⌊), so the fix covers all of them. One-line change, no side effects.

### Affected buttons

All buttons created by `mkHdrBtn` had this issue — the only reason it was most noticeable on +/- is that users naturally keep typing after resizing. Minimize (⌊) would also block input but the card leaves focus so it's not observable. Fit-to-card (⊡) and fit-card-to-terminal (⊞) would also block input.

## Investigation 3: First-principles input path audit (WS-only)

### Keyboard input chain

```
User keypress
  → document.addEventListener('keydown') [dashboard.mjs:4424]
  → shouldSendKeysToTerminal() guard [line 4405]
    → checks: isUiOverlayActive(), document.activeElement tag, contentEditable
  → activeInputSession check [line 4426]
  → focusedSessions.size check [line 4427]
  → t = terminals.get(activeInputSession) [line 4453]
  → t.sendInput({ type: 'input', keys: ... }) [lines 4525/4532/4538/4544]
  → sendDashboardMessage() → ws.send() [single /ws/dashboard connection]
  → server.mjs handleDashboardWs input handler
  → for local tmux: tmuxAsync('send-keys', ...)
  → for cp sessions: cpRequest('sendInput', ...)
```

### Potential issues to investigate

1. **`document.activeElement` is fragile.** Any DOM mutation that creates a focusable element and the browser decides to focus it will block input. The button bug is one instance. Are there others? (SVG `<object>` load, modal open/close, dynamic DOM insertion?)

2. **`activeInputSession` can go stale.** Set by `setActiveInput()` on focus events. If a session is removed while active, does `removeTerminal` clear it? (Line 3019 sets `activeInputSession = null` — YES, covered.)

3. **Race between resize response and next keypress.** After +/- sends resize, the server responds with a new `screen` message. If the user types before the screen arrives, input goes to the old-dimension terminal. Is this a problem? (No — tmux accepts input regardless of pending resize. The input is valid.)

4. **`_suppressRelayout` flag lifetime.** Set on +/- click (lines 2819/2834), cleared when next screen message with changed dimensions arrives (line 3865). If the screen message never arrives (session dies, WS drops), the flag stays set forever. Impact: next external resize of that card won't re-layout. Low severity — flag is per-terminal and doesn't block input.
