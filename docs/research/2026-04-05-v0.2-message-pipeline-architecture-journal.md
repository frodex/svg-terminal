## Verified Architecture (Path A only)

```
tmux PTY
  ↓ (pty callbacks: onScreenChange, onCursorChange, onTitleChange)
TerminalMirror (30ms setInterval poll)
  ↓ (coalesced delta or full screen)
Unix socket (newline-delimited JSON)
  ↓ (cpOnDataChunk, splits on \n, JSON.parse)
bridgeClaudeProxySession handler
  ↓ (synchronous broadcast to subscriber Set)
WebSocket per browser client
  ↓ (routeDashboardMessage → routeEmbedMessageToSvg)
terminal.svg renderMessage()
  ↓ (_screenCallback → updateCardForNewSize if screen)
scheduleTerminalSurfaceRepaint (double rAF + forced reflow)
```

## Verified: TerminalMirror (claude-proxy/src/socket-server.ts)

### Mechanism: Polling, not event-driven
**Verified at:** socket-server.ts lines 122-172

`setInterval(() => { ... }, 30)` — hardcoded 30ms, not configurable.

### PTY callbacks set dirty flags only
**Verified at:** socket-server.ts lines 108-120

```typescript
pty.onScreenChange((startLine, endLine) => {
  this.dirty = true;
  for (let i = startLine; i < endLine; i++) this.dirtyLines.add(i);
});
pty.onCursorChange(() => { this.dirty = true; });
pty.onTitleChange((t) => { this.title = t; this.dirty = true; });
```

No data is sent from callbacks. All data emission happens from the poll timer.

### Coalescing behavior
**Verified at:** socket-server.ts lines 122-172

Each 30ms cycle:
1. Check if dimensions changed → if yes, emit full `screen` message, clear dirty, return
2. Check `this.dirty` → if false, return (no work)
3. Iterate `this.dirtyLines` Set → build `changed` object with only dirty line spans
4. Emit `delta` message with cursor + changed lines
5. Clear `dirtyLines` and `dirty`

**Result:** All changes within a 30ms window coalesce into ONE message. Maximum rate: ~33 messages/sec/session.

### Data emission: direct socket.write
**Verified at:** socket-server.ts lines 402-407

The emit callback passed to TerminalMirror does `client.socket.write(serializeMessage(...))` directly. No queue, no buffer, no backpressure handling. If the socket write buffer is full, the write return value is ignored and data is silently buffered by Node.js (or potentially lost if the buffer overflows).

### No backpressure handling
**Verified at:** socket-server.ts lines 405, 215

Neither the TerminalMirror emit path nor the broadcastSessionEvent path check the return value of `socket.write()` or listen for `'drain'` events. This is a gap but unlikely to matter in practice — Unix domain sockets have large kernel buffers and the data rate is low (~33 messages/sec × ~1KB/message = ~33KB/sec per session).

## Verified: svg-terminal message ingestion (server.mjs)

### cpOnDataChunk
**Verified at:** server.mjs lines 195-223

Newline-delimited JSON parsing with buffer for partial messages:
```javascript
cpBuf += chunk.toString();
const lines = cpBuf.split('\n');
cpBuf = lines.pop() || '';  // incomplete line stays in buffer
```

Processing is synchronous within the callback. Multiple messages in one chunk are processed in a for-loop. Each message is routed by checking `msg.event` and `msg.sessionId` → dispatched to `cpTerminalHandlers` Map.

### bridgeClaudeProxySession event handler
**Verified at:** server.mjs lines 580-623

Handles three event types:
1. `session-settings-changed` → updates `sessionPermCache`, sends `session-settings` to WS clients
2. `session-end` → sends `session-remove` to WS clients, cleans up watcher
3. `terminal` → extracts inner data, broadcasts to WS subscribers

Broadcast is synchronous: iterate `watcher.subscribers` Set, call `ws.send(json)` for each.

### BUG FOUND AND FIXED: upgrade handler dropped non-terminal events
**Location:** server.mjs line 549 (before fix)

The "upgrade watcher" path (when an existing watcher gets its first cp bridge) had:
```javascript
if (cpMsg.event !== 'terminal' || cpMsg.sessionId !== session) return;
```

This filtered out `session-end` and `session-settings-changed` events. Cards never got removed when sessions ended through this path. Fixed to match the full handler.

**When does the upgrade path fire?**
- When `sendSessionDiscovery` discovers a session and creates a watcher
- Then the subscribe handler calls `bridgeClaudeProxySession` again for the same session
- The second call finds the existing watcher and "upgrades" it

After Path B removal, this happens on every session because `sendSessionDiscovery` creates watchers before the subscribe handler runs.

## Verified: Focus/unfocus mechanism

**Verified at:** server.mjs lines 946-957

The focus handler adjusts `setInterval` timers for watchers that have `watcher.timer`. CP-bridged watchers have `timer: null` — the focus handler skips them.

**IMPORTANT:** svg-terminal does NOT send focus state back to claude-proxy. TerminalMirror always polls at 30ms regardless of whether the session is focused or unfocused in the dashboard.

This means:
- A session hidden behind 20 others still generates 33 messages/sec
- Background sessions consume the same CPU as focused ones
- The CAPTURE_INTERVAL_FOCUSED (20ms) and CAPTURE_INTERVAL_UNFOCUSED (500ms) constants are dead code after Path B removal

**Improvement opportunity:** Send focus state to claude-proxy so TerminalMirror can adjust its poll rate. Unfocused sessions could poll at 500ms+ (or even stop), saving CPU on both claude-proxy and svg-terminal.

## Verified: Session lifecycle events from claude-proxy

### Session end
**Verified at:** claude-proxy/src/pty-multiplexer.ts lines 305-315, session-manager.ts lines 170-181, socket-server.ts lines 195-206

Chain: PTY onExit → sessionManager.emit('session-end') → socket-server listener broadcasts to all subscribed clients → svg-terminal handler sends `session-remove` to WS → dashboard calls `removeTerminal()`

### Session settings changed
**Verified at:** claude-proxy/src/socket-server.ts broadcastSessionEvent, svg-terminal handler lines 584-598

Chain: claude-proxy broadcasts `session-settings-changed` with access data → svg-terminal updates `sessionPermCache` and sends `session-settings` to WS clients.

**GAP:** Dashboard has NO handler for `session-settings` messages. Permission changes (viewOnly, public, allowedUsers) update the server-side cache but the browser is never notified. A user who becomes view-only can keep typing until they reconnect.

## Improvement Opportunities

### 1. TerminalMirror: event-driven instead of polling

Currently: 30ms `setInterval` checks dirty flag.
PTY already fires `onScreenChange`, `onCursorChange`, `onTitleChange` callbacks.

**Proposal:** Instead of polling, emit data directly from the PTY callbacks with a debounce/coalesce timer:
```
onScreenChange → add to dirtyLines, start 16ms coalesce timer (if not running)
Timer fires → build delta from dirtyLines, emit, clear
```

Benefits:
- Zero CPU when session is idle (no poll checking dirty=false)
- Lower latency: emit within 16ms of change instead of waiting for next 30ms poll
- Natural coalescing: rapid changes within 16ms window batched into one delta

### 2. Focus-aware polling in claude-proxy

Send focus state from svg-terminal to claude-proxy via RPC. Unfocused sessions:
- Increase poll interval to 500ms (or stop TerminalMirror entirely)
- Resume at 30ms (or event-driven) when refocused
- With 20 sessions and 2 focused, this reduces idle poll load by ~90%

### 3. Dashboard handler for session-settings

Add a `session-settings` case in `routeDashboardMessage` that:
- Updates card UI (viewOnly indicator, lock icon, etc.)
- Blocks input if viewOnly changed to true
- Shows notification if access was revoked

### 4. Backpressure on WS broadcast

If a browser client can't keep up (slow connection, overwhelmed tab), `ws.send()` buffers internally. Eventually the buffer fills and the connection drops. Better approach:
- Check `ws.bufferedAmount` before sending
- If buffer exceeds threshold, skip delta messages (send next screen instead)
- Prevents memory growth from unbounded WS buffers

### 5. Double-rAF repaint hack (dashboard.mjs)

The `scheduleTerminalSurfaceRepaint` function forces two animation frames + two layout reflows per message. Testing showed removing it didn't fix input chunkiness (that was Path B), but we didn't test whether it's still NEEDED for rendering correctness under CSS3DRenderer. Needs isolated test:
- Remove the hack
- Check if terminal content still visually updates
- If it does: remove permanently (saves ~33ms latency per update)
- If it doesn't: investigate lighter alternatives (single rAF, no reflow)
