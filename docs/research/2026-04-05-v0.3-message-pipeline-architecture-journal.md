- Three.js render: 5-15ms

### The Double-rAF Problem

Two independent double-rAF hacks fire per screen/delta message:

1. **`nudgeObjectRepaint()`** in terminal.svg (line 583)
   - rAF → rAF → opacity trick → offsetHeight reflow → clear
   - Purpose: force CSS3DRenderer to notice SVG content changed inside `<object>`

2. **`scheduleTerminalSurfaceRepaint()`** in dashboard.mjs (line 1591)
   - rAF → rAF → translateZ bump → offsetHeight reflow → outline bump → offsetHeight reflow → renderer.render()
   - Purpose: force Chrome compositor to update CSS3D texture + Three.js scene render

Both exist to work around CSS3DRenderer's failure to detect content changes inside `<object>` sub-documents. They add ~64ms of deferred latency per update. At 33 messages/sec (one focused session), that's 33 double-rAF pairs per second — they overlap (rAF is shared across all pending callbacks) so the real overhead is less than 33×64ms, but it's still significant forced reflow work.

**Question for improvement:** Can these be unified into a single coalesced repaint? Or replaced with a lighter invalidation technique?

## Focus-Aware Polling: Current State

**svg-terminal sends focus state** via WS `focus` message (dashboard.mjs sends when cards are focused/unfocused).

**server.mjs handler** (line 946) adjusts `watcher.timer` intervals:
- Focused: 20ms (CAPTURE_INTERVAL_FOCUSED)
- Unfocused: 500ms (CAPTURE_INTERVAL_UNFOCUSED)

**But:** After Path B removal, all cp-bridged watchers have `timer: null`. The focus handler's `if (!watcher.timer) continue;` skips them. **Focus state has zero effect on cp sessions.**

**TerminalMirror always polls at 30ms** regardless of focus. A session hidden behind 20 others generates the same data rate as the focused one.

**Improvement opportunity:** Forward focus state to claude-proxy. TerminalMirror could:
- Unfocused: increase poll to 500ms or stop entirely
- Focused: keep 30ms (or go event-driven with 16ms debounce)
- With 20 sessions and 2 focused: ~90% reduction in idle poll load

## Improvement Candidates (Ranked by Impact)

### 1. Unify/reduce double-rAF overhead (~64ms → ~16ms)
- `nudgeObjectRepaint` (terminal.svg) and `scheduleTerminalSurfaceRepaint` (dashboard.mjs) both force reflows
- Could be replaced with a single coalesced rAF that runs once per frame, not per message
- Remove `renderer.render()` from repaint function — `animate()` already renders every frame
- Potential: **reduce screen update latency by ~48ms**

### 2. Focus-aware TerminalMirror polling
- Forward focus state to claude-proxy via new RPC (`setSessionFocus`)
- TerminalMirror: focused=30ms poll, unfocused=500ms or stopped
- 20 sessions × 33 msg/sec = 660 msg/sec → 2 sessions × 33 + 18 × 2 = 102 msg/sec
- Potential: **~85% reduction in idle CPU on both server and client**

### 3. Event-driven TerminalMirror (replace polling)
- PTY callbacks already fire on changes — currently just set dirty flags
- Could emit directly from callbacks with a 16ms debounce timer instead of 30ms setInterval
- Benefits: zero CPU when idle, lower latency (16ms vs avg 15ms), natural coalescing via debounce
- Risk: xterm.js buffer read timing — need to verify getScreenLine is safe from callbacks
- Potential: **zero idle CPU per session, ~15ms latency reduction when active**

### 4. Pause animate() when tab hidden
- `document.addEventListener('visibilitychange', ...)` → skip rAF when `document.hidden`
- Resume on tab visible
- Potential: **100% CPU savings when tab is in background**

### 5. Backpressure on WS broadcast
- Check `ws.bufferedAmount` before sending to slow clients
- Skip delta messages if buffer exceeds threshold (send next full screen instead)
- Prevents memory growth from unbounded WS send buffers
- Potential: **prevents OOM on slow connections**

### 6. Dead timer cleanup
- Remove focus interval adjustment code in server.mjs (lines 946-957) — dead after Path B removal
- Remove CAPTURE_INTERVAL_FOCUSED/UNFOCUSED constants — unused
- Remove SVG poll() functions in terminal.svg — standalone viewer is broken anyway

## Open Questions

1. **Can nudgeObjectRepaint be removed entirely?** The `animate()` loop renders every frame. If CSS3DRenderer invalidates `<object>` content on its own (or via the translateZ bump alone without the double rAF), the nudge is unnecessary. Needs testing.

2. **Is the 30ms TerminalMirror poll rate optimal?** 33fps of terminal data is more than most terminals need. 16ms (60fps) would be overkill. 50ms (20fps) might be visually acceptable for most use cases. Should this be configurable per session or based on focus?

3. **What triggers composeTitle updates?** Title includes uptime and connected users. Who calls it and how often? If it's called every 30ms poll, that's wasted work for mostly-static data.

4. **Does the WS library (`ws`) handle backpressure internally?** If `ws.send()` returns before the data is flushed, the library must be buffering. What's the buffer limit?
