## Changes Since v0.3

### DONE: TerminalMirrorDebounce replaces polling (claude-proxy)
- **Was:** 30ms `setInterval` poll, checks dirty flag every cycle, CPU cost even when idle
- **Now:** Event-driven 16ms debounce. PTY callbacks trigger `scheduleFlush()`, timer fires only when data changes. Zero CPU when idle.
- **Research confirmed safe:** xterm.js `onWriteParsed` fires after buffer is fully parsed. Node.js single-threaded — no reentrancy risk. (xterm.js issue #2022 confirms async-by-design.)
- **Tested in production:** Lightning fast typing, cross-browser sync instant. Initial lag was from stale browser flooding /ws/terminal (1099 attempts in 5 min), not the debounce.

### DONE: Single rAF repaint (dashboard.mjs)
- **Was:** `scheduleTerminalSurfaceRepaint` used double rAF (rAF→rAF→bumps→renderer.render) = ~32ms overhead per message
- **Now:** Single rAF with CSS3D invalidation bumps only. No `renderer.render()` — `animate()` loop already renders every frame at 60fps.
- **Savings:** ~16ms latency reduction per screen update

### DONE: animate() pauses when tab hidden (dashboard.mjs)
- **Was:** Three.js rendered at 60fps even when tab in background
- **Now:** `if (document.hidden) return;` at top of animate(). Zero GPU/CPU when tab hidden.

### DONE: WS backpressure (server.mjs)
- **Was:** All messages broadcast to all subscribers unconditionally. Slow clients fell further behind.
- **Now:** Skip delta messages when `ws.bufferedAmount > 64KB`. Screen messages always delivered. Prevents growing lag on slow connections.

### DONE: Async handleScroll (claude-proxy)
- **Was:** `execFileSync('tmux', ['capture-pane', ...])` — blocked entire event loop during scroll
- **Now:** `await execFileAsync(...)` — non-blocking. Other sessions keep updating during scroll captures.

### DONE: Fire-and-forget input + scroll (server.mjs)
- **Was:** `await cpRequest('input')` and `await cpRequest('scroll')` serialized messages
- **Now:** Both fire-and-forget with `.catch()` for error logging. Next message processes immediately.

### DONE: Dead code removal
- `CAPTURE_INTERVAL_FOCUSED` / `CAPTURE_INTERVAL_UNFOCUSED` constants — removed
- Focus interval adjustment handler — replaced with no-op + TODO

## Remaining Improvement Candidates (Ranked)

### 1. Focus-aware TerminalMirror
TerminalMirrorDebounce fires for ALL sessions regardless of focus. 16 sessions × up to 60 deltas/sec = high message rate. Forwarding focus state to claude-proxy so unfocused sessions stop their debounce timers would cut idle data rate dramatically.
**Status:** server.mjs focus handler is no-op with TODO. Needs claude-proxy RPC to receive focus state.

### 2. Browser-side message coalescing
528 messages/sec to each browser triggers 528 SVG DOM updates + forced reflows. Batching per session — only apply the latest message per animation frame — would decouple data arrival from rendering.
**Status:** Not started. Design needed.

### 3. nudgeObjectRepaint double-rAF in terminal.svg
terminal.svg:583 still has its own double-rAF opacity bump. This fires per message independently of `scheduleTerminalSurfaceRepaint`. Could potentially be unified or removed if the dashboard-side single-rAF bump is sufficient.
**Status:** Not tested independently.

### 4. Client-side scrollback buffer
Scroll round-trips to claude-proxy. Each browser's scroll affects all viewers. Client-side buffer would enable instant independent scrolling. Needs design — `t.screenLines` exists but no history accumulation.
**Status:** Design exploration done. Option C (claude-proxy sends scrollback events) preferred.

### 5. WS backpressure refinement
Current 64KB threshold is a rough heuristic. Could be adaptive based on client connection speed, or could send a "catch-up" full screen after dropping N deltas.
**Status:** Basic implementation done. Refinement deferred.

## Open Questions

1. **nudgeObjectRepaint vs scheduleTerminalSurfaceRepaint:** Are both CSS3D invalidation hacks needed? Can terminal.svg's nudge be removed if the dashboard-side bump is sufficient? Needs isolated test.

2. **TerminalMirrorDebounce 16ms vs other values:** Is 16ms optimal? Could match display refresh rate. Under heavy output, xterm.js 12ms time-budget chunking fires multiple `onWriteParsed` per parse batch — each starts a new debounce. Could test 8ms or 32ms.

3. **Per-browser identity for viewer tracking:** Single Unix socket means all browser users appear as one claude-proxy client. Need either svg-terminal-side title overlay or a new RPC to report individual viewers.

4. **Version mismatch auto-reload vs banner:** Currently auto-reloads. User preferred banner — revert when stale browser flooding is solved by other means.
