## Changes Since v0.3

### DONE: TerminalMirrorDebounce replaces polling (claude-proxy)
- **Was:** 30ms `setInterval` poll, checks dirty flag every cycle, CPU cost even when idle
- **Now:** Event-driven 16ms debounce. PTY callbacks trigger `scheduleFlush()`, timer fires only when data changes. Zero CPU when idle.
- **Research confirmed safe:** xterm.js `onWriteParsed` fires after buffer is fully parsed. Node.js single-threaded — no reentrancy risk. (xterm.js issue #2022 confirms async-by-design.)
- **Tested in production:** Lightning fast typing, cross-browser sync instant. Initial lag was from stale browser flooding /ws/terminal (1099 attempts in 5 min), not the debounce.

### DONE: Remove redundant renderer.render() from repaint (dashboard.mjs)
- **Was:** `scheduleTerminalSurfaceRepaint` used double rAF + explicit `renderer.render()` — extra Three.js render on every message
- **Now:** Double rAF kept (Chrome requires two frames for CSS3D `<object>` texture invalidation), but `renderer.render()` removed — `animate()` loop already renders every frame at 60fps.
- **Savings:** ~5-15ms CPU per update (no extra Three.js render). Double rAF latency remains (~32ms) — unavoidable for Chrome.
- **Tested:** Single rAF caused cursor lag and dropped visual updates in Chrome. Edge was unaffected. Chrome's CSS3DRenderer needs the second frame to rasterize SVG content changes inside `<object>` elements.

### DONE: animate() pauses when tab hidden (dashboard.mjs)
- **Was:** Three.js rendered at 60fps even when tab in background
- **Now:** `if (document.hidden) return;` at top of animate(). Zero GPU/CPU when tab hidden.

### DONE: WS backpressure (server.mjs)
- **Was:** All messages broadcast to all subscribers unconditionally. Slow clients fell further behind.
- **Now:** Skip delta messages when `ws.bufferedAmount > 64KB`. Screen messages always delivered. Prevents growing lag on slow connections.

### DONE: Async handleScroll (claude-proxy)
- **Was:** `execFileSync('tmux', ['capture-pane', ...])` — blocked entire event loop during scroll
- **Now:** `await execFileAsync(...)` — non-blocking. Other sessions keep updating during scroll captures.

### DONE: Fire-and-forget input + scroll (server.mjs)
- **Was:** `await cpRequest('input')` and `await cpRequest('scroll')` serialized messages
- **Now:** Both fire-and-forget with `.catch()` for error logging. Next message processes immediately.

### DONE: Dead code removal
- `CAPTURE_INTERVAL_FOCUSED` / `CAPTURE_INTERVAL_UNFOCUSED` constants — removed
- Focus interval adjustment handler — replaced with no-op + TODO

## Remaining Improvement Candidates (Ranked)

### 1. Focus-aware TerminalMirror
TerminalMirrorDebounce fires for ALL sessions regardless of focus. 16 sessions × up to 60 deltas/sec = high message rate. Forwarding focus state to claude-proxy so unfocused sessions stop their debounce timers would cut idle data rate dramatically.
**Status:** server.mjs focus handler is no-op with TODO. Needs claude-proxy RPC to receive focus state.

### 2. Browser-side message coalescing
528 messages/sec to each browser triggers 528 SVG DOM updates + forced reflows. Batching per session — only apply the latest message per animation frame — would decouple data arrival from rendering.
**Status:** Not started. Design needed.

### 3. nudgeObjectRepaint double-rAF in terminal.svg
terminal.svg:583 still has its own double-rAF opacity bump. This fires per message independently of `scheduleTerminalSurfaceRepaint`. Could potentially be unified or removed if the dashboard-side single-rAF bump is sufficient.
**Status:** Not tested independently.

### 4. Client-side scrollback buffer
Scroll round-trips to claude-proxy. Each browser's scroll affects all viewers. Client-side buffer would enable instant independent scrolling. Needs design — `t.screenLines` exists but no history accumulation.
**Status:** Design exploration done. Option C (claude-proxy sends scrollback events) preferred.

### 5. WS backpressure refinement
Current 64KB threshold is a rough heuristic. Could be adaptive based on client connection speed, or could send a "catch-up" full screen after dropping N deltas.
**Status:** Basic implementation done. Refinement deferred.

## Wire Format Analysis: JSON Schema Overhead

### Measured (real production data)

**Screen message:** 9165 bytes total, 2878 bytes text content = **68% overhead**
- 40 lines, 212 spans
- 6008 bytes of JSON structure (property names, brackets, quotes, colons)
- 279 bytes of wrapper (event, sessionId, type, width, height, cursor, etc.)

**Span structure (repeated 212 times per screen):**
```json
{"text":" 42 -","fg":"#d75f5f","bg":"#5f0000"}   // 51 bytes → 5 bytes of text, 46 bytes overhead
{"text":"     "}                                   // 15 bytes → 5 bytes of text, 10 bytes overhead
```

Property names `"text"`, `"fg"`, `"bg"`, `"bold"`, `"italic"`, `"underline"` repeat in every span. 212 spans × ~25 bytes of key overhead = ~5000 bytes of pure key repetition.

### Option A: WebSocket permessage-deflate compression

**Tested:** Added `perMessageDeflate: { threshold: 1024 }` to WebSocketServer.

**Pros:**
- Transparent to all existing code — transport-level change only
- JSON with repetitive keys compresses extremely well (~70-80% reduction estimated)
- 9KB screen message → ~2KB compressed
- Threshold skips tiny input messages (<1KB go uncompressed)

**Cons:**
- Adds CPU cost per message: zlib compress (server) + decompress (browser)
- At 528 messages/sec, that's 528 compress + 528 decompress operations per second
- Chrome showed degraded performance — unclear if from compression CPU or coincidental with other issues (stale browser was flooding at the same time)

**Timing impact:**
- Compress: ~0.1-0.5ms per message (zlib on small JSON)
- Decompress: ~0.05-0.2ms per message (browser native zlib)
- Network savings: ~70% bandwidth reduction (9KB → ~2KB per screen message)
- Net: saves network time on slow connections, costs CPU on fast local connections

**Status:** Reverted. Should retest after focus-aware TerminalMirror reduces message rate from 528/sec to ~66/sec — compression overhead would be 8× lower.

### Option B: Compact span format (protocol change)

Replace verbose span objects with positional arrays:

```
Current:   {"text":" 42 -","fg":"#d75f5f","bg":"#5f0000"}         // 51 bytes
Compact:   [" 42 -","#d75f5f","#5f0000"]                          // 31 bytes (39% smaller)
Indexed:   [" 42 -",218,88]                                       // 18 bytes (65% smaller)
```

**Indexed variant:** Map colors to indices in a palette sent once per session. Most terminals use <256 unique color combinations. A palette of `["#d75f5f", "#5f0000", "#ffffff", ...]` sent on first screen, then spans reference by index.

**Timing impact:**
- Serialize: slightly faster (no key strings to write)
- Parse: slightly faster (array access vs property lookup)
- Network: 39-65% reduction in span data
- 9165 byte screen → ~5500 bytes (compact) or ~3600 bytes (indexed)

**Cons:**
- Breaking protocol change — both claude-proxy and terminal.svg must change simultaneously
- Less readable for debugging (array positions vs named properties)
- Palette management adds complexity

### Option C: Binary protocol (MessagePack / CBOR)

Replace JSON entirely with a binary format.

**Timing impact:**
- Serialize: ~2-5× faster than JSON.stringify for structured data
- Parse: ~2-3× faster than JSON.parse
- Size: ~30-50% smaller than JSON (no quotes, shorter type markers)

**Cons:**
- Requires library on both sides
- Not human-readable for debugging
- Largest code change of all options
- Browser needs MessagePack/CBOR decoder

### Option D: Hybrid — compress large messages only

Combine JSON format (keep readable) with selective compression:
- Messages >2KB: apply zlib before ws.send, decompress on receive
- Messages <2KB: send raw JSON
- Application-level, not WS extension — more control over threshold and timing

**Timing impact:**
- Only large messages pay the compression cost
- Small deltas (1-3 changed lines) go uncompressed — fast
- Full screens (all lines) compressed — bandwidth savings where it matters

**Cons:**
- Application-level compression means both sides need explicit compress/decompress code
- More complexity than permessage-deflate (which is transparent)

### Recommendation

**Short term:** Retest `perMessageDeflate` after focus-aware TerminalMirror reduces message rate. At 66 messages/sec instead of 528, compression CPU overhead is negligible and bandwidth savings are real — especially through Cloudflare tunnel.

**Medium term:** Option B (compact span format) if bandwidth remains a concern. The 39% reduction from dropping property names is significant and has zero CPU overhead — it's actually faster to serialize and parse.

**Long term:** Option D (hybrid selective compression) for the best of both worlds — readable protocol for small messages, compressed for full screens.

**Not recommended:** Option C (binary protocol). The complexity cost outweighs the performance gain for this use case. JSON is fine with compact spans + compression.

## CSS3DRenderer Invalidation: Findings

### Double rAF is Chrome-specific requirement

**Tested:** Single rAF with CSS3D invalidation bumps (translateZ + outline).
**Result:** Chrome showed cursor lag and dropped visual updates. Edge was unaffected.

**Root cause:** Chrome's CSS3DRenderer compositor doesn't re-rasterize `<object>` SVG content on the same frame that the DOM mutation happens. It needs one frame to notice the mutation, then a second frame to rasterize the new content. The double rAF ensures both frames complete before the next `animate()` render picks up the change.

**This is not configurable or browser-detectable** — both browsers negotiate the same CSS3D path. The double rAF is harmless on Edge (adds ~16ms of deferred latency that Edge doesn't need) and required on Chrome (prevents visual corruption without it).

**The `renderer.render()` call that was inside `scheduleTerminalSurfaceRepaint` IS redundant** — the `animate()` loop renders every frame at 60fps. Removing it saves ~5-15ms of CPU per update (528 × 5-15ms = 2.6-7.9 seconds of Three.js render time saved per second). This is the only optimization we can make here without losing Chrome compatibility.

2. **TerminalMirrorDebounce 16ms vs other values:** Is 16ms optimal? Could match display refresh rate. Under heavy output, xterm.js 12ms time-budget chunking fires multiple `onWriteParsed` per parse batch — each starts a new debounce. Could test 8ms or 32ms.

4. **Version mismatch auto-reload vs banner:** Currently auto-reloads. User preferred banner — revert when stale browser flooding is solved by other means.
