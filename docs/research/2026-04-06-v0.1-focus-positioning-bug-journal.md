## Bug Description

When focusing a card (especially newly created, first-time focus), the card lands ~half off the top of the screen and shifted to one side. It doesn't center in the usable viewport area.

## Investigation

### What `focusTerminal()` does (line 4439-4505)

Single-card focus positions:
- Card at world origin: `t.targetPos = { x: 0, y: 0, z: 0 }` (line 4459)
- Camera at `(0, 0, camDist)` looking at `(0, 0, 0)` (lines 4491-4498)
- Camera distance calculated to fill ~90% of viewport height

**Problem:** This centers the card on the full viewport, not the usable area. Three UI elements eat into the viewport:
- **Sidebar:** 140px on the left (`SIDEBAR_WIDTH`, line 1000)
- **Top bar:** 44px at the top (`TOP_BAR_H`, line 1002)
- **Status bar:** 50px at the bottom (`STATUS_BAR_H`, line 1142)

The card appears shifted left and up because the "center" of the usable area is not the center of the full viewport.

### What `calculateFocusedLayout()` does correctly (line 1145-1309)

Multi-focus layout correctly accounts for all three offsets:
- `availW = screenW - SIDEBAR_WIDTH` (line 1167)
- `availH = screenH - STATUS_BAR_H - TOP_BAR_H` (line 1168)
- Screen positions computed within the usable area, then converted to world coords
- Card screen center: `p._cx = originX + colX[p.col]` (relative to usable area, line 1269)
- Card screen center Y: `TOP_BAR_H + originY + ...` (accounts for top bar, line 1270)
- World conversion: `wx = (p._cx - screenW / 2) * px2w` (line 1289)

The camera still goes to `(0, 0, camZ)` in multi-focus, but all the cards are positioned in world space to account for the sidebar/bar offsets. This works because the cards move, not the camera.

### Interesting detail: dead code at lines 4487-4489

```javascript
// Offset for sidebar (card at origin, camera shifted so card centers in usable area)
const visHAtDist = 2 * camDist * halfTan;
const px2w = visHAtDist / window.innerHeight;
```

The comment says "offset for sidebar" and `px2w` is computed, but **never used** — the offset is never applied. This looks like someone started the fix but didn't finish it.

### `calculateSlotLayout()` (line 1318+)

Also correctly uses `availW = screenW - SIDEBAR_WIDTH` and `availH = screenH - STATUS_BAR_H - TOP_BAR_H`.

## Fix Options

### Option A: Offset the card position (match multi-focus pattern)

Move the card to a world position that maps to the usable-area center. Camera still at `(0, 0, camDist)`. This matches how `calculateFocusedLayout` works — cards move, camera stays centered.

**Pro:** Consistent with multi-focus approach.
**Con:** Card is no longer at world origin; any code that assumes focused card is at (0,0,0) breaks.

### Option B: Offset both card and camera together

Move card to offset position, camera looks at that position. Card and camera travel together.

**Pro:** Camera always looks directly at the card. No assumptions about origin.
**Con:** Slightly more complex, but actually cleaner since the lookAt target matches the card.

### Option C: Offset camera only (shift the camera, card stays at origin)

Keep card at `(0,0,0)`, shift camera X/Y so the origin maps to the usable-area center.

**Pro:** Card stays at origin, simplest change.
**Con:** Camera is off-center, which may interact badly with orbit controls. The camera's forward vector isn't along Z anymore.

## Root Cause Found

**Lines 4216-4217 in `updateCardForNewSize()`** unconditionally overwrite `targetPos` with `currentPos`:

```javascript
t.targetPos.x = t.currentPos.x;
t.targetPos.y = t.currentPos.y;
```

This is inside a block guarded by `dwWorld !== 0 || dhWorld !== 0` (card size changed). With center anchor (0.5), the shift to `currentPos` is zero — but `targetPos` still gets overwritten with whatever `currentPos` is at that moment.

### The race condition

1. `focusTerminal()` sets `targetPos = {0, 0, 0}`, starts 1.5s morph
2. During morph, card resize fires (SVG measurement correction at +100ms, or first screen data, or perf tier resize cascade)
3. `updateCardForNewSize` enters anchor block because card size changed
4. With center anchor, shift is 0 — `currentPos` unchanged
5. **But targetPos gets overwritten with mid-morph currentPos** (somewhere between ring and origin)
6. Card converges to wrong position and stays there

### Why intermittent
- Depends on whether a resize arrives during the 1.5s morph window
- New cards have `_needsMeasuredCorrection = true` → almost always triggers
- Perf tier cascading also triggers resizes
- Re-focusing works because no resize happens (measurement already applied)

### Reproduction log (2026-04-06)
- Two focusTerminal calls: cp-SVG-UI-Doctor-02 then cp-BATTLETECH-Session-01
- Heavy perf tier activity: 0→2→0→1→2, multiple resize tests
- Last card (BATTLETECH) exhibited the bug

### Fix
Lines 4216-4217 should apply the same shift formula to `targetPos` instead of replacing it:
```javascript
t.targetPos.x -= dwWorld * (fx - 0.5);
t.targetPos.y += dhWorld * (fy - 0.5);
```
With center anchor, shift is 0, targetPos untouched. With non-center anchor, targetPos shifts in sync with currentPos.
