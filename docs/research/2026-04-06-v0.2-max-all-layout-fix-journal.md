## Problem Statement

Max All currently only expands cards in one dimension. Cards don't fully fill their layout slots. The user wants a clear 4-step pipeline:

1. **Frustum fit** (no mutation) — position card in 3D so it fills the slot as much as possible while preserving current aspect ratio (letterboxed)
2. **Aspect expansion** — grow the card DOM to fill the remaining gap (either horizontal or vertical), changing the card's aspect ratio
3. **Fit terminal to card** — resize the terminal (cols/rows) to match the new card aspect
4. **Font equalization** (future) — equalize POV font size across all cards

## Current Implementation Analysis

### `maxAllFocused()` (line 506-525)
```
1. Clear slotOrder
2. calculateFocusedLayout()  — positions cards, sets _slotRect and _slotFit
3. For each focused card: maximizeCardToSlot(t, true)  — sends resize command
4. Pin slotOrder
5. calculateFocusedLayout()  — re-layout with new pinned order
```

### `maximizeCardToSlot()` (line 937-993)
Uses `_slotFit` to determine which dimension is constrained:
- If `constrainedBy === 'width'`: grows rows (rowScale = slotH / fitH)
- If `constrainedBy === 'height'`: grows cols (colScale = slotW / fitW)

Then sends resize command to terminal and sets `_fillSlot = { cols, rows }`.

### `updateCardForNewSize()` (line 4156+)
When the resize response arrives with matching `_fillSlot` cols/rows:
- Uses a fixed `baseH = 1200` DOM pixels
- Derives cardW from terminal aspect
- Does NOT use the slot dimensions to size the card

### `calculateSlotLayout()` (line 1318+)
- Computes letterbox fit: `fitW/fitH` based on card aspect vs slot aspect
- Sets `_slotFit = { fitW, fitH, constrainedBy, slotW, slotH }`
- Sets `_slotRect = { x, y, w, h }` (slot pixel bounds)
- Positions card at Z-depth where `fitH` fills the right screen fraction

## Problems Identified

### Problem 1: Card sizing in updateCardForNewSize doesn't match slot
When `fillSlotActive`, the code sets card size from a hardcoded `baseH = 1200` and derives width from terminal aspect. This creates a card with the right aspect ratio but NOT sized to fill the slot. The subsequent layout re-run then letterboxes this new-aspect card into the slot, potentially leaving gaps again if the slot's aspect doesn't exactly match.

### Problem 2: Order of operations is wrong
Current flow:
1. Layout → sets _slotFit with current card aspect
2. maximizeCardToSlot → sends resize based on _slotFit
3. Terminal resizes → updateCardForNewSize fires → card DOM resized
4. Layout re-runs → card has new aspect, may still not fill slot

The user's desired flow:
1. Layout → letterbox fit (frustum only, no mutation) ← this already happens
2. Expand card DOM to fill slot gap ← this should be a DOM/3D operation, not a terminal resize
3. Fit terminal to new card size ← then resize terminal to match

### Problem 3: maximizeCardToSlot conflates card expansion with terminal resize
It calculates new cols/rows and sends resize immediately. But the card DOM should expand FIRST (step 2), then the terminal should be told to match (step 3). Currently there's no step where the card DOM fills the slot independently of the terminal.

## User's Desired Pipeline

```
Step 1: calculateFocusedLayout()
  → Cards positioned in 3D, letterboxed into slots
  → _slotRect and _slotFit set
  → This already works

Step 2: expandCardToFillSlot(t)
  → Read _slotFit to know which dimension has gap
  → Expand card DOM to fill the full slot (not letterboxed)
  → Change baseCardW/H to match slot proportions
  → Re-position in 3D at correct Z-depth for full slot coverage
  → Card aspect changes, terminal content stretches

Step 3: fitTerminalToCard(t)  
  → Calculate cols/rows that fit the new card dimensions
  → Using measured cell size
  → Send resize to terminal
  → When response arrives, card is already the right size

Step 4: (future) equalizeAllFocused()
  → Already implemented (line 531)
  → Would run after step 3 completes
```

## Key Insight

The current code tries to do steps 2 and 3 simultaneously by computing final cols/rows and letting the resize response drive the card sizing. This creates a chicken-and-egg problem: the card needs to be slot-sized to calculate the right cols/rows, but the card sizing waits for the resize response.

The fix: make step 2 a pure DOM/3D operation (expand card, reposition in frustum) and step 3 a consequence (terminal told to match the already-expanded card).

## Confirmed Pipeline (user-corrected)

### Step 1: Frustum fit (existing `calculateFocusedLayout`)
Card positioned at Z-depth where it letterbox-fits the slot. For a 4:3 card in a 16:9 slot: card touches top and bottom, gaps on sides. `_slotFit` records which dimension is constrained and the fit/slot dimensions.

### Step 2: Mutate unconstrained dimension (new `expandCardsToSlots`)
Expand the card DOM in the unconstrained dimension so its aspect matches the slot. Key insight from user: **no Z repositioning needed**. The Z-depth from step 1 is correct for the constrained dimension. Expanding the other dimension at the same depth fills the gaps exactly.

Math: if constrained by width (height letterboxed), scale `baseCardH` by `slotH / fitH`. If constrained by height (width letterboxed), scale `baseCardW` by `slotW / fitW`.

### Step 3: Fit terminal to card (`optimizeTermToCard` per card)
Card is now slot-shaped. Terminal content is stretched. `optimizeTermToCard` measures the card, calculates cols/rows that fit, sends resize. `_lockCardSize = true` prevents the resize response from changing the card size.

### Step 4 (future): Font equalization
`equalizeAllFocused()` already exists. Not part of this change.
