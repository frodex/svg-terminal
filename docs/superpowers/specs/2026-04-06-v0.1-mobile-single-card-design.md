# Mobile Single-Card Terminal View — Design Spec v0.1

**Date:** 2026-04-06
**Status:** Draft — needs brainstorm session
**Context:** Mobile browsers can't handle 16 simultaneous SVG `<object>` elements + Three.js CSS3DRenderer + 528 messages/sec. The 3D dashboard is a desktop experience. Mobile needs a purpose-built view.

---

## Core Concept

One terminal card visible at a time. No 3D scene, no ring layout, no CSS3DRenderer. The SVG terminal fills the screen. Session switcher in the top bar. Input via bottom text field + voice.

---

## Layout

```
┌──────────────────────────────┐
│ ☰  [session-name ▼]   [👤]  │  ← Top bar: hamburger, session dropdown, user
├──────────────────────────────┤
│                              │
│                              │
│     SVG Terminal             │  ← Full-screen terminal (single <object>)
│     (single session)         │     No CSS3DRenderer — direct DOM
│                              │
│                              │
│                              │
├──────────────────────────────┤
│ [input field        ] [🎤]  │  ← Bottom bar: text input + voice button
└──────────────────────────────┘
```

### Top Bar
- Hamburger menu (same as desktop — new session, restart, fork, admin, help)
- Session dropdown: lists all available sessions by name, shows owner
  - Selecting a session: unsubscribes current, subscribes new, loads SVG
  - Current session highlighted
  - Badge for sessions with activity since last viewed
- User pill (same as desktop)

### Terminal Area
- Single SVG `<object>` element, no Three.js, no CSS3DRenderer
- Fills available space between top bar and input bar
- Touch scroll = terminal scrollback (not page scroll)
- Pinch zoom for text size (CSS transform on the object)
- No card chrome, no header buttons (+/−/⊡/⊞/⌊)
- Resize managed automatically based on viewport

### Bottom Input Bar
- Persistent text input field (always visible, not the desktop's direct-keydown capture)
- On submit (Enter or send button): sends input to terminal, clears field
- Voice button (🎤): browser Speech Recognition API
  - Tap to start, tap to stop (or auto-stop on silence)
  - Transcribed text appears in input field
  - User can edit before sending
  - Visual indicator when listening (pulsing border or mic icon change)
- Special key buttons: row of common keys above the input field
  - Ctrl+C, Tab, Up, Down, Escape, Ctrl+Z
  - Scrollable horizontal row for more keys

## Subscription Model

**Single-session subscription:**
- On session select: `sendDashboardMessage({ type: 'unsubscribe', session: previous })`
- Then: `sendDashboardMessage({ type: 'subscribe', session: selected })`
- Server bridges only one session at a time for this client
- No WS backpressure issues — max ~33 messages/sec (one session)

**Session list:**
- Fetched via `get-sessions` WS message (same as fork/restart dialogs use)
- Refreshed on session-add / session-remove events
- Session-add/remove events still received for ALL sessions (lightweight — just names, no screen data)

**Activity tracking:**
- When a session-add event arrives for a non-subscribed session, mark it as "has activity" in the dropdown
- Clear the badge when user switches to that session

## Detection: Mobile vs Desktop

Options:
- **A) URL path:** `/mobile` serves mobile view, `/` serves desktop. User chooses.
- **B) User-Agent / viewport detection:** Auto-detect and redirect. `window.matchMedia('(max-width: 768px)')` or `navigator.maxTouchPoints > 0`.
- **C) Responsive:** Same page, CSS media queries hide 3D scene and show mobile layout below breakpoint.

**Recommendation:** Option A with auto-redirect. Mobile detection is unreliable (tablets, desktop touch screens). A dedicated `/mobile` path is explicit and testable. Auto-redirect based on viewport width with a "switch to desktop" link.

## Voice Input

### Browser Speech Recognition API
```javascript
const recognition = new (window.SpeechRecognition || window.webkitSpeechRecognition)();
recognition.continuous = false;      // stop after one utterance
recognition.interimResults = true;   // show partial transcription
recognition.lang = 'en-US';

recognition.onresult = (event) => {
  const transcript = event.results[0][0].transcript;
  inputField.value = transcript;     // populate input field
};
```

**Browser support:** Chrome (Android + desktop), Edge, Safari (iOS 14.5+). Firefox: not supported.

**UX flow:**
1. Tap 🎤 → mic icon turns red/pulsing, "Listening..." label
2. Speak command → interim results show in input field (gray text)
3. Silence detected → final result replaces input (black text)
4. User reviews, edits if needed, taps Enter/Send
5. Or: tap 🎤 again to cancel

**Voice commands (stretch goal):**
- "switch to [session name]" → switches session without typing
- "scroll up / scroll down" → scroll terminal
- "control c" → sends Ctrl+C
- These would be processed before the text reaches the input field

## What's Shared with Desktop

- `/ws/dashboard` WebSocket connection (same protocol)
- `terminal.svg` rendering (same SVG, same renderMessage)
- Session lifecycle messages (session-add, session-remove, create, fork, restart)
- Auth flow (same OAuth, same API keys)
- All server-side code unchanged

---

## What's Different from Desktop

| Desktop | Mobile |
|---------|--------|
| Three.js CSS3DRenderer, 3D scene | Direct DOM, no 3D |
| All sessions subscribed simultaneously | Single session subscribed |
| Direct keydown capture (document-level) | Input field + voice |
| Multi-focus (ctrl+click) | Single session always |
| Sidebar thumbnails | Session dropdown list |
| Card chrome (+/−/⊡/⊞/⌊) | Auto-sized, no controls |
| Mouse orbit/drag/zoom | Touch scroll/pinch |
| 528+ messages/sec | ~33 messages/sec |

## Open Questions

1. **Terminal resize on mobile:** Viewport changes (keyboard appears/disappears, orientation change). How many cols/rows? Should it resize the tmux session (affects other viewers) or just scale the SVG?

2. **Touch selection:** How does text selection work on mobile? Long-press? The desktop uses Alt+drag which doesn't exist on touch.

3. **Landscape mode:** Should it rotate the terminal? Some phones have wide-but-short landscape viewports — might want more cols, fewer rows.

4. **Offline/PWA:** Should the mobile view work as an installable PWA with service worker caching?

5. **Notification on session activity:** When viewing session A, session B has output. Push notification or just the badge in the dropdown?
