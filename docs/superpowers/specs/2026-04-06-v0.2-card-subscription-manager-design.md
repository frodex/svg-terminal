# Card Subscription Manager — Design Spec v0.2
**Date:** 2026-04-06
**Status:** Draft
**Preceding:** 2026-04-06-card-subscription-manager-design.md
**Changes from v0.1:** User NOTES feedback — layout changes, thumbnail buttons, admin terminate, save current state

---

## Summary

A persistent card subscription system that controls which terminal sessions appear in the dashboard. Users manage subscriptions via a "CARDS >" sub-panel in the hamburger menu. Unsubscribed sessions are never sent to the browser — zero rendering and WebSocket cost. A status bar shows card counts, and an orange badge on the top menu alerts when cards are hidden.

---

## User-Facing Behavior

### Top Menu Bar
- Orange badge: `YOU HAVE 7 HIDDEN CARDS` — visible when any cards are unsubscribed. Clicking opens the CARDS sub-panel **directly** (not hamburger menu first).

### Hamburger Menu
- New menu item: `CARDS >` — opens a sub-panel (flyout or inline expansion).

### CARDS Sub-Panel

**Header controls:**
- `[ ] Always show new cards` — auto-subscribe to any new session that appears
- `[ ] Always show MY new cards` — auto-subscribe to sessions owned by the current user (includes sidecars like browser cards)
- `[Save Current State]` button — snapshots the current dashboard state and writes it to the profile. Cards currently displayed → subscribed, cards currently thumbnail-paused → sticky-paused, cards removed from dashboard → unsubscribed. One-click persistence of whatever you've set up via thumbnails.

**Session list** (scrollable, grouped by state):

Action buttons justified **LEFT** (before session info). Status indicator on **RIGHT** (shows current *dashboard* state, which may differ from the configured subscription state — e.g., a subscribed card could be temporarily paused via thumbnail).

```
DISPLAYED
  [⏸][⏹]  cp-SVG-UI-Doctor-02 | root | 2h31m         ● active
  [⏸][⏹]  cp-BATTLETECH-01    | cp-aaronb | 45m       ● active
  [⏸][⏹]  cp-AARON            | root | 1d2h           ⏸ temp-paused

PAUSED (sticky)
  [▶][⏹]  cp-STEWARD-01       | root | 5h              ⏸ paused

HIDDEN
  [▶]     cp-old-session       | cp-joshm | 3d          ○ hidden

ADMIN: TERMINATE                              (admin/root only)
  [☐] cp-SVG-UI-Doctor-02    [☐] cp-BATTLETECH-01    ...
  [Submit with PIN]
```

Each row shows: **action buttons (left), session name, owner, age, status indicator (right)**

**Action buttons (left-justified):**
- `▶` (green) — subscribe / resume
- `⏸` — sticky-pause (stops data, persists across reload)
- `⏹` (red stop) — unsubscribe (removes card, not loaded on reload). NOT ✕ — that implies "close/terminate" which is a different action.

**Status indicators (right-justified):**
- `● active` — card is displayed and receiving data
- `⏸ temp-paused` — temporarily paused via thumbnail (not persistent)
- `⏸ paused` — sticky-paused via CARDS menu (persistent)
- `○ hidden` — unsubscribed, not in dashboard

### Search and Sort

**Search bar** at top of session list:
- Text input filters sessions by title (substring match, case-insensitive)
- Results update live as you type
- Works across all states (displayed, paused, hidden) since server provides titles for all sessions

**Sort options** (dropdown or toggle buttons):
- By name (A-Z)
- By age (newest first / oldest first)
- By owner
- By state (displayed → paused → hidden, default)

**Phase 2 — Content search:**
- `[ ] Search content` checkbox next to search bar
- When enabled, also searches the terminal text buffer (`screenLines[]`)
- Only works for displayed and paused cards (they have buffers in memory)
- Hidden/unsubscribed cards have no buffer — title-only match for those
- Visual indicator on results showing whether match was in title or content

**Admin-only: Terminate section**
- Visible only to admins and root
- Checkbox per session for batch selection
- `[Submit with PIN]` button — prompts for admin PIN, then terminates selected sessions on the server
- Terminated sessions are killed (not just unsubscribed)

### Thumbnail Buttons

Current: only ⏸ (pause, non-persistent)

New thumbnail button/indicator set:
- `▶` (green play) — **state indicator**, shows when card is active. Click toggles to paused.
- `⏸` (orange pause) — **state indicator**, shows when card is paused. Click toggles to active.
- `⏹` (red stop) — **action button**, unsubscribe. Persistent — removes card from dashboard and thumbnails, writes to profile. Must go to CARDS menu to re-activate.
- `✕` (red X, admin-only) — **action button**, terminate session on server. Requires PIN prompt.

The ▶/⏸ toggle is the same non-persistent pause as today (just with clearer icons). The red ⏹ is new and persistent.

### Status Bar (Bottom)
- Text: `25 available / 7 displayed / 2 paused`
- Updated on `card-counts` message
- Clicking opens CARDS sub-panel **directly** (not hamburger menu first)

### Save Current State

The `[Save Current State]` button in the CARDS panel captures the current dashboard state and writes it to the profile:

- Cards currently in the dashboard and active → `subscribed`
- Cards currently in the dashboard but thumbnail-paused (`_muted = true`) → `paused` (sticky)
- Cards not in the dashboard (previously removed or never loaded) → `unsubscribed`

This bridges the non-persistent thumbnail actions with the persistent profile. Users can casually manage cards via thumbnails during a session, then "lock in" the setup when they're happy.

### Pause Behavior — Three Layers

| Action | Source | Persistent? | Effect |
|--------|--------|-------------|--------|
| Pause via thumbnail ⏸ | Thumbnail button | No — resets on reload | Stops data, card stays in UI with overlay |
| Pause via CARDS menu ⏸ | Subscription manager | Yes — stored in profile | Stops data, card shows paused on reload |
| Save Current State | CARDS panel button | Yes — bulk write | Snapshots ALL current states to profile |
| Unsubscribe via thumbnail ⏹ | Thumbnail button | Yes — writes to profile | Card removed, not loaded on reload |
| Unsubscribe via CARDS menu ⏹ | Subscription manager | Yes — stored in profile | Card removed, not loaded on reload |

Thumbnail pause (⏸) does NOT write to the profile. All other actions write immediately.

## Data Model

### New table: `card_subscriptions`

```sql
CREATE TABLE IF NOT EXISTS card_subscriptions (
  user_email TEXT NOT NULL,
  session_name TEXT NOT NULL,
  state TEXT NOT NULL DEFAULT 'subscribed',  -- subscribed | paused | unsubscribed
  updated_at TEXT NOT NULL DEFAULT (datetime('now')),
  PRIMARY KEY (user_email, session_name)
);
```

States:
- `subscribed` — session loaded and active on connect
- `paused` — session loaded but data transmission stopped (sticky)
- `unsubscribed` — session not sent to browser at all

Sessions with no row default to `subscribed` (backward compatible — existing users see all sessions).

### New table: `card_preferences`

```sql
CREATE TABLE IF NOT EXISTS card_preferences (
  user_email TEXT PRIMARY KEY,
  auto_show_new INTEGER NOT NULL DEFAULT 1,      -- always show new cards
  auto_show_own INTEGER NOT NULL DEFAULT 1,       -- always show MY new cards
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

Default: both enabled (current behavior — all sessions auto-appear).

## Server Changes (server.mjs)

### Session Discovery Filter

`sendSessionDiscovery()` currently sends `session-add` for every session from `cpRequest('listSessions')`. Change to:

1. Query `card_subscriptions` for the user
2. Skip sessions with state `unsubscribed`
3. For sessions with state `paused`, send `session-add` with a `paused: true` flag
4. Sessions with no row or state `subscribed` → send normally

### New Session Handling

When a `session-add` event arrives from claude-proxy (new session created):

1. Check `card_preferences` for the user:
   - If `auto_show_new = 1` → send to browser
   - Else if `auto_show_own = 1` AND session owner matches user → send to browser
   - Else → suppress (don't send `session-add`), increment hidden count
2. Broadcast updated available/displayed/paused counts to the browser

### WS Message Handlers

New messages from browser → server:

| Message | Fields | Server Action |
|---------|--------|--------------|
| `card-set-state` | `{session, state}` | Update `card_subscriptions` row. If `unsubscribed`: remove watcher, send `session-remove` to browser. If `paused`: stop data relay. If `subscribed`: bridge session, send screen data. |
| `card-set-prefs` | `{autoShowNew, autoShowOwn}` | Update `card_preferences` row. |
| `card-list-all` | `{}` | Return all sessions the user can see (including unsubscribed) with name, owner, age, current state. |
| `card-save-state` | `{states: [{session, state}, ...]}` | Bulk update `card_subscriptions` for all sessions. Used by "Save Current State" button. |
| `card-terminate` | `{sessions: [...], pin: "..."}` | Admin-only. Verify PIN, terminate listed sessions on server. |

New messages from server → browser:

| Message | Fields | Purpose |
|---------|--------|---------|
| `card-list` | `{sessions: [{name, owner, age, state}, ...]}` | Full list for CARDS panel |
| `card-counts` | `{available, displayed, paused}` | Status bar + badge update |

### Count Tracking

Server maintains per-user counts and sends `card-counts` on:
- Initial connect (after discovery)
- Any `card-set-state` or `card-save-state` change
- New session appears / session ends

## Browser Changes (dashboard.mjs)

### CARDS Sub-Panel

- Triggered by: `CARDS >` menu item, orange badge click, or status bar click
- Sends `card-list-all` to server on open
- Renders session list grouped by state (displayed → paused → hidden)
- Each row: action buttons (left), name, owner, age, status indicator (right)
- Toggle buttons for auto-show preferences
- `[Save Current State]` button at top
- Admin terminate section at bottom (admin/root only)
- Action buttons send `card-set-state` immediately

### Save Current State Handler

When clicked:
1. Iterate all sessions from `card-list` response
2. For each: check if card exists in `terminals` Map
   - Exists and `_muted = false` → state `subscribed`
   - Exists and `_muted = true` → state `paused`
   - Does not exist → state `unsubscribed`
3. Send `card-save-state` with the full state array

### State Change Handling

On receiving `card-set-state` response:
- `unsubscribed`: call existing `removeTerminal()` flow (remove DOM, thumbnail, CSS3DObject)
- `paused`: apply same visual as thumbnail pause (overlay, stop data) but mark as `_stickyPaused`
- `subscribed`: if card doesn't exist, server sends `session-add` + screen data; if exists (was paused), resume

### Orange Badge

- New DOM element in top bar: `<span id="hidden-cards-badge">`
- Styled: orange background, white text, visible when count > 0
- Updated on `card-counts` message
- Text: `YOU HAVE N HIDDEN CARDS`
- Click opens CARDS sub-panel directly

### Status Bar Counts

- New element in bottom status bar
- Text: `X available / Y displayed / Z paused`
- Updated on `card-counts` message
- Click opens CARDS sub-panel directly

### Thumbnail Button Changes

Add to each thumbnail (alongside existing ⏸):
- `▶` (green) — visible when card is paused, resumes it
- `⏹` (red stop) — unsubscribes card (persistent, sends `card-set-state`)
- `✕` (red X) — admin-only, terminates session (prompts for PIN)

Existing ⏸ behavior unchanged (temporary, non-persistent).

## Session Lifecycle

### On page load:
1. Browser connects WS
2. Server runs `sendSessionDiscovery()`:
   - Queries `card_subscriptions` for user
   - Sends `session-add` for subscribed sessions
   - Sends `session-add` with `paused: true` for sticky-paused sessions
   - Skips unsubscribed sessions
3. Server sends `card-counts` with available/displayed/paused

### On new session created (server event):
1. Check `card_preferences` for connected users
2. For each user: if auto-show criteria met → send `session-add`, else increment hidden count
3. Send updated `card-counts` to affected browsers

### On session ended (server event):
1. Send `session-remove` to subscribed browsers
2. Remove `card_subscriptions` row (session no longer exists)
3. Send updated `card-counts`

### On user opens CARDS panel:
1. Browser sends `card-list-all`
2. Server returns full list (all sessions user can see) with current state
3. Browser renders list with live dashboard status indicators

### On user changes subscription state (menu or thumbnail):
1. Browser sends `card-set-state { session, state }`
2. Server updates `card_subscriptions`
3. Server acts on state change (subscribe/pause/unsubscribe watcher)
4. Server sends `card-counts` update
5. Browser updates UI (add/remove/pause card)

### On user clicks Save Current State:
1. Browser builds state array from current dashboard
2. Sends `card-save-state { states: [...] }`
3. Server bulk-updates `card_subscriptions`
4. Server sends `card-counts` update

### On admin terminates sessions:
1. Browser sends `card-terminate { sessions: [...], pin: "..." }`
2. Server verifies PIN against admin_pin_hash
3. Server kills sessions via claude-proxy
4. Server sends `session-remove` to all subscribed browsers
5. Server cleans up `card_subscriptions` rows

## Edge Cases

- **Session ends while unsubscribed**: Server cleans up `card_subscriptions` row. No browser notification needed.
- **New session while CARDS panel is open**: Panel should update (server sends `card-counts`, browser can re-request `card-list-all`).
- **Multiple browser tabs**: Each tab has its own WS. State changes via one tab are persisted server-side. Other tabs get `card-counts` updates but don't auto-sync the full list until they open the CARDS panel.
- **User has no `card_preferences` row**: Defaults to `auto_show_new = 1, auto_show_own = 1` (current behavior).
- **User has no `card_subscriptions` rows**: All sessions default to subscribed (current behavior).
- **Admin terminates own session**: Allowed. Session is killed, card removed from admin's dashboard too.
- **Save Current State with no changes**: No-op. Server upserts same values.

## What Stays Unchanged

- Thumbnail ⏸ button behavior (temporary, non-persistent) — KEPT as-is
- Session discovery via claude-proxy `listSessions`
- WebSocket subscribe/unsubscribe messages (used internally by watchers)
- Card DOM creation/removal flow (reused by subscription manager)
- All existing layout, focus, Max All functionality
