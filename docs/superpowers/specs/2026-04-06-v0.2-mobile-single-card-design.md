# Mobile Single-Card Terminal View — Design Spec v0.2

**Date:** 2026-04-06
**Status:** Draft
**Preceding:** v0.1
**Changes from v0.1:** User decisions on detection (auto-redirect), scaling (fit-to-width, +/- text size), gestures (drag=scroll, sideways+up/down=select, release=copy, swipe=next card). Removed open questions that are now resolved.

---

## Core Concept

One terminal card visible at a time. No 3D scene, no CSS3DRenderer. Terminal scales to fit phone screen. Swipe left/right to switch sessions. Input via bottom text field + voice.

## Detection & Routing

**Auto-redirect:** Server checks User-Agent or viewport hint on initial page load. Mobile devices get redirected to `/mobile`. Desktop link available to switch back.

```
GET / → check User-Agent / viewport
  → mobile → 302 /mobile
  → desktop → serve index.html (current dashboard)
/mobile → always serves mobile view
/desktop → always serves desktop view (override for tablets)
```

## Terminal Scaling

**Fit to width (portrait):** Terminal scales so cols fill the screen width. Rows extend below — vertical drag scrolls the terminal viewport (scrollback), not the page.

**Fit to height (landscape):** If rotated, fit to height may give more usable rows. Auto-detect orientation and pick the better fit.

**+/- text size:** Buttons in top bar. This resizes the terminal (sends resize to claude-proxy with different cols/rows) which WILL affect other viewers. This is understood and accepted — the user is choosing to change the terminal dimensions.

**No pinch zoom on the terminal itself** — that would fight with the scroll/select gestures. +/- buttons are the explicit control.

## Gestures

| Gesture | Action |
|---------|--------|
| **Drag up/down** (vertical) | Scroll terminal (scrollback history) |
| **Drag sideways then up/down** | Start text selection — sideways motion initiates selection mode, then vertical extends it |
| **Release after selection** | Auto-copy to clipboard (same as desktop mouse selection) |
| **Swipe left** | Next session (switch to next in session list) |
| **Swipe right** | Previous session |
| **Tap** | Focus input field (if not already focused) |

### Gesture Discrimination

The key challenge: distinguishing scroll (vertical drag) from selection (sideways-then-vertical) from swipe (fast horizontal). Detection logic:

```
touchstart → record start position
touchmove:
  if (horizontal distance > 20px AND vertical distance < 10px) → SWIPE candidate
  if (horizontal distance > 10px THEN vertical distance grows) → SELECTION mode
  if (vertical distance > 10px AND horizontal distance < 10px) → SCROLL mode

touchend:
  if SWIPE candidate AND horizontal distance > 100px → switch session
  if SELECTION mode → copy selected text, clear selection
  if SCROLL mode → momentum scroll (optional)
```

## Session Switching

### Dropdown (top bar)
- Lists all sessions by display name (cp- prefix stripped)
- Shows owner name
- Badge indicator for sessions with activity since last viewed
- Tap to switch — unsubscribes current, subscribes new

### Swipe
- Swipe left → next session in list order
- Swipe right → previous session
- Animate card sliding off and new card sliding in
- Session name briefly shown as overlay during transition

### Subscription model
- Only ONE session subscribed at a time via WS
- On switch: `unsubscribe` old, `subscribe` new
- Session-add/session-remove events received for ALL sessions (name/metadata only, no screen data)
- Max message rate: ~33/sec (one session at debounce rate)

## Input

### Text Field
- Persistent at bottom of screen
- Standard mobile keyboard opens when focused
- Enter/Send submits to terminal, clears field
- Supports paste (long-press paste from clipboard)
- Multi-line input: future consideration (for pasting blocks)

### Special Key Row
- Horizontal scrollable row above the input field
- Common keys: `Ctrl+C`, `Tab`, `↑`, `↓`, `Esc`, `Ctrl+Z`, `Ctrl+D`, `Ctrl+L`
- Customizable order (future)
- Each button sends the appropriate key sequence via WS

### Voice Input
- Browser Speech Recognition API (`webkitSpeechRecognition`)
- Tap 🎤 → listening (red pulse on mic icon, "Listening..." label)
- Speak → interim results in input field (gray)
- Silence → final result in input field (black)
- User reviews/edits, then Enter to send
- Tap 🎤 again to cancel
- **Browser support:** Chrome Android, Edge, Safari iOS 14.5+. Firefox: not supported (show disabled icon).

### Voice Commands (stretch goal)
- "switch to [session name]" → switch without typing
- "scroll up / down" → scroll terminal
- "control c" → Ctrl+C
- Processed before text reaches input field

## What's Shared with Desktop

- `/ws/dashboard` WebSocket connection (same protocol, same auth)
- `terminal.svg` rendering (if we use SVG `<object>` approach)
- Session lifecycle messages (session-add, session-remove, create, fork, restart)
- Auth flow (OAuth, API keys, session cookies)
- All server-side code unchanged

## What's Different from Desktop

| Desktop | Mobile |
|---------|--------|
| Three.js CSS3DRenderer, 3D scene | Direct DOM, no 3D |
| All sessions subscribed | Single session subscribed |
| Direct keydown capture | Input field + voice + special key row |
| Multi-focus (ctrl+click) | Single session, swipe to switch |
| Sidebar thumbnails | Session dropdown + swipe |
| Card chrome (+/−/⊡/⊞/⌊) | +/− in top bar only |
| Mouse orbit/drag/zoom | Touch: drag=scroll, sideways=select, swipe=switch |
| 528+ messages/sec | ~33 messages/sec |
| CSS3D repaint hacks | Not needed (no CSS3DRenderer) |

## Rendering Options

The terminal doesn't have to be an SVG `<object>`. On mobile without CSS3DRenderer, options:

**A) SVG `<object>` (same as desktop):** Reuse terminal.svg as-is. Simple but inherits SVG rendering overhead and the `<object>` sub-document complexity.

**B) Inline SVG:** Render terminal directly in the mobile page's DOM. No `<object>` isolation. Font metrics may differ (known issue from sessions.md). But no cross-document communication needed.

**C) Canvas:** Render terminal to a `<canvas>` element. Fastest for mobile GPUs. But loses text selectability and accessibility.

**D) HTML `<pre>` with spans:** Render as styled HTML text. Simplest, most accessible, works with native text selection. But may not match terminal appearance exactly.

**Recommendation:** Start with A (SVG `<object>` — known working, reuse existing code). If mobile performance is insufficient, try D (HTML pre/spans — simplest mobile-native option). B and C are fallbacks.

## Open Questions

1. **PWA / installable:** Should mobile view be installable as a home screen app? Service worker for offline caching of static assets?

2. **Push notifications:** When viewing session A, session B has output. Browser push notification? Or just badge in dropdown?

3. **Session creation on mobile:** Should the "new session" form work on mobile, or just viewing existing sessions?

4. **Keyboard handling:** When the mobile keyboard is open, the viewport shrinks. Should the terminal resize to fit, or should the terminal scroll behind the keyboard with the input field pinned above it?
